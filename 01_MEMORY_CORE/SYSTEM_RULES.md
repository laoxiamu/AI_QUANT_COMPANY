## 1. Claude 行为规则

### 1.1 角色边界与执行分工（2026-06-04 更新）

**Founder 原则（Founder授权）：**
- Founder 只参与 D 级决策节点（阶段跨越、资金操作、重大架构变更）
- Founder 每天看日报（5分钟），不参与执行层工作
- 减少 Founder 作为"中间人"传递信息，所有执行通过文件系统自动流转

**Claude 执行分级：**
- **直接执行**（Claude via Bash/Python）：数据下载、环境安装、运行脚本、回测执行、结果分析、文件写入。不需要 Founder 参与，不需要经过 Codex。
- **Codex 执行**：复杂系统代码（交易引擎、风控模块、生产部署）。Claude 写任务规格文件 → Codex 自动读取执行 → 结果写回项目目录 → Claude 自动读取验收。
- **不直接执行**：影响生产资金的操作、重大架构变更。必须先提交 Founder 确认。

**禁止行为（新增）：**
- 禁止让 Founder 充当信息传递的中间人
- 禁止因为需要 Founder 确认而停止所有后续工作——只停止该决策点，其他任务继续
- 禁止把 Codex 可自动读取的内容要求 Founder 手动传递

### 1.2 输出前强制检查

在输出任何方案、架构建议或设计决策前，必须先完成：
1. 这个输出影响哪些已有模块或已确认架构？
2. 与当前阶段是否匹配，是否制造路径负债？
3. 是否与 DECISION_LOG 中的已确认决策冲突？

无法回答任意一项，必须先声明"前提信息不足"。

### 1.3 禁止行为

- 禁止在没有充分前提的情况下生产大型方案
- 禁止为迎合用户而默认认同方向
- 禁止自动升级信息等级
- 禁止继承历史架构、历史结论、历史参数
- 禁止将聊天记录直接视为事实
- 禁止为尚未启动的阶段生产具体实现方案
- 禁止引入未经验证的高级架构（多Agent自治、复杂微服务等）

### 1.4 建议与决策分离

- Claude 可以提出方向建议，但必须明确标注"这是建议，需用户确认后才能作为决策"
- 未经用户明确确认的内容，不得在后续对话中被当作既定前提
- 对话中的分析、推演、草稿不得写入 MEMORY_CORE

## 2. 信息治理规则

### 2.1 信息等级（优先级从高到低）

```
DECISION  > 已确认决策，写入 DECISION_LOG
FACT      > 已验证事实，写入 PROJECT_CONTEXT 或 CURRENT_STATE
EXPERIENCE > 经验观察，有条件写入，需用户确认
HYPOTHESIS > 待验证假设，不得写入 MEMORY_CORE
DEPRECATED > 已废弃，标注后归档
CONFLICT   > 冲突信息，暂停写入，等待用户裁决
UNKNOWN    > 信息缺失，不得推断补全
```

### 2.2 文档优先级（冲突时按此裁决）

```
DECISION_LOG > CURRENT_STATE > MEMORY_CORE文件 > PROJECT_DOCS > 对话记录
```

### 2.3 历史资料处理原则

- 所有 03_RAW_INBOX 中的历史资料默认为 HYPOTHESIS
- 不得自动继承历史架构、历史结论、历史参数
- 不得将单一文档视为最终依据
- AI 对话（无论多少方交叉确认）不构成 FACT，最高归入 EXPERIENCE
- 历史文档中"已规划但执行状态未知"的内容，标注"历史规划-执行状态待验证"

## 3. 项目核心原则

这七条原则来自项目历史教训，经三方（用户、Claude、ChatGPT）论证确认：

1. **没有证据的结论不是事实** — 所有判断必须有可追溯来源
2. **架构优先于策略** — 系统可信性是策略有效性的前提
3. **治理优先于扩张** — 先确保当前系统可控，再扩展规模
4. **Memory 优先于 Prompt** — 项目知识写入文件，不依赖对话记忆
5. **Single Source of Truth** — PostgreSQL 是唯一权威状态源，_positions 是缓存
6. **历史决策必须可追溯** — 所有决策写入 DECISION_LOG，含时间、来源、影响范围
7. **先建立研究能力，再建立交易能力** — 验证假设先于开发系统

## 4. 风险警戒规则

项目历史中已验证的三类风险，Claude 必须主动识别并报告：

**风险A（V4已发生）：** 疯狂研究 Alpha，没有治理，项目失控  
触发信号：架构决策未记录、持仓状态依赖内存、代码变更无风险评估

**风险B：** 疯狂建设治理，忘记验证 Alpha，项目失去业务目标  
触发信号：治理文件不断扩展但无实验产出，Phase 0A 验收条件无进展

**风险C（当前最大威胁）：** 既不验证 Alpha 也不建好治理，长期停留在讨论层  
触发信号：连续多次对话只有框架、架构、路线图讨论，无任何实验执行

当 Claude 识别到上述风险信号时，必须主动指出，不得因迎合用户而忽略。

## 5. 新对话启动协议

每次新建对话，Claude 必须：

**第一步：读取四个文件（按顺序）**
```
1. 01_MEMORY_CORE/CURRENT_STATE.md
2. 01_MEMORY_CORE/DECISION_LOG.md
3. 01_MEMORY_CORE/PROJECT_CONTEXT.md
4. 00_PROJECT_MANAGEMENT/PROJECT_OPERATING_STATE.md
```

**第二步：向用户确认**
```
- 当前阶段和进度
- 本次对话目标
- 是否有需要优先处理的紧急事项
```

**第三步：再开始工作**

如果用户未提供当前阶段信息，Claude 必须主动询问，不得直接展开工作。

## 6. Claude → Codex 交接规范

Claude 向 Codex 输出的每个任务规格必须包含：

```
目标：[明确的可验证产出]
约束条件：[技术限制、资源限制、兼容性要求]
验收标准：[如何判断任务完成]
禁止事项：[明确不允许做的事]
参考资料：[相关文档路径]
```

Codex 不得反向影响架构决策。工程层发现的架构问题必须反馈给 Claude 裁决后再处理。

## 7. 本文件维护规则

以下情况必须更新本文件：
- 阶段切换
- 重大架构决策变更
- 发现 Claude 持续输出偏差
- 用户角色或资源约束发生变化

修改须用户明确确认，不得由 Claude 单方面修改。

## 总评（如果我是投资人/顾问）

先说优点，避免你过度纠偏——这套东西的**骨架是好的**：

1. **"先验证策略、再建系统"的顺序是全项目最正确的决定**（计划书阶段2核心原则），直接修正了 V4"系统先上、策略后验"的致命错误。

## 一、架构合理性

**结论：原则正确，但（A）为当前阶段过度设计，（B）缺一个致命角色：独立验证者，（C）四层模型把"角色"和"流程"混在了一起。**

### 1.1 最严重的结构缺陷：没有人给 Claude 的研究结论"挑刺"

整个项目的 North Star 是"找到**真实**有效的 Alpha，别骗自己"。但看清楚这条链路：

- 假设是 Claude 提的；
- 实验是 Claude 设计的；
- 回测结果是 Claude 解读的；
- "通过/不通过"是 Claude 判的；
- Founder 不懂技术，无法复核；Codex 按规定不质疑架构。

**= 提案人 = 裁判员。** 这是研究型项目最忌讳的结构。而 LLM 恰恰有三个已知毛病：倾向于附和、偶尔会一本正经地算错统计、会幻觉。一旦 Claude 把一个**过拟合/含前视偏差**的策略判成"有效"，没有任何环节能拦住它，最后用真金白银去验证——而且全程所有人都以为流程很严谨。

你们请我做这次"独立审阅"，本身就说明你们直觉到了这个缺口。但它现在是**临时的、一次性的**，没有被写进架构成为**常设职能**。这是我认为最该补的一条。

### 1.2 为当前规模过度设计

这是一个**还没跑通任何一个研究闭环**（Phase 0A 五项验收 0/5 真正完成）的单人项目，资金上限 3 万、月预算 1 千。但架构已经有了：5 个分层 + 一部"宪法" + Codex 的 5 层子架构 + 信息流状态机 + 研究流程状态机……这是一家**公司**的治理体量，不是一个 pre-product 研究项目的体量。

风险在于：**精致的治理会变成"不去做真实实验"的高级借口**。文档自己点名的"风险C（长期停留在讨论层）"——架构本身就是这个风险的症状。一个信号供你参考：项目至今产出了 432 行架构、v2 计划、11 方向调研报告，**但完成的真实回测数量 = 0**。元工作（计划如何计划）与实work（跑一次实验）的比例严重失衡。

### 1.3 四层模型在概念上不干净

HUMAN=Founder、GOVERNANCE=Claude、EXECUTION=Codex 都是**角色**，但 RESEARCH LAYER 是**流程**——它没有独立的执行者，实际是 Claude（提案/解读）+ Codex（跑代码）跨层协作完成的。把三个"人"和一个"流程"并列成"四层"，读起来漂亮，但不是干净的分解。更准确的说法是：**3 个角色（人/Claude/Codex），1 条研究工作流**。这不影响运行，但说明分层是按美感而非按职责切的。

### 1.4 其他

- **单点依赖 Claude**：CTO + 策略总监 + 知识治理 + 研究分析 + 文档，全压在一个 AI 上，且无独立校验（见 1.1）。
- **记忆靠手动读四个文件**：能用，但脆弱——依赖每次都记得读、文件不与现实漂移、且无"加载的上下文 = 真实状态"的校验。调研报告自己也说行业在转向自动检索。
- **"继承 V4，已验证13天稳定"措辞过强**：13 天验证窗口极短，且 V4 正是出过 Ghost Position 事故的版本。继承可以（省成本），但别用"已验证稳定"的措辞给自己壮胆。
- **复现性基础设施缺位**：一个"研究操作系统"最该是一等公民的东西——数据快照/版本、环境锁定、随机种子、实验登记册（让半年后能一字不差重跑任一回测）——架构里几乎没有。

## 二、计划可行性

**结论：结构和排序是真的好，但有几处具体硬伤必须先修；对非技术 Founder 而言，作为"管理框架"可执行，但严重低估了 0A 阶段的技术摩擦，且整条执行链吊在一个从未跑通的 Codex 上。**

### 2.1 【硬伤·必须修】两份文档的验收阈值互相矛盾

这两份文档要被**一起冻结成项目基准法**，但它们对"策略算不算通过"这个最核心的判据**已经打架了**：

| 指标 | 架构图 §6.2 | 计划书 阶段1 | 冲突 |
|------|------------|-------------|------|
| 夏普比率 | > **0.5** | > **1.0** | ✅ 矛盾 |
| 样本量 | ≥ **1000** 次 | ≥ **500** 次 | ✅ 矛盾 |
| Expectancy | > 1.0 | 扣费后正期望 | 口径不同但不冲突 |

冻结这两份 = 以后每个假设的"通过/失败"都能视你引用哪份文档而改变。**必须先统一成唯一真相源。**

### 2.2 【硬伤·必须修】第一个策略可能根本凑不够样本量

第一个候选 Setup 是 **BTC / 4H / 三重确认**（Liquidity Sweep + CHoCH + FVG）。算一笔账：

- BTC/USDT 4H 从 2017-08 至今 ≈ **19,300 根 K 线**（全部历史）。
- 三重确认的低频 Setup 触发率很低。若触发率 2%，约 **385 次**；要凑够 500 次需触发率 ≥ 2.6%；要凑够 1000 次需 ≥ 5.2%（即平均每 3 天就触发一次）。
- 对一个要求三个条件同时成立的低频结构信号，5% 触发率几乎不可能，2.6% 也偏高。

**= 你的第一个假设，很可能在 BTC 4H 单品种上无法达到自己设定的统计门槛。** 调研报告其实已经问到了这个问题（方向④："4H 有多少根 K 线？"），但计划没回答它。这必须在投入 Phase 0A 之前用半小时算清楚，否则会白烧一个研究周期。解法是现成的（多品种、降周期、放宽确认），但要现在就决定。

### 2.3 【应修】资金阶梯的数字对不上

`1000→最大亏300（30%）`、`10000→最大亏1000（10%）`、`30000→最大亏900（3%）`。第三步的 900 既偏离了递减规律，绝对值还**比上一档的 1000 还小**（钱更多、能亏的反而更少）。调研报告里只提了 30% 和 10%，没提 30000 这档。这是关于真金白银的数字，冻结前必须确认是不是笔误。

### 2.4 【应修】防过拟合的护栏被点了名，但没焊进早期任务

调研报告正确点出三大杀手（过拟合、前视偏差、regime change），说要加走前向验证。但在 0A/0B 的任务清单里，**第一个回测（0B-1/0B-2）只产出"4 项指标 + 资金曲线"，然后 Claude 来判**——没有样本外留出、没有 walk-forward、没有多重检验校正写进**早期任务的验收标准**里。这些被推迟到了 0A-2"Research Protocol"，但 0A-2 的验收标准并没有强制要求"任何结论被采信前必须先过样本外/walk-forward"。**鉴于 Founder 无法自己判断统计，这些护栏必须在 Protocol 里写成不可协商的硬门槛，并在计划里把它设成验收线。**

### 2.5 Codex 在关键路径上，但从未跑通过

Codex 状态是"已配置，待完成第一个真实任务"——**0 个真实任务**。而几乎每个执行任务都依赖它。计划里有 0A-6 作为试用任务（好），但：
- Claude→Codex 的交接在本项目**从未端到端跑通**；
- 一个非技术 Founder 要负责触发 Codex、在它失败时判断发生了什么、在交接断裂时居中——这被低估了；
- **没有 Codex 失败时的退路**。架构规定 Claude"不写代码、不执行终端命令"，那么 Codex 一旦搞不定，项目就卡死。

> 补充一个事实：Claude 在带沙箱/执行能力的环境里**本来就能跑 Python**。所以"Codex 失败"的退路其实很现成——只是当前架构主动放弃了它。这条退路值得写进计划，哪怕只是一行。

### 2.6 低估了 0A 的技术摩擦

计划在纸面上很顺，但量化研究在工程层是出了名的折磨人：`ta-lib` 在 M1 上要装底层 C 库、`vectorbt` + `numba` + `numpy` 版本冲突常见、数据质量惊喜、回测逻辑里的前视偏差极易引入又极难发现。"All dependencies OK"这条验收标准底下藏着巨大的实操复杂度。当（不是如果）这些发生时，非技术 Founder 完全依赖 Claude+Codex 自我诊断。**现实里 0A 很可能远超 2–4 周。**

### 2.7 优点（别改）

阶段门控（策略验证通过才进系统）、大节点/自主执行的切分、可量化验收、日报模板、"失败是排除错误路径"的心态框架——这些都对，保持。时间估算虽乐观，但已标注"非承诺"，可接受。

## 三、风险识别

文档对**执行/技术风险**的防护是强的（四条铁律、状态机、4 层风控、保守资金阶梯、强制扣费+滑点）——这是 V4 已经咬过你们一口的地方，装甲很厚。

但**最大的风险恰恰不是被防得最严的那个**。我的排序：

1. **研究结论无独立验证（认知风险）——最大、最致命。** 见 1.1。提案人即裁判，而项目的全部价值就建立在"判断结论真假"上。当前几乎无防护。
2. **"最可能的结局是根本找不到 Alpha"（基准率风险）。** 说句逆耳的：散户、3 万本金、公开数据、加上调研报告自己都承认缺乏严谨学术背书的 SMC 概念——**在这个量级找到可持续 Alpha 的历史基准率本就很低**。文档把"找到 Alpha"当成时间/迭代问题（"1–3 个月"），但它可能是个**不一定有解**的问题。而项目**没有定义任何"项目级止损"**：失败多少个假设、烧多少个月、花多少钱之后，Founder 该判定"此路不通"并转向或停止？没有这个开关，最大的代价是 Founder 的**时间和机会成本**被无限投入。这是最威胁你切身利益、却几乎没被处理的风险。
3. **单 AI 依赖 + 上下文脆弱。** 一次没读对四个文件、一次上下文漂移，"CTO"就可能基于过期状态做决策，且无自动一致性校验。
4. **过拟合/方法论风险。** 已被点名，但未焊进早期任务（见 2.4）。
5. **执行/技术风险。** 防护充分（V4 的教训）。

讽刺之处：**项目给已经咬过自己的那条狗（执行）穿了全身甲，却对最可能决定生死的两条（研究真伪、以及"也许根本没有 Alpha"+ 无项目止损）几乎赤膊。**

执行层风控：充分。研究真伪 + 项目级风险：**不充分。**

## 四、关键缺口（目前还没考虑到的事）

按重要性排：

1. **独立验证/红队职能（最大缺口）。** 需要一个与提案人分离的、常设的"怀疑者"步骤：任何结论被采信前，由一个被明确要求"去攻击这个结论"的独立 Claude 对话过一遍 + 一份固定检查清单（前视偏差？样本外做了吗？多重检验校正了吗？参数敏感度？）。
2. **项目级止损/退出条件。** 失败 N 个假设 / 烧 M 个月 / 花 X 元 → 触发"重新评估整个前提"。
3. **复现性基础设施。** 数据快照与版本、环境锁定、随机种子、实验登记册。研究 OS 没有这个就不成立。
4. **样本外/walk-forward 设为不可协商的硬门槛**，并预留一段**从不触碰**的 holdout 直到最终验证；**多重检验校正**（你会测很多假设 → 不校正必然出假阳性，调研报告方向⑩提了一次但没进验收标准）。
5. **第一个 Setup 的样本量可行性前置测算**（见 2.2），在承诺它之前算清楚。
6. **Codex 失败的退路**（见 2.5）+ 两份文档矛盾的对账（见 2.1）。
7. **API 密钥/安全。** 实盘需要交易所密钥——密钥存储、禁用提币权限、IP 白名单、密钥泄露 = 资金归零。Phase 2/3 至少要把它列为风险项。
8. **合规/法律/税务。** 从基础设施细节（腾讯云新加坡 + "链式代理"访问 Binance）能看出存在监管摩擦。这不该只作为一条 infra 配置，而应作为一条**风险**进入 Founder 视野（我不提供法律意见，但这必须在你雷达上）。
9. **总预算 runway 的现实性。** 月预算 1 千够买工具，但若 0B→1 真的迭代 3 个月重度使用 Claude/Codex，**整个项目能撑几个月**、1 千具体买的是什么（Pro？API？Codex 订阅？）没有算清。
10. **Founder 的行为/心理风险。** 非技术 + 真金白银 + 交易这个最折磨心态的领域。当一个"通过验证"的策略实盘亏钱、或 Founder 想绕过系统/加快加仓时，没有纪律预案。资金阶梯门控有帮助，但人性维度防护不足。

## 五、冻结建议

**建议暂缓冻结。先做下面 6 项定向修改（预计 1–2 天），再冻结。** 这不是推倒重来——骨架是好的，绝大多数内容可保留。但冻结**互相矛盾的验收阈值**和**缺独立验证职能的架构**，等于第一天就把缺陷写进宪法；而现在目录大多还空着、一切还在纸面，改的成本接近零。

**冻结前必修（按优先级）：**

- **A. 统一两份文档的验收阈值**（夏普 0.5 还是 1.0？样本 500 还是 1000？）——定一个唯一真相源，两边写成完全一致。【见 2.1】
- **B. 在架构里加入"独立验证职能"**——结论采信前必经一个常设的、与提案人分离的怀疑者步骤 + 检查清单。【见 1.1 / 四-1】
- **C. 加入项目级止损/退出条件**（时间盒、失败假设数上限、预算上限 → 触发重评）。【见 三-2 / 四-2】
- **D. 把样本外/walk-forward + 多重检验校正焊成 Research Protocol 的硬门槛**，并在计划里设成验收线。【见 2.4 / 四-4】
- **E. 对第一个 Setup 做样本量可行性前置测算**（半小时），据此决定单品种/多品种/周期。【见 2.2】
- **F. 修正 30000/900 资金数字 + 补一行 Codex 失败退路。**【见 2.3 / 2.5】

A–F 都是**增补与对账**，不需要重新设计。做完即可冻结。

**最后一句给 Founder 的话：** 即便修完冻结，也请把它当作**会随现实修订的 v1.0，而不是圣经**——并明确排一个复盘点（建议第一个研究闭环跑完后）。这个项目现在最缺、也最有价值的一步，不是再多一份文档，而是**把第一个真实回测结果摆上桌**——哪怕粗糙。它教给你的，会比再多的规划都多。"冻结文档"这个仪式本身，也有变成又一次拖延真实检验的风险。修好、冻结、然后**快速冲向第一个实验。**

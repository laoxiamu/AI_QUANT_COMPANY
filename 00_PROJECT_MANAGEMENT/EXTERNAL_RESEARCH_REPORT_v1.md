## 方向②：Context Engineering / Memory 系统设计

### 核心发现

**行业已形成标准分层框架（2026年现状）：**

"Context Pyramid" 四层模型：**Identity → Knowledge → State → Task**，每层职责独立，出问题时能精确定位是哪一层失效。

**关键设计原则：**

1. **层级分离**：Working Memory（当前任务）、Long-term Memory（历史知识）、Retrieval Memory（按需检索）三类必须独立管理，不能混用。
2. **信息位置效应**：LLM 对 context 头部和尾部的关注度显著高于中间段，关键信息必须放在开头或结尾。
3. **可溯源性**：2026年行业趋势是对每条答案记录"使用了哪些信息生成"，对应我们的 MEMORY_EXTRACTION_PROTOCOL 中的来源标注。
4. **渐进式积累**：系统应逐层建立理解，只保留 working memory 中必要的内容（对应"最小必要原则"）。

**工具方向**：行业标准组合为 LlamaIndex（数据摄取+索引）+ LangChain/LangGraph（编排+Agent逻辑）。

### 对我们架构的影响

✅ **验证**：Memory Core 的分层设计（CURRENT_STATE / DECISION_LOG / PROJECT_CONTEXT / SYSTEM_RULES）与行业 Context Pyramid 框架一致，设计方向正确。  
⚠️ **补充**：目前每次对话需手动读取四个文件，行业标准已在向"自动检索"方向演进。Phase 0B 可考虑通过 Codex 实现自动化 context 注入。

## 方向③：Claude + Codex 多工具协作工作流落地案例

### 核心发现

**工程实践已形成标准模式：**

1. **Handoff 机制**：Claude Code 官方文档明确支持跨 Agent 的结构化 handoff——生成包含"接收方所需精确字段"的上下文转移文档，支持"在 Claude Code 规划、在 Codex 执行"的分工模式（与我们的 Claude→Codex 交接规范完全一致）。
2. **Operator 模式**：主控 Agent 将高层目标分解为子任务分发给子 Agent，再综合结果，每个子 Agent 有明确角色和边界。
3. **并行 subagent**：Claude 可动态生成编排脚本，同时启动多个协调的 subagent 并行执行（当前为 research preview）。

**工程红线（与我们的架构高度一致）：**
- Handoff 越模糊，结果越模糊——交接文档必须包含明确的任务分解、失败处理方式、输出格式规范
- 每个并行 branch 的输出 schema 必须一致，才能合并
- 初始 prompt 质量决定整体质量，多 Agent 场景没有中途纠正的机会

### 对我们架构的影响

✅ **验证**：我们的 Claude→Codex 交接规范（目标+约束+验收+禁止事项）符合行业最佳实践。  
⚠️ **补充**：目前缺少"Codex 执行报告回传 Claude"的标准格式，建议在 Phase 0B 建立 Codex 任务模板时同步设计执行报告模板。

## 方向④：量化交易 Research OS 方法论

### 核心发现

**行业标准：走前向验证（Walk-Forward Validation）**

标准框架包含：将历史数据划分为多个重叠窗口，每个窗口含训练段和验证段，按时间顺序滚动推进，强制模拟真实约束（不允许用未来数据）。

**95%回测策略在实盘失败的根因（三条）：**
1. **过拟合**：样本内参数优化——策略记住了历史，不是学到了规律
2. **前视偏差**：使用了真实交易中无法获得的信息
3. **可解释性缺失**：黑盒模型无法判断为何有效，无法在环境改变时判断是否还有效

**单变量原则的行业背书**（与我们 D31 铁律一致）：一次只改一个变量，否则无法追溯每个改动的独立贡献。

**正确的假设文档应包含**：假设名称、描述、研究范围（universe）、信号定义、预期效果、背后逻辑。

### 对我们架构的影响

✅ **验证**：架构草案中的研究闭环（假设→实验设计→数据→回测→分析→结论→归档）符合行业标准。  
✅ **验证**：假设验证标准阈值（Expectancy>1.0, Sharpe>0.5, MaxDD<25%）属于合理范围。  
⚠️ **补充**：当前 D31 研究协议缺少"走前向验证"的具体分窗方法，建议在 Research Protocol 建立时纳入。  
⚠️ **补充**：样本量阈值"≥1000次穿仓"需要结合历史数据量评估可行性（BTC 4H级别历史数据约有多少根K线？）。

## 方向⑤：SMC/FVG/CHoCH 结构行为统计验证研究

### 核心发现

**现状：有实践案例，缺乏严谨学术验证**

- FVG（Fair Value Gap）：有实践者报告填充率约70%，但这是描述性统计，非控制实验结论。
- 2026年3月，一篇基于2600笔交易的SMC回测文章（Medium）声称"某些结构信号有正期望"，但样本周期仅2个月（2025年7-9月），统计显著性存疑。
- 学术层面：关于 Liquidity Sweep + 微观结构的研究集中在"机构订单流对价格的影响"，而非 SMC 概念本身的验证。
- GitHub 存在 Python SMC 开源实现（joshyattridge/smart-money-concepts），可编程化，但无系统性统计验证数据。

**关键风险识别：**

SMC/ICT 概念本质上是对市场微观结构的描述性框架，其可程序化的部分（CHoCH、BOS、FVG）需要独立统计验证才能判断是否有可持续 Alpha。"70% FVG 填充率"的说法在多数场合缺乏样本量、时间段、资产类别的完整披露。

### 对我们架构的影响

✅ **支持**：V5 策略方向"不信仰 SMC 概念，而是将结构行为拆解为可统计验证的行为因子"——这个方向**比直接信仰 SMC 更科学**，与行业最严谨的做法一致。  
⚠️ **风险**：第一个假设 Setup（Liquidity Sweep + CHoCH + FVG 三重确认）目前没有可直接引用的严谨学术背书，需要我们自己做统计验证，这是合理的，但不能预设结论。  
🔴 **重要提示**：这正是为什么"第一个研究闭环"是 Phase 0A 的最重要验收条件——不是因为我们相信它有效，而是要**亲自验证**它是否有效。

## 方向⑥：小资金加密货币量化实盘经验

### 核心发现

**3万人民币（≈4000美元）量级的核心约束：**

1. **手续费是最大杀手**：每笔交易 0.05% 手续费在高频策略中会侵蚀全部利润。Binance Spot Maker/Taker 费率约0.1%，VIP等级降费需要交易量门槛。对于低频策略，单笔盈利目标必须远大于手续费。
2. **滑点建模**：小资金下滑点影响成比例更大，回测必须加入保守滑点假设（建议按历史成交量的保守估计建模）。
3. **流动性约束**：BTC主流品种流动性充足，但低市值币种的流动性会让小资金也产生显著市场冲击。BTC/USDT 4H级别低频策略是小资金最友好的选择。
4. **保证金与爆仓风险**：杠杆交易在小资金下极易爆仓，建议验证阶段全程使用现货或低杠杆（≤2x）。
5. **95%的回测策略在实盘失败**：主要原因是过拟合 + 滑点低估 + 制度变迁（regime change）。

**最小可行策略特征**（适合我们阶段）：
- 交易频率：低频（每周不超过10笔）
- 品种：BTC/USDT 主流对
- 周期：4H+（减少手续费侵蚀）
- 止损逻辑：结构止损（非固定比例）
- 资金管理：固定比例风险，单笔不超过总资金2%

### 对我们架构的影响

✅ **验证**：架构草案中"BTC 4H + 低频 + 结构止损"的策略方向，是小资金量化最合理的选择。  
✅ **验证**：风险容忍设定（1000元最大亏损30%，10000元最大10%）——换算为单笔风险是合理的。  
⚠️ **补充**：研究协议中的回测必须包含真实手续费（0.1% per trade）+ 滑点假设（建议0.05%-0.1%额外成本）。

## 方向⑩：IC 分析方法论

### 核心发现

**IC 计算的正确方法：**

1. **IC 定义**：因子值与未来收益的 Spearman 秩相关系数（不是 Pearson，加密市场分布非正态）
2. **滚动窗口**：使用30天滚动IC，而非全局IC，观察预测能力的稳定性
3. **现实基准**：IC 在股票市场中"几乎无法与零有实质性差异"，加密市场噪声更大——V4 的 IC≈0.05-0.08 在量化研究中属于弱但有意义的信号
4. **常见陷阱**：
   - 用 Pearson 而非 Spearman（对异常值敏感）
   - 用全局IC掩盖了时期间的不稳定性
   - 未做多重检验校正（测试100个因子，5个偶然显著）
5. **最新研究**：2025年学术界在加密市场验证了多因子模型，波动率相关因子在ETH上有统计显著性（与我们 V4 中波动压缩因子 IC≈0.06-0.08 的历史发现一致）。

### 对我们架构的影响

✅ **验证**：V4 中波动压缩因子的 IC 验证方法论方向正确，但需确认是否用了 Spearman（而非 Pearson）。  
⚠️ **补充**：Research Protocol 建立时，IC 计算标准应明确指定使用 Spearman + 30日滚动窗口 + 多重检验校正。

## 方向⑪：节约 Token / 降低 AI 工具成本方案

### 核心发现

**可节省 40%-90% Token 成本的实践方法：**

**架构层面（最高效）：**
1. **Prompt Caching**：对固定的架构摘要、系统规则等内容启用缓存，同样内容不重复计费。Claude API 提供 `cache_control` 参数，可节省对应部分约90%的输入 token 成本。
2. **模型分级**：Haiku 成本约为 Sonnet 的1/4——简单任务（数据下载、格式转换、简单问答）用 Haiku，复杂架构决策用 Sonnet/Opus。
3. **上下文控制**：最大单一来源：不加控制的 context 爆炸。每次对话只加载当前任务所需的最小上下文（对应我们的"四文件"而非加载所有文档）。

**工作流层面：**
4. **Sprint 化**：将大任务分解为短对话 sprint，每个 sprint 结束时 /clear（context 清空），下一个 sprint 重新注入精简 context。实测可减少67%成本。
5. **搜索范围限制**：在 Claude Code/Codex 中明确限制搜索范围（指定目录、文件类型），防止 agent 扫描全仓库。
6. **文件优于对话**：把需要反复引用的内容写入文件（我们的 Memory Core），而不是每次在对话中重新描述——文件可被缓存，对话不能。

**我们项目的具体建议：**

| 措施 | 预估节省 | 实施难度 |
|------|---------|---------|
| Cowork 对话 sprint 化，避免单次超长对话 | 30-50% | 低 |
| CLAUDE.md 保持简洁（现134行合理），避免无限膨胀 | 10-20% | 低 |
| Codex 任务明确指定文件范围 | 20-40% | 低 |
| 简单文件操作改用 Haiku（而非 Sonnet） | 40-60% | 中 |
| Prompt Caching（需要API层配置） | 50-90%（对缓存内容） | 高 |

**对我们架构的影响：**  
✅ **验证**："文件写入优于对话记忆"（SYSTEM_RULES 原则4）与 token 成本最优策略高度一致——好的治理架构同时也是低成本架构。  
⚠️ **行动项**：当前在 Cowork 的工作方式已接近最优（每次新对话读取四文件+执行任务），继续保持。避免在单次对话中累积超过必要的上下文。

# Claude ↔ Codex 文件协作规范

**版本：** 1.1 | **更新：** 2026-06-06 | **制定人：** Claude（CTO）

---

## 专业负责人标准（P0·强制，DEC-057）—— Codex 同样适用

> 根因：把"严格执行任务书"置于"先判断任务是否值得执行"之前 = 局部修补搜索。

**Codex 在每个研究任务执行前，必须先在执行报告开头输出"专业审查七问"：**
① 本实验在验证什么交易机制？② 现有证据是否支持继续研究该机制？③ 有无更上游问题未答？
④ 变量能否真正作用于目标指标？⑤ 最可能的失败原因能否由本实验区分？
⑥ 是否存在信息增益更高的实验？⑦ 若方向有误——**先在报告提专业异议并暂停落地，交 Claude/Founder 裁决，不机械执行**。

**职责边界 ≠ 思考边界：** Codex 无权改重大架构决策，但发现研究方向有问题（不只是工程冲突）时，
有义务暂停并提出异议与替代框架。已在工程冲突上做到，须扩展到交易研究方向。

**研究顺序铁律：** 市场状态划分→各状态归因→确认edge存在的状态→状态过滤后信号验证→仓位成本优化；
禁止"信号失败→改止损→改退出→改仓位→加过滤"。

## 核心原则

Claude 和 Codex 共用同一个项目目录。为避免文件碰撞和治理混乱，双方各有明确的写入权限边界。

---

| 文件 | 原因 |

| `00_PROJECT_MANAGEMENT/` 下所有文件 | 架构/计划/审计文档，由 Claude 负责 |

## Codex 的决策性输出处理流程

当 Codex 在执行过程中做出或确认了重要判断（如样本量、参数选择、品种扩展），应按以下流程处理：

```
Codex 在执行报告中记录判断 + 标注「需 Claude 审阅升级」
    ↓
Claude 读取执行报告，审阅内容合法性
    ↓
Claude 决定是否写入 DECISION_LOG（需标注 Founder 是否已确认）
```

**Codex 可以代为整理决策草稿**（写在执行报告的特定区域），但不能直接写入 DECISION_LOG 作为正式决策。

---

## 文件写入前的检查规则

**Claude：** 写入任何共享文件前，先读取确认当前版本，避免覆盖 Codex 的有效输出。

**Codex：** 读取 CURRENT_STATE 了解状态，但不覆盖写入。执行完毕后在报告中说明「需要 Claude 更新以下文件」。

---

## 版本号约定

CURRENT_STATE 版本号规则：
- **Claude 更新** → 版本号直接升级（如 2.5 → 2.6）
- **Codex 更新** → 不得修改版本号，在执行报告中注明需要 Claude 更新

---

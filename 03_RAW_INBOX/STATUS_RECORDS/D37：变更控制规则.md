```Markdown
## 一、文档目的

本文件定义V4.6.2项目后续所有变更的审批、执行、验证和回滚规则。SCORE_A在13天内经历“原始值→降低→回滚”的完整循环，同向覆盖从部署到发现Ghost Position跨越了整整3天才被偶然发现——这两起事故是建立本规则的直接证据。

本规则确保任何变更在被部署到生产代码之前，都经过可追溯的风险评估和审批流程。任何变更在部署之后，都经过可验证的完整性确认和文档同步。


## 二、变更分级标准

### 2.1 变更等级定义

| 等级 | 定义 | 审批要求 | 风险评估 | 重新观察 | 案例 |
|------|------|---------|---------|---------|------|
| **P0 致命变更** | 涉及持仓生命周期（创建/修改/删除/覆盖/平仓）的任何代码变更。涉及风控完整性（止损触发逻辑、Phase 1输入源、Reconciliation Loop）的任何代码变更 | 必须经项目负责人书面确认。必须经至少两个独立AI交叉验证 | 必须包含：影响范围分析、最坏情况推演、回退计划、验证方式 | 必须。变更部署后必须在下一轮扫描中验证Phase 1是否覆盖全部持仓 | FIX-5引入同向覆盖逻辑；禁用同向覆盖；修复`log_trade()`匹配逻辑 |
| **P1 高危变更** | 修改策略参数的数值（止损百分比/时间阈值/评分门槛/过滤阈值/回撤参数）。修改退出机制（新增退出类型/调整退出优先级） | 必须经项目负责人确认。建议经至少一个AI验证 | 必须包含：参数变更可能导致的信号产出率变化、PnL分布变化、最大回撤变化预估 | 建议。如果参数变更涉及风险敞口大小，必须重新观察 | SCORE_A从18→16（HOLD-2）；`holding_days`从5天→24h |
| **P2 中等变更** | 修改非核心策略逻辑（环境分类阈值、仓位权重系数）。新增辅助功能（新的过滤条件、新的监控指标）。修改文档但不修改代码 | 必须记录到变更日志。不需要额外的独立审批 | 必须包含：变更的功能描述和预期效果 | 不需要 | 调整环境分类器中ADX阈值；新增复盘报告中的某个统计字段 |
| **P3 次要变更** | 代码注释、日志格式、文档格式。不影响任何策略逻辑、风控逻辑或数据记录的修改 | 不需要审批 | 不需要 | 不需要 | 给某个函数添加注释；统一日志输出格式 |

### 2.2 变更等级的判定流程

1. **自检**：修改者（人或AI）在提交变更前，首先自检此变更属于哪个等级
2. **如果对自己判定不确定**：一律按更高等级处理
3. **如果变更涉及多个文件或模块**：按最高影响等级的文件判定整个变更的等级
4. **禁止降级**：任何情况下不得将P0致命变更降为P1处理


## 三、变更执行标准流程

### 3.1 所有变更都必须经过的五个阶段
```

\[1\] 提出 → \[2\] 评估 → \[3\] 审批 → \[4\] 实施 → \[5\] 验证

```Plain Text

### 3.2 阶段详情

#### 阶段1：提出

| 必须记录 | 说明 |
|---------|------|
| 变更内容 | 具体描述改了什么 |
| 变更原因 | 为什么需要这个变更（解决问题/优化性能/新增功能） |
| 影响范围预估 | 涉及哪些文件、哪些模块、哪些数据 |
| 变更等级自评 | P0/P1/P2/P3 |

#### 阶段2：评估

| 评估维度 | P0致命变更 | P1高危变更 | P2/P3 |
|---------|-----------|-----------|-------|
| 影响范围分析 | 必须 | 必须 | 不需要 |
| 最坏情况推演 | 必须 | 建议 | 不需要 |
| 回退计划 | 必须 | 必须 | 不需要 |
| 验证方式 | 必须（包含具体的验证步骤和验收标准） | 必须 | 不需要 |
| AI交叉验证 | 必须（至少两个独立AI） | 建议（至少一个） | 不需要 |

#### 阶段3：审批

| 变更等级 | 审批人 |
|---------|--------|
| P0 | 项目负责人书面确认 |
| P1 | 项目负责人确认（可口头） |
| P2/P3 | 不需要审批，但必须记录到变更日志 |

#### 阶段4：实施

所有变更必须遵循以下规则：

| 规则 | 说明 |
|------|------|
| **一次一个变更** | 不在同一轮部署中混合多个P0或P1变更。每个P0变更独立部署、独立验证 |
| **先备份后修改** | 修改任何配置文件前，必须先创建备份（`cp file.json file.json.bak.YYYYMMDD`） |
| **记录到变更日志** | 变更完成后立即在MEMORY.md或D02变动清单中记录：变更内容、变更原因、变更时间、实施者 |
| **同步文档** | 如果此变更涉及任何参数值的变化，必须在同一轮部署中同步更新MEMORY.md中的参数记录。**禁止“代码已改、文档未更新”的漂移** |
| **标注状态** | 在MEMORY.md中使用标记格式标注变更后的参数状态：✅已对齐 / ⏳待执行 / 📝临时值 / ⚠️不一致 |

#### 阶段5：验证

| 验证维度 | P0变更 | P1变更 | P2/P3 |
|---------|--------|--------|-------|
| **功能验证** | 变更是否达到了预期效果？ | 必须 | 必须 | 不需要 |
| **完整性验证** | 变更是否影响了其他模块？Phase 1是否仍然覆盖全部持仓？ | 必须 | 建议 | 不需要 |
| **日志确认** | 变更后下一轮扫描日志中是否有异常ERROR/WARNING？ | 必须 | 必须 | 不需要 |
| **数据验证** | 变更后数据库中的新记录是否正确？ | 必须 | 建议 | 不需要 |

**P0变更的特殊验证要求**：
- 变更部署后必须立即在下一轮扫描中验证Phase 1是否覆盖全部活跃持仓
- 必须查看`grep "Phase 1开始" /opt/v4/logs/strategy_runner.log | tail -1`确认检查到的持仓数量与数据库中的活跃持仓数量一致
- 如有任何差异，立即回滚


## 四、回滚机制

### 4.1 回滚触发条件

| 条件 | 说明 |
|------|------|
| 验证阶段发现变更未达到预期效果 | 功能不符合设计目标 |
| 验证阶段发现变更引入新问题 | 日志中出现新的ERROR/WARNING |
| Phase 1覆盖完整性验证失败 | 检查到的持仓数量与DB不一致 |
| 变更部署后2轮扫描内出现任何CRITICAL ALERT | 保护性回滚 |

### 4.2 回滚流程

1. **确认回滚**：完成当前轮次扫描后，确认此轮数据已完整写入PostgreSQL
2. **恢复配置文件**：从备份文件恢复变更前的配置（`cp file.json.bak.YYYYMMDD file.json`）
3. **重启引擎**：`systemctl restart v4-strategy-runner`
4. **验证回滚**：确认引擎正常运行、Phase 1覆盖全部持仓
5. **记录回滚**：在变更日志中记录回滚原因、回滚时间、恢复的文件和版本
6. **保留原变更记录**：回滚不删除原变更记录——保留完整的尝试→失败→回滚的审计链

### 4.3 不可回滚的变更

| 变更类型 | 不可回滚的原因 | 预防措施 |
|---------|-------------|---------|
| 数据库表结构修改 | `ALTER TABLE`不可逆 | 先在测试表上验证 |
| 数据库记录修改 | `UPDATE`不可自动回滚 | 先执行`SELECT`确认影响范围，备份受影响的行 |

对于此类变更，必须在评估阶段确认替代的回退方案（如手动SQL回退脚本）。


## 五、观察期内的特殊变更规则

### 5.1 观察期约束

| 允许的变更 | 禁止的变更 |
|-----------|-----------|
| 修复P0致命缺陷（如Bug导致的平仓失效） | 修改策略参数（如止损百分比、评分门槛） |
| 修改巡检/监控/告警脚本 | 新增策略功能（如新退出类型、新过滤条件） |
| 修改数据查询方式（如CSV→DB切换） | 修改评分逻辑、入场规则、退出规则 |

### 5.2 观察期内的变更记录要求

任何在观察期内执行的变更必须在效果评估报告中单独标注：变更时间、变更原因、变更对数据的影响。如果观察期内变更过多导致无法统计单因子的效果，必须在评估报告中声明“观察期数据不具备统计结论效力”。


## 六、变更记录格式

### 6.1 每次变更必须记录到变更日志

变更日志位置：MEMORY.md中的“📝 近期变更日志”部分；D02变动清单中的对应条目；D34决策记录（如果变更涉及关键决策）。

### 6.2 变更记录必须包含的字段

| 字段 | 说明 | 示例 |
|------|------|------|
| 变更日期 | YYYY-MM-DD | 2026-05-08 |
| 变更等级 | P0/P1/P2/P3 | P1 |
| 变更内容 | 具体描述 | SCORE_A从16回滚至18 |
| 变更原因 | 为什么改 | 多方论证确认HOLD-2在22分制预测力不足的情况下放大噪声 |
| 变更文件 | 涉及的文件路径 | `/opt/v4/strategy/config.py` |
| 变更前值 | 变更前的代码值 | 16 |
| 变更后值 | 变更后的代码值 | 18 |
| 实施者 | 人或AI | OpenClaw（经项目负责人确认） |
| 验证结果 | 是否通过验证 | ✅ 通过。下一轮扫描信号产出正常，评分分布回归HOLD-2部署前水平 |
| 回滚计划 | 如有问题如何回滚 | 将SCORE_A改回16，重启引擎 |

### 6.3 文档同步验证

每次变更后必须立即执行文档一致性检查：
- `grep`确认代码中的值是否与变更记录一致
- 确认MEMORY.md中对应参数的状态标记是否已更新
- 确认执行计划中是否已标注变更状态

**如代码、文档、计划三者不一致，变更不算完成。**
```

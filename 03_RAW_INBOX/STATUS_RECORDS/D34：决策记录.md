## 一、架构决策

### AD-1：PostgreSQL 为唯一权威状态源

| 维度 | 内容 |
|------|------|
| **决策日期** | 2026-05-07 |
| **决策内容** | 确立 PostgreSQL `v4_trading.signals` 表为系统中唯一的权威状态源。所有巡检、复盘、评估、风控检查必须直接从PostgreSQL读取数据。CSV文件降级为历史快照，仅供参考。`_positions`字典降级为缓存，不再作为风控决策的输入源 |
| **决策理由** | 数据一致性排查中发现三个CSV文件（`signals_all.csv`、`open_positions.csv`、`closed_trades.csv`）全部为48小时未更新的死文件。巡检报告曾基于这些死数据做出错误判断。同时风控架构确认Phase 1依赖不可靠的`_positions`字典是Ghost Position的根本原因 |
| **替代方案** | ①继续用CSV但增加自动更新脚本（被否决——增加维护复杂度且无法解决多数据源漂移问题）。②建立数据仓库统一管理（被否决——过度设计，当前规模不需要） |
| **风险评估** | 低。切换为DB直读是纯查询操作，不修改任何策略逻辑。唯一风险是巡检脚本的SQL查询需要适配现有表结构，但已在同一天完成切换验证 |
| **影响范围** | 巡检脚本、Cron复盘查询、所有手动数据查询。策略引擎代码不受影响 |

### AD-4：同向覆盖必须禁用

| 维度 | 内容 |
|------|------|
| **决策日期** | 2026-05-08 深夜 |
| **决策内容** | 立即禁用同向覆盖机制。当新信号到达时，如果已有同币种同方向的活跃持仓，直接拒绝新信号，不给覆盖任何机会。后续通过REPLACED状态实现有序替换（先平旧仓→记录PnL→再开新仓）。在完整Lifecycle Engine建立之前，禁止任何形式的自动持仓替换 |
| **决策理由** | Ghost Position Incident确认同向覆盖是P0致命缺陷——593号ETHUSDT LONG从风控系统消失，亏损穿透至-5.47%。49笔僵尸持仓的盈亏被永久丢弃。这是唯一被三方一致确认的紧急修复项 |
| **替代方案** | ①保留覆盖逻辑但加强Phase 1检查（被否决——不解决根因）。②改为信号评分比较替换（被否决——同样会丢失旧仓盈亏）。③改为人工确认后替换（被否决——不适合自动化系统） |
| **风险评估** | 低。禁用覆盖会略微减少信号产出率（同向新信号被拒绝），但不创造新的风险。受益远大于代价 |
| **影响范围** | `strategy_runner.py`中新信号到达时的持仓检查逻辑 |


### AD-5：每个持仓有全局唯一 trade_id

| 维度 | 内容 |
|------|------|
| **决策日期** | 2026-05-08 深夜 |
| **决策内容** | 废弃以`symbol_direction`作为持仓唯一标识的设计。每个持仓必须有全局唯一的`trade_id`。格式：`{symbol}_{direction}_{timestamp}_{hash4}`。`log_trade()`使用`WHERE trade_id=?`精确匹配，废弃`ORDER BY timestamp DESC LIMIT 1`粗粒度匹配 |
| **决策理由** | Ghost Position和`log_trade()`错配都源于同一个根因——一个币种同一方向的所有交易被视为“同一个持仓”。这导致覆盖、错配、审计断裂。唯一trade_id是交易系统的基础需求 |
| **替代方案** | 保留`symbol_direction`但增加序列号区分（被否决——增加复杂度但不解决根因） |
| **风险评估** | 中。需要修改`signals`表结构（新增`trade_id`字段）、`_positions`字典的key结构、`log_trade()`的匹配方式、`log_signal()`的插入逻辑。但所有修改都在V5新代码中进行，不影响V4冻结基线 |
| **影响范围** | `signals`表结构、`strategy_runner.py`、`sell_manager.py`、`signal_logger.py` |


### AD-6：Reconciliation Loop 作为强制防御层

| 维度 | 内容 |
|------|------|
| **决策日期** | 2026-05-08 深夜 |
| **决策内容** | 每轮Phase 1风控检查启动前，强制执行Position Reconciliation Loop。PostgreSQL中的活跃持仓列表与`_positions`缓存中的管理持仓列表进行逐笔对账。发现任何差异（DB有但缓存无、缓存有但DB无）立即触发CRITICAL ALERT推送到Telegram。同时建立Orphan Detection机制——任何持仓超过2轮扫描未被风控检查自动标记为ORPHANED并告警 |
| **决策理由** | Ghost Position在引擎重启时被偶然发现，不是被主动检测出来的。依赖偶然不是防御策略。Reconciliation Loop是风控完整性的最后一道防线 |
| **替代方案** | 无。风控系统必须具备自我校验能力 |
| **风险评估** | 低。对账是纯查询操作，不影响风控逻辑。告警触发后需要人工确认，但不会自动执行任何交易操作 |
| **影响范围** | Phase 1平仓检查的前置步骤 |


## 二、策略决策

### SD-1：HOLD-2 部署与回滚

| 维度 | 内容 |
|------|------|
| **决策日期** | 2026-05-02（部署），2026-05-08（回滚） |
| **决策内容** | HOLD-2：将SCORE_A从18降低至16，使更多B级上沿信号达到A级门槛，从而跳过微观结构过滤直接发送。5月2日部署，5月8日回滚至18 |
| **部署理由** | 当时认为A级门槛过高导致信号通过率偏低，降低门槛可以增加信号密度 |
| **回滚理由** | 观察期内97笔APPROVED中A级仅2笔，5笔真实平仓全为B级（14-15分）。多方论证确认HOLD-2在22分制预测力不足的情况下只起到了"降低门槛→放大噪声"的作用。回滚是"先止血"，后续通过Exit体系优化来提升信号质量 |
| **替代方案** | ①保留16但在Exit体系优化后重新评估（被否决——Exit优化需要单独验证，不应与HOLD-2的效果混杂）。②改为17折中（被否决——没有数据支持这个阈值比16更有效） |
| **风险评估** | 低。回滚至18恢复原初设计，不会引入新变量 |
| **影响范围** | `config.py`中SCORE_A的值。评分等级判定逻辑 |


### SD-2：波动压缩因子保留为辅助信号

| 维度 | 内容 |
|------|------|
| **决策日期** | 2026-05-03 |
| **决策内容** | 三个基础因子（动量、波动压缩、均值回归）IC分析全部未通过。但波动压缩因子在状态分层IC中volatile环境下有显著预测力（BTC +0.0579, SOL +0.0816）。决定不废弃此因子，而是作为22分制的补充加成项保留（BTC/SOL +2分，ETH不适用，其他币种默认+1分） |
| **决策理由** | 这是唯一被统计验证通过的因子。在V5中保留为辅助信号，不替代任何核心评分逻辑 |
| **替代方案** | ①废弃波动压缩因子一并转入V5重新设计（被否决——保留此因子可以在V5中作为对照基准）。②将波动压缩因子升级为主要评分项（被否决——IC太小，不足以独立承担方向判断） |
| **风险评估** | 低。当前为补充加成（+2分），不会单独决定信号方向 |
| **影响范围** | `config.py`中的VOL_COMP_BONUS和VOL_COMPRESSION_THRESHOLD；`entry_signal.py`中的`_score_22()`函数 |

|------|------|

| **决策内容** | 观察期从原定5月10日提前至5月8日深夜终止。数据立即定格。引擎停止运行 |

## 三、AI协作决策

### AID-1：三方论证作为标准方案验证流程

| 维度 | 内容 |
|------|------|
| **决策日期** | 2026-05-06 |
| **决策内容** | 任何重要方案和诊断结论必须在至少两个独立AI之间交叉验证。验证数据（系统日志、数据库记录、配置文件）由OpenClaw提取并提供给其他AI。DeepSeek负责方案整合和长期决策连贯性，ChatGPT负责方向校准和方法论校核。人类最终确认所有关键决策 |
| **决策理由** | Exit体系设计、Ghost Position诊断等多次交叉验证中，每个AI独立发现的盲区被另一个AI补充。ChatGPT发现Exit设计方案中的参数冲突，OpenClaw发现代码层面的多持仓并发问题，DeepSeek确认了Exit优先级和整体架构方向。任何单一AI的分析都可能存在盲区 |
| **替代方案** | ①只用单一AI（被否决——多次交叉验证证明盲区互补）。②所有方案过所有AI（过度设计——时间成本过高）。③AI直接决策不经过人类确认（被否决——AI不能替代项目负责人的责任判断） |
| **风险评估** | 低。交叉验证时间增加（每次方案需要多轮讨论），但方案质量显著提升 |
| **影响范围** | 所有重要方案讨论和诊断确认的工作流程 |

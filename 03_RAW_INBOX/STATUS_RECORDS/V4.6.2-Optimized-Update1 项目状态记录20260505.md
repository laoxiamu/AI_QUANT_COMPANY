### 九、故障经验记录



**说明**：以下记录从 V4\.6\.2\-Optimized\-Update1 定稿后，在系统运行、配置调整和策略验证过程中遇到的所有异常事件。每条经验包含故障表现、根因分析、处理过程和关键教训。





#### EXP\-1：Kimi 海外版凌晨执行任务后不回复



|维度|详情|
|---|---|
|**时间**|2026年5月3日 凌晨 01:24 左右|
|**触发操作**|用户在 OpenClaw 面板通过 Main Agent（绑定海外版 Kimi K2\.6，`moonshot\-intl`，`api\.moonshot\.ai`）发送了 Hermes 配置检查任务（四个 shell 命令：检查 SOUL\.md、MEMORY\.md、nudge\_interval、compression 配置）|
|**故障表现**|海外版 Kimi 正确执行了四个工具调用，返回了 `SOUL\.md不存在`、`MEMORY\.md不存在`、`config\.yaml不存在`、`compression配置未找到` 的结果。之后用户追问三次均无响应，切换到 DeepSeek 模型的其他 Agent 则正常回复|
|**排查过程**|① 初期怀疑是上下文处理中断，尝试 `/clear`（致历史记录丢失）② 检查 Kimi API 连通性，curl 测试返回 HTTP 200 正常 ③ 多次修改 openclaw\.json 配置文件，反复重启 Gateway ④ 发现配置文件中有两个同名 Kimi Provider（国内版\+海外版），怀疑是配置冲突 ⑤ 最终在凌晨约 3:00\-4:00 之间自动恢复|
|**根因确认**|海外版 Kimi API 服务端发生了**短暂的 API 服务中断**（约为数十分钟到数小时），与该服务的特定端点或路由有关。在此期间，OpenClaw 请求被挂起或连接被拒绝，表现为超时。当 API 服务恢复后，连接自动恢复正常|
|**处理过程**|用户主要等待服务自然恢复；期间通过 `openclaw gateway restart` 重启了服务网关，并尝试通过修改配置文件解决，但实际并未加快恢复进程|
|**关键教训**|① **排查 API 故障时，先用裸 curl 验证 API 本身的连通性，再查 OpenClaw 配置** ② 不要急于修改配置文件或执行大量修复脚本，先区分是 API 故障还是配置故障 ③ **添加新模型时应立即配置 fallback 机制**，防止单点故障导致主 Agent 完全不可用|





#### EXP\-2：Kimi 上下文超限陷入死循环



|维度|详情|
|---|---|
|**时间**|2026年5月3日 夜间至5月4日 凌晨|
|**触发操作**|用户在与 Kimi K2\.6 的长时间对话中，陆续上传了大量项目方案文档、变动的清单文件和日志内容，导致对话上下文逐渐接近模型的窗口上限|
|**故障表现**|当上下文占用达到 100% 时，Kimi 开始对用户的提问**完全不响应**，聊天界面上持续显示“等待回复”或超时错误。用户发出 `/new`、`/clear` 等命令均无效，陷入死锁|
|**根因分析**|**Kimi K2\.6 的特殊错误处理机制**：当 OpenAI 等模型遇到上下文超限时，会返回 \&\#34;context length exceeded\&\#34; 错误；但 Kimi 遇到类似情况时**不返回任何错误信息，直接进入超时状态**。OpenClaw 的自动压缩机制需要模型返回溢出错误才会触发，因此无法自动清理上下文。这个死锁只能通过重启 Gateway 或开启全新会话来跳出|
|**处理过程**|① 用户尝试切换模型到 DeepSeek，但 Gateway 本身已卡死，切换无效 ② 通过 `openclaw gateway restart` 重启解决 ③ 事后与 OpenClaw 合作排查，OpenClaw 查阅官方 Schema 后部署了修正版的 compaction 配置|
|**修正内容**|在 `openclaw\.json` 中部署了专门的 compaction 策略：\&lt;br\&gt;\- `mode: \&\#34;safeguard\&\#34;`（更激进的保护模式，修正了不存在的 `\&\#34;auto\&\#34;` 模式）\&lt;br\&gt;\- `reserveTokens: 60000`（预留 60K tokens）\&lt;br\&gt;\- `keepRecentTokens: 40000`（保护最近 40K 上下文）\&lt;br\&gt;\- `reserveTokensFloor: 40000`（下限保护）\&lt;br\&gt;\- `maxActiveTranscriptBytes: 1048576`（\~1MB 硬上限）\&lt;br\&gt;\- `truncateAfterCompaction: true`（主动截断）|
|**关键教训**|① **不同模型对超限的处理方式不同**，不能假设都会返回标准错误 ② **Kimi K2\.6 的沉默超时是已知特征**，需要通过独立的超限检测和主动清理机制来解决 ③ 长时间文档同步任务应拆分为独立会话，避免单个会话上下文无限膨胀|





#### EXP\-3：Gateway 卡死导致所有模型不回复



|维度|详情|
|---|---|
|**时间**|2026年5月3日 凌晨和5月4日 凌晨，反复出现|
|**故障表现**|OpenClaw 面板中**所有模型都不回复**（包括 Kimi 和 DeepSeek），切换模型无效|
|**根因分析**|Gateway 进程内部被**大量 Kimi 超时请求积压在处理队列中**，导致所有新请求都被阻塞，包括发给 DeepSeek 的请求也无法被处理|
|**处理过程**|每次通过 `openclaw gateway restart` 重启后立即恢复|
|**关键教训**|Gateway 重启是万能解药但不是根本解决方案。根本问题在于 Kimi 超时后会阻塞请求队列，需要在 Gateway 层面设置请求超时丢弃机制来防止此类连锁反应|

#### EXP\-5：引擎日志被覆盖，历史记录丢失



|维度|详情|
|---|---|
|**时间**|2026年5月4日 00:24|

|**故障表现**|引擎重启后，之前从 4月26日至5月3日的历史日志**全部消失**。`tail \-3000 /opt/v4/logs/strategy\_runner\.log` 只显示当天 00:24 之后的新日志|
|**根因分析**|Shell 重定向符号 `\&gt;` 是**覆盖重定向**——执行时会先把目标文件的内容全部清空（即截断到零字节），再从零开始写入新内容。正确的做法是使用 `\&gt;\&gt;` **追加重定向**——保留原有内容，把新日志追加到文件末尾|

|**关键教训**|① **`\&gt;`**** 和 ****`\&gt;\&gt;`**** 的区别至关重要**：`\&gt; file\.log` = 清空后重写（旧内容永远消失），`\&gt;\&gt; file\.log` = 不碰旧内容直接追加 ② 引擎重启命令已标准化为 `\&gt;\&gt;` 模式 ③ systemd 服务配置中已指定 `StandardOutput=append:`，从源头上杜绝覆盖问题|





#### EXP\-6：Kimi 60K\+ 上下文反复超时（非流式模式问题）



|维度|详情|
|---|---|
|**时间**|2026年5月4日 凌晨 01:41 左右|
|**发现过程**|OpenClaw 通过对比测试发现：Kimi 国内版在处理约 60K\+ tokens（约 108KB）的大请求时，**100% 触发 30 秒超时**（HTTP\_CODE=000）；而 DeepSeek 处理同样的约 120K tokens 请求，**3\.3 秒即返回 HTTP 200**。Kimi 处理小型请求（约 12 tokens）则表现为 1\.6 秒返回，正常|
|**根因分析**|OpenClaw 随后在 Kimi 官方文档 FAQ 中找到明确说明：**未启用流式输出（stream=true）时，Kimi 大模型必须等全部 token 生成完毕后，才发送第一个 HTTP 响应头（status\_code 和 header）给客户端**。在这段等待时间中，链路上某个网关（可能是 Kimi 服务端的负载均衡器，或 API 路径上的中间代理）因长时间未收到任何数据，触发 **idle timeout**，主动关闭了连接。上下文越大→Kimi 生成首字节的时间（TTFT）越长→越容易触及网关超时阈值|
|**处理过程**|在 `openclaw\.json` 中为两个 Kimi Provider（`moonshot` 国内版 \+ `moonshot\-intl` 海外版）均添加 `\&\#34;stream\&\#34;: true`，重启 Gateway 生效|
|**关键教训**|① **排查 API 问题时，官方文档 FAQ 是第一来源**——不要只依赖经验猜测 ② 非流式模式下的大上下文请求在 Kimi 上是已知性能瓶颈 ③ 启用流式模式后，模型可以边生成边返回，网关能持续收到数据，不会触发 idle timeout|

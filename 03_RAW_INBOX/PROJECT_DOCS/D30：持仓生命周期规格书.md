```Markdown
# 持仓生命周期规格书

**文档编号**：D30
**版本**：v1.0
**创建日期**：2026-05-09
**关联事故**：INC-2026-05-08（Ghost Position Incident）
**关联文档**：D29 SYSTEM_ARCHITECTURE， D10 INCIDENT_REPORT， D35 TECH_DEBT_REGISTER


## 一、文档目的

本规格书定义V4.6.2系统中每一笔持仓的完整生命周期。Ghost Position Incident的根本原因就是持仓生命周期从未被显式定义——系统不知道一个持仓应该经历哪些状态、在哪些状态之间可以转换、以及在任何时刻应该由哪个模块负责跟踪它。

本规格书是V5 Phase B Position Engine重写的核心任务书。


## 二、当前状态（V4.6.2 Frozen）与缺陷

### 2.1 当前实现

当前系统以 `symbol_direction` 作为持仓唯一标识，用 `_positions` 内存字典存储运行时状态。状态仅有一级隐式区分：`exit_price IS NULL`（活跃）vs `exit_price IS NOT NULL`（已平仓）。

### 2.2 致命缺陷

1. **无唯一标识**：`symbol_direction` 无法区分同币种同方向的不同交易
2. **状态不可审计**：49笔僵尸持仓只有 `exit_price IS NULL`，无法区分是“仍在持仓”还是“已被覆盖但未记录”
3. **覆盖无预警**：旧持仓被新信号覆盖时没有任何状态变更记录
4. **风控盲区**：Phase 1依赖不可靠的 `_positions` 字典作为输入源


## 三、目标状态机定义（V5规范）

### 3.1 状态枚举
```

PENDING → OPEN → ACTIVE → MANAGING → EXITED

↓           ↓

ORPHANED   INVALIDATED

↓

REPLACED

```Plain Text

| 状态 | 含义 | 进入条件 |
|------|------|---------|
| **PENDING** | 信号已通过Gateway裁决，等待订单成交确认 | Gateway返回APPROVED，`trade_id`已分配 |
| **OPEN** | 订单已成交，持仓进入市场。初始止损已设置 | 交易所返回FILLED或成交确认 |
| **ACTIVE** | 持仓正常存在于Phase 1的检查列表中。每轮检查止损/止盈条件 | 从OPEN自动转换 |
| **MANAGING** | 浮盈首次超过激活阈值。`peak_pnl`开始被跟踪。回撤止盈逻辑激活 | `current_pnl > trail_start` |
| **EXITED** | 已触发任一退出条件，已完成平仓。`exit_price`和`pnl`已记录 | 止损/止盈/时间退出/结构退出触发且平仓确认 |
| **INVALIDATED** | 生命周期断裂。持仓曾被系统管理但后来因覆盖、丢失或其他原因无法闭合 | 对账发现DB有但`_positions`无，或在未被检查的情况下超过N轮扫描 |
| **REPLACED** | 被同币种同方向的新持仓替换。**仅在明确的有记录的替换操作下允许此状态** | 先平旧仓（记录PnL）→再开新仓。旧仓标记为REPLACED，新仓从PENDING开始新生命周期 |
| **ORPHANED** | 持仓在DB中存在但超过N分钟未被任何风控检查过。这是Ghost Position的正式检测机制 | Reconciliation Loop发现DB有活跃持仓但`last_risk_check_at`超过阈值 |

### 3.2 状态转移规则

| 起点 | 终点 | 条件 | 触发者 |
|------|------|------|--------|
| PENDING | OPEN | 订单成交确认 | Execution Engine |
| PENDING | INVALIDATED | 订单被拒绝或超时未成交 | Execution Engine + Risk Engine |
| OPEN | ACTIVE | 成交后自动转换（止损已设置，Phase 1开始检查） | Risk Engine |
| ACTIVE | MANAGING | `current_pnl > trail_start` | SellManager |
| ACTIVE | EXITED | 止损/止盈/时间退出/结构退出触发，平仓已确认 | Risk Engine + SellManager |
| ACTIVE | ORPHANED | Reconciliation Loop发现此持仓超过N分钟未被风控检查 | Reconciliation Loop |
| MANAGING | EXITED | 回撤止盈触发 | SellManager |
| MANAGING | ACTIVE | 浮盈回落至激活阈值以下（回撤止盈已重置） | SellManager |
| 任何活跃状态 | INVALIDATED | 对账发现DB有但`_positions`无，或生命周期断裂无法通过 | Reconciliation Loop |
| EXITED | 禁止转换 | EXITED是终态，不可逆转 | — |
| INVALIDATED | 禁止转换 | INVALIDATED是终态，仅用于审计。不从统计中剔除 | — |
| REPLACED | 禁止转换 | REPLACED是终态，旧仓已关闭且盈亏已记录 | — |

### 3.3 显式禁止的状态转移

| 禁止路径 | 原因 |
|---------|------|
| ACTIVE → ACTIVE（通过覆盖） | 同向覆盖 = Ghost Position。禁止不经REPLACED状态的直接覆盖 |
| EXITED → 任何其他状态 | 已平仓的持仓不可复活 |
| ORPHANED → ACTIVE | 必须先通过人工审核确认此持仓是否仍然真实存在，才能重新激活 |


## 四、持仓唯一标识

### 4.1 V4缺陷

当前以 `symbol_direction`（如 `BTCUSDT_LONG`）作为持仓唯一标识。同一币种同一方向的所有交易被视为“同一个持仓”，无法区分第1笔和第N笔交易。

### 4.2 V5规范

**每个持仓必须有全局唯一的 `trade_id`**。格式：`{symbol}_{direction}_{timestamp}_{hash4}`——例如 `BTCUSDT_LONG_20260508_22:01:00_a3f2`。

| 维度 | 旧标识 | 新标识 |
|------|--------|--------|
| 标识字段 | `symbol_direction` | `trade_id` |
| 区分能力 | 无法区分同向的多笔交易 | 每笔交易独立追踪 |
| `log_trade()`匹配 | `ORDER BY timestamp DESC LIMIT 1`（粗粒度） | `WHERE trade_id = ?`（精确匹配） |
| 覆盖行为 | 静默覆盖旧持仓 | 必须先执行REPLACED转换→记录旧仓PnL→再开新仓 |


## 五、Phase 1 风控检查的标准流程（V5规范）

### 5.1 输入源

**必须从PostgreSQL权威源读取**：`SELECT * FROM signals WHERE exit_price IS NULL AND trade_status IN ('OPEN', 'ACTIVE', 'MANAGING') ORDER BY last_risk_check_at ASC`。

`_positions`字典降级为缓存，仅作性能优化用途。**禁止作为风控决策的输入源。**

### 5.2 检查流程

每轮Phase 1启动前：

1. **Reconciliation Loop**：PostgreSQL活跃持仓列表 vs `_positions`缓存。差异 → CRITICAL ALERT
2. **Orphan Detection**：任何 `last_risk_check_at` 超过2轮扫描 → 立即标记为ORPHANED + CRITICAL ALERT
3. **Risk Check**：按优先级顺序检查每个活跃持仓的退出条件（STOP_LOSS > TIME_EXIT > STRUCT_EXIT > TRAIL_EXIT）
4. **更新 `last_risk_check_at`**：每轮检查后更新此字段

### 5.3 退出优先级

| 优先顺序 | 退出类型 | 触发条件 |
|---------|---------|---------|
| 1 | STOP_LOSS | PnL ≤ -1.5%（当前），V5改为结构止损 |
| 2 | TIME_EXIT | `holding_time > max_holding_hours`（当前5天，5/10后24h） |
| 3 | TRAIL_EXIT | 从峰值回撤超过 `trail_back`（需`current_pnl > trail_start`先激活） |
| 4 | STRUCT_EXIT | 多头：`price < swing_low` 或 `price < MA20`（V5升级为SMC三重确认）；空头：对应条件取反 |


## 六、重启恢复流程

### 6.1 当前问题

引擎重启时 `_restore_positions_from_db()` 用 `DISTINCT ON (symbol, direction)` 只恢复每个币种+方向的最新一条。这导致同一方向的多笔历史记录中会有N-1笔被遗漏。Ghost Position Incident中两笔LONG正是在这个恢复函数调用后立即被覆盖逻辑再次消灭的。

### 6.2 V5规范

1. 重启时从PostgreSQL权威源读取所有 `trade_status IN ('OPEN', 'ACTIVE', 'MANAGING')` 的持仓
2. 每个持仓按 `trade_id` 恢复到 `_positions` 缓存中
3. `_positions` 的key改为 `trade_id`，而非 `symbol_direction`
4. 启动后立即执行完整的Reconciliation Loop，确认DB与缓存完全一致
5. 如有差异立即告警，不静默跳过


## 七、新信号到达时的处理规范

### 7.1 当前问题（同向覆盖）

已有 `BTCUSDT_LONG` 活跃持仓，新LONG信号到达。当前逻辑直接用新入场价覆盖 `_positions[BTCUSDT_LONG]`。旧仓盈亏静默丢失，旧仓从Phase 1检查中消失。

### 7.2 V5规范

当新信号到达时，检查是否已有同币种同方向的活跃持仓：

| 情况 | 处理 |
|------|------|
| 无同向活跃持仓 | 正常开仓。创建新Position，状态=PENDING→OPEN |
| 有同向活跃持仓 | **先平旧仓**：执行旧仓的REPLACED流程→记录旧仓PnL→旧仓状态=REPLACED。**再开新仓**：新仓创建新的trade_id，状态从PENDING开始新生命周期。**禁止覆盖，禁止跳过REPLACED状态直接替换** |


## 八、数据库字段需求

在现有 `signals` 表基础上新增以下字段：

| 字段 | 类型 | 用途 |
|------|------|------|
| `trade_id` | VARCHAR(64) UNIQUE | 全局唯一持仓标识 |
| `trade_status` | VARCHAR(20) | 状态枚举值（PENDING/OPEN/ACTIVE/MANAGING/EXITED/INVALIDATED/REPLACED/ORPHANED） |
| `max_profit` | NUMERIC(15,6) | 持仓期间最大浮盈 |
| `exit_reason` | VARCHAR(30) | 退出原因（STOP_LOSS/TIME_EXIT/TRAIL_EXIT/STRUCT_EXIT/MANUAL） |
| `holding_time` | INTEGER | 持仓持续秒数 |
| `last_risk_check_at` | TIMESTAMP | 最后一次被Phase 1检查的时间，用于孤儿检测 |
| `peak_pnl` | NUMERIC(15,6) | 用于回撤止盈的峰值跟踪 |
| `replaced_by_trade_id` | VARCHAR(64) | 如果此持仓被REPLACED，记录新持仓的trade_id |

## 九、与现有系统的兼容性

### 9.1 V4遗留数据

V4冻结时数据库中的全部历史记录（754条信号、49笔僵尸、5笔真实平仓）保留不动。但所有标记为 `exit_price IS NULL` 的旧记录需要做一次状态回填：能追溯到Phase 1完整检查记录的标记为 `ACTIVE`，不能确认的标记为 `INVALIDATED`（原因：Lifecycle Broken）。

### 9.2 数据口径

`INVALIDATED` 和 `ORPHANED` 状态的持仓从所有绩效统计（胜率、PnL、盈亏比）中排除。审计报告中标注排除原因和数量。
```

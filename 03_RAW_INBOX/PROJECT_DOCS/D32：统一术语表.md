## 一、交易与持仓

### 1.1 核心对象

| 术语 | 英文 | 定义 |
|------|------|------|
| **信号** | Signal | 策略引擎在每一轮扫描中产生的入场建议。包含币种、方向、评分、置信度、止损价、止盈价。信号不等于交易——信号需要经过微观结构过滤和Gateway裁决后才能转化为交易 |
| **交易** | Trade | 一个已通过全部过滤和裁决、进入市场的不可逆决策。每笔交易有全局唯一的`trade_id`。交易的生命周期从APPROVED开始，到EXITED/INVALIDATED/REPLACED结束 |
| **持仓** | Position | 一个时间段内币种+方向的风险敞口。同一币种同一方向只能有一个活跃持仓。持仓是风险管理的对象，交易是研究对象。**Trade ≠ Position**——一个Position可能经历多次Trade变更（如加仓、减仓、替换），但每次变更必须有完整的审计记录 |
| **入场** | Entry | 交易进入市场的动作。包含入场价、入场时间、入场信号评分、入场时的市场环境 |
| **出场** | Exit | 交易退出市场的动作。包含出场价、出场时间、退出原因（止损/止盈/时间退出/结构破坏/手动）、PnL |
| **平仓** | Close | 与Exit同义。特指已完成的退出动作 |

### 1.2 持仓状态

| 术语 | 定义 |
|------|------|
| **PENDING** | 信号已通过Gateway裁决，等待交易所订单成交确认。`trade_id`已分配 |
| **OPEN** | 订单已成交，持仓正式进入市场。初始止损已设置。此状态通常短暂存在，成交后自动转为ACTIVE |
| **ACTIVE** | 持仓正常存在于Phase 1的检查列表中。每轮扫描检查止损、止盈、时间退出、结构退出条件 |
| **MANAGING** | 浮盈首次超过激活阈值（`trail_start`）。`peak_pnl`开始被跟踪。回撤止盈逻辑激活 |
| **EXITED** | 已触发任一退出条件，已完成平仓。`exit_price`和`pnl`已记录。这是终态，不可逆转 |
| **INVALIDATED** | 生命周期断裂。持仓曾被系统管理但后来因覆盖、丢失或其他原因无法闭合。从所有统计中剔除。原因：Lifecycle Broken。不回推盈亏，不补历史 |
| **REPLACED** | 旧持仓被同币种同方向的新持仓有序替换。必须先平旧仓（记录PnL）→再开新仓。旧仓标记为REPLACED。REPLACED是终态 |
| **ORPHANED** | 持仓在DB中存在但超过N分钟未被任何风控检查过。这是Ghost Position的正式检测机制。触发后需人工审核确认持仓是否仍然真实存在 |

### 1.3 交易类型

| 术语 | 英文 | 定义 |
|------|------|------|
| **多头** | LONG | 买入，预期价格上涨。止损价低于入场价 |
| **空头** | SHORT | 卖出，预期价格下跌。止损价高于入场价 |
| **对冲** | Hedge | 同一币种同时持有LONG和SHORT。当前Gateway风控禁止对冲 |
| **僵尸持仓** | Zombie Position | 特指V4.6.2观察期内因同向覆盖而产生的49笔`exit_price IS NULL`但实际已被替换的持仓。全部标记为INVALIDATED |
| **幽灵持仓** | Ghost Position | 特指V4.6.2中593号和605号两笔LONG——真实持仓存在但风控系统不知道它存在的持仓。Ghost Position是Position Discovery Failure的特例 |


## 二、策略相关

### 2.1 评分系统

| 术语 | 英文 | 定义 |
|------|------|------|
| **22分制** | 22-Point Scoring | V4.6.2的手工规则评分系统。四个子项：大周期方向（4分）+ 中周期趋势（6分）+ 小周期位置（6分）+ 执行周期确认（6分）= 最高22分。当前已冻结为Baseline。IC≈50%（随机猜测），整体缺乏统计预测力 |
| **A级** | A-Level | 评分≥18（原为16，5/8回滚至18）。A级信号直接发送，跳过微观结构过滤 |
| **B级** | B-Level | 评分≥13。B级信号需经过微观结构过滤方可发送。当前所有真实平仓全为B级信号 |
| **C级** | C-Level | 评分<13。直接丢弃，不发送 |
| **波动压缩因子** | Vol Compression Factor | 唯一通过IC验证的微结构特征（BTC IC=+0.0579, SOL IC=+0.0816）。当前作为22分制的补充加成（+2分）。触发条件：当前20周期振幅标准差 < 历史均值×0.7 |

### 2.2 入场逻辑

| 术语 | 英文 | 定义 |
|------|------|------|
| **后视镜入场** | Rearview Entry | 所有评分项基于已收盘K线的历史指标（收盘价、均线、ADX），没有一个在前瞻性地判断价格会去哪里。这是V4.6.2 Entry的根本设计缺陷 |
| **顺大逆小** | HTF Bias / LTF Entry | 大周期定方向（日线/4h），小周期找入场点（15m/5m）。当前V4.6.2完全缺失此机制——方向判断只有一条线（price vs MA20 1h） |
| **关键位交易** | Key Level Trading | 基于前高前低、支撑阻力、成交量分布POC识别价格会停、会反转的位置。从"价格现在往哪个方向走"转向"价格在哪里会停"。这是V5 Entry重构的核心方向 |
| **事件驱动** | Event-Driven | 从基于固定时间间隔（每1h）的扫描模式，转向基于市场事件（关键位触及、结构突破、流动性扫单）的触发模式。属于V5远期架构升级方向 |
| **设置** | Setup | 一个完整的可重复交易模板。包含：大周期方向条件 + 入场触发条件 + 止损规则 + 退出规则 + 仓位规则。V5第一阶段只做一个Setup |


## 三、退出机制

| 术语 | 英文 | 定义 |
|------|------|------|
| **固定止损** | Fixed Stop Loss | 当前使用的止损方式：PnL ≤ -1.5%立即退出。在加密市场极度脆弱（1h K线极端波动3%-5%）。V5将改为结构止损 |
| **固定止盈** | Fixed Take Profit | 当前使用的止盈方式：PnL ≥ +3%立即退出。静态对称退出模型，主动截断盈利 |
| **结构止损** | Structure Stop | V5采用的止损方式。LONG：`stop = swing_low - 0.5×ATR`。SHORT：`stop = swing_high + 0.5×ATR`。止损线绑定市场结构而非固定百分比 |
| **时间退出** | Time Exit | 持仓时间超过预设阈值仍未触发任何其他退出条件，主动退出。用于清理"死单"（既达不到止盈也没触及止损）。当前代码值为5天（临时值），计划改为24h |
| **浮盈回撤止盈** | Trailing Profit Exit | 浮盈超过激活阈值（`trail_start`）后激活。当价格从`peak_pnl`回撤超过`trail_back`时退出。用于保护已到手的利润，让盈利的头寸多跑一会儿 |
| **结构破坏退出** | Structure Break Exit | 原方向的技术结构被反向突破，说明入场逻辑已失效。当前简化版为MA20破位。V5升级版为CHoCH+Liquidity+FVG三重确认 |
| **退出优先级** | Exit Priority | 多个退出条件同时满足时的判断顺序：STOP_LOSS > TIME_EXIT > TRAIL_EXIT > STRUCT_EXIT。止损永远是第一位 |


## 四、市场结构（SMC）

| 术语 | 英文 | 定义 |
|------|------|------|
| **FVG** | Fair Value Gap | 三根K线结构中的价格跳空区。Bullish FVG：`low[i+1] > high[i-1]`。Bearish FVG：`high[i+1] < low[i-1]`。理论：价格倾向于回补FVG区域。**注意**：这是待验证的行为假设，不是确定的交易规则。必须先通过IC验证方可引入策略 |
| **CHoCH** | Change of Character | 市场结构的转折点。Bullish CHoCH：价格突破前一个Lower High。Bearish CHoCH：价格跌破前一个Higher Low。用于识别趋势转换 |
| **流动性扫单** | Liquidity Sweep | 价格突破前高/前低后又收回。Bullish Sweep：跌破前低后快速收回。Bearish Sweep：突破前高后快速收回。被认为是"大资金在行动"的信号 |
| **BOS** | Break of Structure | 市场结构的延续性突破。Bullish BOS：价格突破并站稳Higher High。Bearish BOS：价格跌破并站稳Lower Low |
| **SMC宗教化** | SMC Dogma | 指不经验证地将FVG/CHoCH/Liquidity Sweep等SMC概念视为"确定性的交易规则"的错误倾向。ChatGPT的明确警告：很多SMC内容是后验解释，不是稳定alpha。正确做法是把SMC概念拆解为可统计验证的行为因子，然后做IC分析 |


## 五、系统架构

| 术语 | 英文 | 定义 |
|------|------|------|
| **权威状态源** | Authoritative State Source | 任何数据在任何时刻的唯一权威存储位置。当前确认为：持仓状态→PostgreSQL `signals`表。`_positions`字典和Redis缓存只是性能优化，不作为决策依据 |
| **Reconciliation Loop** | 对账循环 | 每轮Phase 1启动前的防御性检查：PostgreSQL中的活跃持仓列表与`_positions`缓存中的管理持仓列表逐笔对账。发现差异立即触发CRITICAL ALERT |
| **Orphan Detection** | 孤儿持仓检测 | 任何持仓超过N分钟未被风控检查，自动标记为ORPHANED并告警。防的就是Ghost Position |
| **Phase 1** | 平仓检查阶段 | 每轮扫描的第一部分：在生成新信号之前，先检查所有活跃持仓是否满足退出条件。当前依赖`_positions`字典（不安全），V5改为从PostgreSQL权威源读取 |
| **Gateway裁决** | Gateway Adjudication | OpenClaw Gateway在信号发送至交易所前的最终安全检查。包括熔断检查（回撤≥20%、日亏≥3%）、风险预算检查（总风险≤2%）、冲突检查（禁对冲）和资金分配（Kelly + Risk Parity）。Gateway是系统中唯一拥有交易否决权的组件 |
| **单向控制流** | One-Way Control Flow | 决策权与执行权分离的核心架构原则。Hermes只能建议不能下令，OpenClaw只能执行不能决策，Gateway是唯一裁决者 |


## 六、项目管理

| 术语 | 英文 | 定义 |
|------|------|------|
| **观察期** | Observation Period | 策略部署后的纸上谈兵验证阶段。核心约束：不改策略代码、不改评分逻辑、不改退出机制。目标是积累足够的平仓数据来评估策略效果 |
| **观察期失效** | Observation Invalidation | 观察期因核心逻辑被修改、数据记录不完整或Bug修复导致变量混杂，使得观察期数据无法提供具有统计意义的效果评估。V4.6.2观察期因7个核心文件被修改、同向覆盖导致49笔数据污染而失效 |
| **POST-MORTEM** | 事后审计 | 观察期结束后的完整复盘分析。不是总结报告，而是对系统缺陷、研发过程和决策质量的深度诊断。在当前项目中，观察期失效本身就是POST-MORTEM的核心发现 |
| **Baseline** | 基准线 | V4.6.2冻结后作为V5所有新setup的对照基准。任何一个V5 setup的回测结果必须与V4 Baseline对比，确认改善幅度 |
| **单Setup实验室** | Single Setup Lab | 先在一个币种、一种市场结构、一个交易设置上跑出正期望，再扩展。不搞多策略、不搞多币种、不搞复杂因子。V5第一阶段的唯一任务 |
| **技术债务** | Tech Debt | 已知的设计缺陷或临时方案，在特定条件下允许存在，但必须在计划时间内偿还。不同于Bug——Bug是代码逻辑错误需立即修复；技术债务是已知风险，允许在计划时间内解决 |
| **版本漂移** | Version Drift | 文档、代码、执行计划三者之间参数值不一致的现象。V4.6.2观察期内发现 `holding_days` 代码=5天、文档=24h、计划=待执行——典型的版本漂移。防范手段：参数标记规范（✅/⏳/📝/⚠️）+ 每次修改后强制执行一致性检查 |

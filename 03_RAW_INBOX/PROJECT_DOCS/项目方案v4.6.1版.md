# 第一章：系统总览与架构蓝图



## 1\.1 系统定位与设计哲学



### 1\.1\.1 系统定位

V4\.6\.1 不是一个单纯的量化交易策略脚本，而是一个**面向生产环境的、具备自进化能力的智能交易操作系统**。其核心定位如下：



|定位维度|描述|
|---|---|
|**功能定位**|加密货币永续合约的全自动、多策略、组合式交易系统。|
|**技术定位**|基于 Agent 架构的 **Alpha 策略与 Infrastructure 风控解耦** 的分布式系统。|
|**风险定位**|以 **生存为首要目标** 的防御型系统，单日亏损与最大回撤具有不可绕过的硬件级熔断。|



### 1\.1\.2 核心设计哲学

系统在 V4\.6 基础上进行架构重构，遵循以下三条铁律：



1. **单向控制流（One\-Way Control Flow）**  

    - 决策权与执行权必须分离。  

    - **Hermes（大脑）只能建议，不能下令。**  

    - **OpenClaw（手脚）只能执行，不能决策。**

        

2. **风险绝对优先（Risk\-First Architecture）**  

    - 策略层产生的任何信号，在触碰交易所 API 之前，**必须**经过独立的、不可绕过的 `Decision Gateway` 校验。  

    - 风控规则（如回撤 20% 清仓）属于系统层硬编码，**禁止任何 Agent 或 RL 模型修改**。

        

3. **唯一真实数据源（Single Source of Truth）**  

    - 系统内所有模块（OpenClaw、Hermes、Gateway）**不持有私有状态**。  

    - 所有状态（持仓、余额、信号历史）必须读写同一个 `State Center` \(Redis \+ PostgreSQL\)。

## 1\.3 数据流向与单向控制铁律



### 1\.3\.1 标准交易请求流（Happy Path）

下图展示了一个完整的、合规的交易信号从产生到成交的数据流：



```Plain Text
[1] 行情到达 (Market Data) → State Center (Redis 更新价格/OBI)
                              ↓
[2] OpenClaw 读取状态 → 运行 V4.6 策略逻辑 → 生成 Raw Signal
                              ↓
[3] OpenClaw 请求 Hermes 权重 (可选) → Hermes 返回 Weight Suggestion
                              ↓
[4] OpenClaw 组装 Final Signal → 发送至 Decision Gateway
                              ↓
[5] Decision Gateway 读取 State Center (验证风险、回撤、冲突) → 校验通过
                              ↓
[6] Decision Gateway 生成 Approved Order → 发送至 Execution Engine
                              ↓
[7] Execution Engine 签名并发送至交易所 → 更新 State Center (挂单状态)
```



### 1\.3\.2 单向控制强制约束（代码级防呆设计）

为了防止 Agent 失控或人为配置错误，系统在 **Docker 网络层** 和 **API 鉴权层** 实施了以下硬隔离：



|约束项|规则内容|违规后果|
|---|---|---|
|**API 密钥隔离**|Execution Engine 持有 **交易\+读取** 权限；Hermes 仅持有 **读取** 权限（只读 API Key）。|Hermes 无法构造有效的签名请求。|
|**网络防火墙**|Hermes 容器 **禁止访问** Execution Engine 容器的内部端口。|即使 Hermes 被入侵，也无法直接向执行层发送指令。|
|**代码接口限制**|OpenClaw 内部的 `execute\_order\(\)` 函数 **仅接收** `ApprovedOrder` 对象，且该对象必须包含 Gateway 的 **数字签名**。|绕过 Gateway 生成的对象无法通过执行层校验。|

## 2A\.1 选币雷达：三层动态候选池构建



策略层的第一要务是**缩小搜索范围**。全市场监控数千个交易对会消耗大量计算资源并引入噪声。V4\.6\.1 采用 **“三层动态候选池”** 机制，每日仅维护 6\-10 个高质量标的。



### 2A\.1\.1 候选池结构定义



|池类型|来源规则|数量上限|设计意图|
|---|---|---|---|
|**核心池**|固定配置：BTCUSDT, ETHUSDT, SOLUSDT|3|提供稳定的流动性基准，系统永不离线。|
|**动量池**|实时扫描：满足成交量放大、波动率扩张、区间突破条件的币种|3|捕捉**刚启动**的趋势行情，避免追高末期币种。|
|**极端池**|每日排序：24h 涨跌幅排名前 3 与后 3（流动性合格）|6|捕捉**情绪顶部的反转机会**或**恐慌底部的反弹机会**。|



**更新频率**：

- 核心池：系统启动时加载，不变更。

- 动量池：每 1 小时扫描一次全市场（或 15 分钟扫描一次 Top 50 成交量币种）。

- 极端池：每 4 小时或 UTC 0 点更新一次。

    

### 2A\.1\.2 流动性硬性过滤（准入机制）

**任何币种在进入上述任何池子之前，必须通过此过滤器。**

- **必要条件**：24 小时成交额 ≥ **50,000,000 USD**。

- **数据源**：CoinGecko / Binance 24hr Quote Volume。

- **动作**：未通过过滤的币种直接放入 **“禁做池”** ，不参与后续任何评分和信号计算。

- **例外处理**：若 BTC/ETH/SOL 的 24h 成交额低于此阈值（几乎不可能），系统将自动暂停交易并发出警报。

    

### 2A\.1\.3 选币评分规则（满分 17 分）

通过流动性过滤的币种将进入 **“选币雷达评分系统”** 。此评分旨在衡量该币种 **当前的“可交易性”**（即：是否容易进出、是否毛刺过多、是否趋势清晰）。



|维度|分值|量化标准与计算逻辑|
|---|---|---|
|**1\. 点差与深度**|0\-2|**2分**：买卖价差 \(Ask\-Bid\)/Mid \&lt; 0\.05%；\&lt;br\&gt;**1分**：\&lt; 0\.1%；\&lt;br\&gt;**0分**：≥ 0\.1%。\&lt;br\&gt;*\(数据来源：订单簿 Best Bid/Ask\)*|
|**2\. 插针程度**|0\-2|统计最近 4 小时 5m K线。\&lt;br\&gt;**2分**：上下影线 \&gt; 实体 2 倍的 K 线数量 ≤ 1；\&lt;br\&gt;**1分**：≤ 3；\&lt;br\&gt;**0分**：\&gt; 3。|
|**3\. 1h 趋势清晰度**|0\-3|**3分**：价格在 MA20 同一侧且 MA20 斜率绝对值 \&gt; 0\.1% per K；\&lt;br\&gt;**2分**：仅在同侧（斜率平）；\&lt;br\&gt;**1分**：频繁穿越 MA20；\&lt;br\&gt;**0分**：MA20 绝对走平（斜率 \&lt; 0\.01%）。|
|**4\. 15m 结构完整度**|0\-3|最近 15 根 K 线。\&lt;br\&gt;**3分**：高低点推进波次 ≥ 2 且无反向突破（Higher Highs）；\&lt;br\&gt;**2分**：波次 ≥ 1；\&lt;br\&gt;**0分**：无明显结构或已破位。|
|**5\. 5m 节奏感**|0\-2|最近 20 根 5m K 线涨跌幅的标准差。\&lt;br\&gt;**2分**：标准差 \&lt; 0\.2%（走势流畅）；\&lt;br\&gt;**1分**：\&lt; 0\.5%；\&lt;br\&gt;**0分**：≥ 0\.5%（杂乱无章）。|
|**6\. 波动率适配性**|0\-2|ATR\(14\) / 当前价格。\&lt;br\&gt;**2分**：比值在 **0\.3% \~ 1\.5%** 之间（既有波动空间，又不易插针）；\&lt;br\&gt;**1分**：0\.15%\~0\.3% 或 1\.5%\~2\.5%；\&lt;br\&gt;**0分**：\&lt; 0\.15%（死币）或 \&gt; 2\.5%（极度危险）。|
|**7\. 相对强弱 \(vs BTC\)**|0\-2|计算该币种与 BTC 的 1h 涨跌幅差值。\&lt;br\&gt;**2分**：差值 \&gt; \+2%（显著强于大盘，做多备选）或 \&lt; \-2%（显著弱于大盘，做空备选）；\&lt;br\&gt;**1分**：差值在 \-2% 到 \+2% 之间；\&lt;br\&gt;**0分**：无特殊 Alpha。|
|**8\. 消息干扰风险**|0\-1|基于消息面模块输出。\&lt;br\&gt;**1分**：最近 1 小时无重大负面/监管消息；\&lt;br\&gt;**0分**：存在未消散的重大消息影响。|



**评级与动作映射**：

- **A 级 \(14\-17 分\)**：放入 **主做池**。具备极佳的交易条件。

- **B 级 \(11\-13 分\)**：放入 **备选池**。若主做池无信号，可考虑轻仓介入。

- **C 级 \(≤10 分\)**：放入 **禁做池**。即便出现技术信号，策略层也不予理睬。

## 2A\.2 市场环境分类（技术面量化）



策略层在产生信号前，必须对 **整体大盘背景** 进行定性。不同环境适用不同的子策略（趋势/回调/震荡）。



### 2A\.2\.1 技术环境判定矩阵

系统通过 1 小时级别的指标组合进行判定，采用 **“满足 2 条以上即确认”** 的多数决原则。



|环境名称|量化条件（满足 ≥2 条触发）|对应子策略倾向|
|---|---|---|
|**强趋势日**|① MA20 斜率 \&gt; 0\.1%；\&lt;br\&gt;② ADX\(14\) \&gt; 25；\&lt;br\&gt;③ BOLL 带宽 \&gt; 过去 20 期均值 × 1\.2。|**TrendStrategy**|
|**趋势回调日**|① 1h 方向明确（ADX 15\-25）；\&lt;br\&gt;② 15m 出现回撤且未破大周期结构低点；\&lt;br\&gt;③ 5m 回踩 MA20 后反弹。|**PullbackStrategy**|
|**震荡日**|① MA20 斜率绝对值 \&lt; 0\.05%；\&lt;br\&gt;② ADX\(14\) \&lt; 20；\&lt;br\&gt;③ BOLL 带宽收窄至 20 期均值的 0\.8 倍以下。|**RangeStrategy**|
|**情绪极端日**|① 最近 1 小时振幅 \&gt; 5%；\&lt;br\&gt;② 1m/5m K线连续 3 根反向吞没；\&lt;br\&gt;③ 成交量暴增 200% 但价格原地踏步。|**禁止开仓，等待**|



### 2A\.2\.2 消息面强度评分算法

消息面模块独立运行，输出 **0\-10 分** 的强度评分。此评分将影响 `Environment Agent` 的最终判断。



**核心计算公式**：

```Plain Text
Score = Base_Weight × Time_Decay × (1 + Sentiment_Coefficient)
```

**变量定义**：

1. **Base\_Weight \(基础权重 1\-10\)**：

    - 10 分：美联储议息决议、SEC 监管重锤、交易所被盗。

    - 7 分：头部项目方重大升级/漏洞、CPI 数据大幅偏离预期。

    - 4 分：一般性项目公告、知名 KOL 喊单、常规宏观数据发布。

    - 1 分：日常新闻、无意义社交媒体讨论。

        

2. **Time\_Decay \(时效衰减\)**：

    - 公式：`0\.5 ^ \(Hours\_Passed / Halflife\)`

    - 参数：半衰期 `Halflife = 2` 小时。

    - 示例：4 小时前的消息，强度衰减为原始的 25%。

        

3. **Sentiment\_Coefficient \(情感系数 ±0\.2\)**：

    - 基于 NLP 情感分析结果。利多 \+0\.2，利空 \-0\.2（利空同样是强驱动，不降权）。

        

**关键修正规则（防噪音）**：

- **利多不涨**：若消息利多但价格 15 分钟内未上涨或反而下跌，**强度降级 50%**（视为利多出尽或骗炮）。

- **利空不跌**：若消息利空但价格拒绝下跌，**强度降级 50%**（视为底部承接强）。

    

### 2A\.2\.3 环境综合判定表（融合技术面与消息面）

策略层最终使用的环境标签由以下逻辑生成：



|最终环境标签|触发逻辑|策略响应|
|---|---|---|
|**强趋势**|技术面 = 强趋势 **且** 消息面强度 \&lt; 4|允许使用趋势策略正常仓位。|
|**消息驱动**|消息面强度 **≥ 6**|**切换至 NewsStrategy**，忽略部分技术指标，专注超短线动能。|
|**极端避险**|技术面 = 极端 **或** 消息面强度 ≥ 9|**强制空仓等待**，仅允许平仓，禁止开新仓。|
|**震荡**|技术面 = 震荡 **且** 消息面强度 \&lt; 4|允许使用网格/反转策略轻仓操作。|

# 第二章\-B：策略层详解——信号评分与微观结构

本章将详细展开 **多空强弱评分体系**、**三周期 22 分制入场信号** 以及 **HFT\-lite 微观结构层** 的完整算法。

## 2B\.1 多空强弱评分体系（满分 10 分）



在确定了可交易的币种池和市场环境后，策略层需要对 **特定方向** 的强度进行精细化评估。该评分决定了信号是“强信号”还是“勉强信号”，直接影响后续仓位系数。



### 2B\.1\.1 多头强势评分表（做多专用）



|维度|分值|量化标准与计算逻辑|
|---|---|---|
|**1\. 回踩浅**|0\-2|计算最近一次回调幅度 / ATR\(14\)。\&lt;br\&gt;**2分**：回调 \&lt; 0\.5 × ATR（强势横盘代替下跌）；\&lt;br\&gt;**1分**：0\.5 \~ 1\.0 × ATR（正常回踩）；\&lt;br\&gt;**0分**：\&gt; 1\.0 × ATR（深度回调，多头结构受损）。|
|**2\. 修复快**|0\-2|从回调低点到突破前高所用的 15m K线根数。\&lt;br\&gt;**2分**：≤ 5 根 K线（V型反转）；\&lt;br\&gt;**1分**：6\-15 根；\&lt;br\&gt;**0分**：\&gt; 15 根或未创新高。|
|**3\. 创高能力**|0\-2|最近 10 根 1h K线中，连续创出更高高点的次数。\&lt;br\&gt;**2分**：≥ 3 次（趋势强劲）；\&lt;br\&gt;**1分**：1\-2 次；\&lt;br\&gt;**0分**：无新高。|
|**4\. 量价配合**|0\-2|对比上涨 K 线均量与下跌 K 线均量（最近 10 根）。\&lt;br\&gt;**2分**：上涨均量 \&gt; 下跌均量 × 1\.5 **且** 回调均量 \&lt; 上涨均量 × 0\.8；\&lt;br\&gt;**1分**：仅满足其一；\&lt;br\&gt;**0分**：放量下跌或缩量上涨。|
|**5\. 相对 BTC 强度**|0\-2|计算该币种与 BTC 的 1h 涨跌幅差值。\&lt;br\&gt;**2分**：差值 \&gt; \+1%（显著领涨）；\&lt;br\&gt;**1分**：差值在 \-1% \~ \+1% 之间；\&lt;br\&gt;**0分**：差值 \&lt; \-1%（弱于大盘）。|



**评级映射**：

- **≥ 8 分**：强多候选，可进入下一环节（三周期入场评分）。

- **5\-7 分**：普通多候选，降低预期仓位。

- **≤ 4 分**：不适合主动做多，信号直接丢弃。

    

### 2B\.1\.2 空头弱势评分表（做空专用）



|维度|分值|量化标准与计算逻辑|
|---|---|---|
|**1\. 反弹弱**|0\-2|计算最近一次反弹幅度 / ATR\(14\)。\&lt;br\&gt;**2分**：反弹 \&lt; 0\.3 × ATR（极弱，多头无力）；\&lt;br\&gt;**1分**：0\.3 \~ 0\.7 × ATR；\&lt;br\&gt;**0分**：\&gt; 0\.7 × ATR（反弹过强，空头风险大）。|
|**2\. 破低快**|0\-2|最近 10 根 1h K线中，连续创出更低低点的次数。\&lt;br\&gt;**2分**：≥ 3 次（下跌趋势流畅）；\&lt;br\&gt;**1分**：1\-2 次；\&lt;br\&gt;**0分**：未创新低。|
|**3\. 量价配合**|0\-2|对比下跌 K 线均量与反弹 K 线均量（最近 10 根）。\&lt;br\&gt;**2分**：下跌均量 \&gt; 反弹均量 × 1\.5 **且** 反弹均量 \&lt; 下跌均量 × 0\.8；\&lt;br\&gt;**1分**：仅满足其一；\&lt;br\&gt;**0分**：放量反弹或缩量阴跌。|
|**4\. 相对 BTC 强度**|0\-2|**2分**：该币种跌幅 \&gt; BTC 跌幅 \+1%（显著领跌）；\&lt;br\&gt;**1分**：差值在 ±1% 之间；\&lt;br\&gt;**0分**：跌幅小于 BTC（相对抗跌，不适合空）。|
|**5\. 中轨压制**|0\-2|价格与 BOLL\(20,2\) 中轨的关系。\&lt;br\&gt;**2分**：连续 5 根 K 线收盘价位于中轨下方且受压制；\&lt;br\&gt;**1分**：位于中轨下方但有穿刺；\&lt;br\&gt;**0分**：站上中轨。|



**评级映射**：

- **≥ 8 分**：强空候选。

- **5\-7 分**：普通空候选。

- **≤ 4 分**：不适合主动做空。

## 2B\.2 三周期入场信号（硬过滤 \+ 22 分制评分）



这是 V4\.6\.1 策略层最核心的信号产生器。它必须在 **大、中、小** 三个时间周期上同时满足结构要求，才能输出交易信号。



### 2B\.2\.1 硬性过滤条件（必须全部满足）

以下四条缺一不可，若有一条不满足，**直接跳过评分环节，信号作废**。



|编号|硬过滤条件|量化检查逻辑|
|---|---|---|
|**F1**|大周期方向一致|价格必须在 4h MA20 的同一侧（做多在上方，做空在下方），且 MA20 斜率绝对值 \&gt; 0\.05% per K。|
|**F2**|中周期结构清晰|1h 周期最近 10 根 K 线至少形成 **2 个** 清晰的高低点推进（HH/HL 或 LH/LL），且未出现反向突破前低/前高。|
|**F3**|小周期到达关键区|15m 周期当前价格距离最近支撑/压力位（前高/前低、BOLL轨、成交密集区）的距离 \&lt; 0\.5 × ATR\(14\)。|
|**F4**|盈亏比预估值 ≥ 1\.5|基于预设止损（前低/前高外扩 0\.2ATR）与止盈（最近阻力/支撑），计算 `\(TP \- Entry\) / \(Entry \- SL\) ≥ 1\.5`。|



### 2B\.2\.2 评分细则（满分 22 分）

通过硬过滤后，信号进入详细评分环节。评分分为四个子项：大周期方向（4分）、中周期趋势（6分）、小周期位置（6分）、执行周期确认（6分）。



#### ① 大周期方向评分（满分 4 分）

|子项|分值|标准|
|---|---|---|
|价格在 MA20 同侧|\+1|符合 F1 即得分。|
|MA20 斜率强劲|\+1|斜率 \&gt; 0\.1%/K（趋势加速）。|
|高低点递增/递减|\+1|最近 4 根 4h K线形成 HH/HL（多）或 LH/LL（空）。|
|与中周期方向一致|\+1|1h 周期趋势方向与 4h 周期一致。|



#### ② 中周期趋势评分（满分 6 分）

|子项|分值|标准|
|---|---|---|
|价格在 MA20 同侧|\+1|1h 周期收盘价位于 MA20 趋势侧。|
|推进波次充足|\+2|最近 10 根 K 线有 ≥ 2 次推进波（\+2）；1 次（\+1）。|
|BOLL 中轨支撑/压制|\+1|价格持续在 1h BOLL 中轨同侧运行。|
|ADX 强度|\+1|ADX\(14\) \&gt; 25（趋势强劲）。|
|成交量支持|\+1|趋势方向 K 线均量 \&gt; 反向 K 线均量 × 1\.2。|



#### ③ 小周期位置评分（满分 6 分）

|子项|分值|标准|
|---|---|---|
|触及关键位|\+2|15m 价格触及 BOLL 上下轨 或 前高/前低（差值 \&lt; 0\.2×ATR）。|
|多重共振|\+1|同时触及两个以上关键位（如 BOLL 下轨 \+ 前低）。|
|回踩幅度合理|\+1|回踩幅度 \&lt; 1\.5×ATR 且未破结构前低/前高。|
|盈亏比达标|\+1|符合 F4 即得分。|
|距离关键位极近|\+1|距离关键位 \&lt; 0\.2×ATR（最佳狙击点）。|



#### ④ 执行周期评分（满分 6 分）

|子项|分值|标准|
|---|---|---|
|反转 K 线形态|\+2|实体占比 \&gt; 70%，影线 \&lt; 30%（Pin Bar / 吞没形态）。|
|假突破确认|\+1|价格突破关键位后 2 根 K 线内收回（诱多/诱空陷阱）。|
|成交量放大确认|\+1|该反转 K 线成交量 \&gt; 前 5 根均量 × 1\.5。|
|均线收复|\+1|价格重新站上/跌破 15m MA20。|
|指标辅助确认|\+1|RSI\(14\) \&gt; 50（多）或 \&lt; 50（空）**或** MACD 金叉/死叉（取其一）。|



### 2B\.2\.3 评级与仓位系数映射

|总分区间|等级|基础仓位系数|动作|
|---|---|---|---|
|**18 \- 22**|A|1\.0|正常开仓（允许满额风险预算）。|
|**14 \- 17**|B|0\.5|轻仓介入（风险预算减半）。|
|**10 \- 13**|C|0\.2|仅当无 A/B 信号且环境极佳时，允许微量测试。|
|**\&lt; 10**|D|0\.0|**放弃信号**。|

## 2B\.3 微观结构层：HFT\-lite 信号过滤器



此模块**不独立产生交易信号**，仅用于对上述三周期信号进行 **“入场时机确认”** 和 **“噪音过滤”**。



### 2B\.3\.1 核心特征定义与降噪处理



|特征名称|原始计算公式|**降噪处理（关键）**|
|---|---|---|
|**Order Book Imbalance \(OBI\)**|`\(Bid\_Vol\_10 \- Ask\_Vol\_10\) / \(Bid\_Vol\_10 \+ Ask\_Vol\_10\)`|计算 `EMA\(OBI, span=5\)`，要求 **连续 3 个 tick** 保持在阈值同侧。|
|**Order Flow Momentum**|`Market\_Buy\_Vol\_1s / Market\_Sell\_Vol\_1s`|比值 \&gt; 1\.5 且持续 0\.5 秒（防止单笔大单脉冲）。|
|**Liquidity Sweep**|前 5 档挂单量 0\.5 秒内减少 \&gt; 50%，且价格跳变 \&gt; 0\.05%|若检测到扫单，**强制将该信号置为 ****`pending`**，等待 2 秒后重新评估 OBI。|



### 2B\.3\.2 两阶段过滤逻辑



**阶段一：入场允许性过滤（Gateway 前置）**

```Python
# 伪代码逻辑
if macro_signal.direction == "LONG":
    if smoothed_OBI > 0.15 or order_flow_bullish:
        allow_entry = True
    else:
        allow_entry = False  # 微观不配合，延迟最多 5 秒，否则丢弃信号

if macro_signal.direction == "SHORT":
    if smoothed_OBI < -0.15 or order_flow_bearish:
        allow_entry = True
    else:
        allow_entry = False
```



**阶段二：入场价格优化（Execution 层执行）**

- 若通过阶段一，执行层将以 **Maker 策略** 挂单。

- **价差保护**：若买卖价差 \&gt; 0\.05%，挂单价 = `\(Best\_Bid \+ Best\_Ask\) / 2`。

- **超时熔断**：若 2 秒内价差未缩窄至 0\.05% 以内，转为市价单（但需风控层额外确认滑点）。

    

### 2B\.3\.3 微观层专属风控参数

|参数项|值|说明|
|---|---|---|
|信号有效期|5 秒|微观信号发出后 5 秒内未执行则作废。|
|重复触发限制|10 秒内最多 1 次|防止高频反复触发同一币种。|
|滑点作废阈值|0\.1%|实际成交滑点超过此值，交易作废并报警。|

# 第三章：决策与风控层——风控裁决与资金分配

本章是系统从“信号”到“实际仓位”的核心转换枢纽，包含 Final Risk Gate 的完整裁决逻辑、Kelly 公式推导、Risk Parity 相关性惩罚矩阵算法。

## 3\.1 Final Risk Gate（最终裁决器）完整逻辑



### 3\.1\.1 模块定位

`Decision Gateway` 是系统中 **唯一拥有交易否决权** 的组件。策略层产生的任何信号，无论评分多高、微观结构多配合，**必须经过此网关校验**，否则无法到达执行层。



**输入数据源**：

|输入项|来源|格式|
|---|---|---|
|`Signal`|OpenClaw 策略层|包含 symbol、direction、score、confidence、entry\_zone、stop\_loss、tp1|
|`State`|State Center \(Redis\)|实时持仓列表、当前总风险占用、当日已实现亏损、权益曲线高点与当前回撤|
|`Weights`|Hermes Agent|各子策略的推荐权重（若无则为默认等权）|
|`Config`|system\_config\.json|风险上限参数、熔断阈值、相关性惩罚系数|



**输出**：

- `APPROVED`：生成 `ExecutionOrder` 发送至执行引擎。

- `REJECTED`：记录拒绝原因，丢弃信号。

- `ADJUSTED`：修改仓位大小后批准（如因相关性惩罚降低名义仓位）。

    

### 3\.1\.2 裁决逻辑判断树（强制执行顺序）



Gateway 按照以下 **优先级顺序** 逐条校验，**任一条件触发 REJECTED 则立即终止**，不再进行后续检查。



```Plain Text
[1] 账户级熔断检查
    ├── 当前回撤 (Peak - Current) / Peak ≥ 0.20 ？
    │   └── YES → REJECTED (Max Drawdown Liquidation)
    ├── 当日累计已实现亏损 ≥ 账户余额 × 0.03 ？
    │   └── YES → REJECTED (Daily Loss Stop)
    └── 消息风暴模式激活 (消息强度 ≥ 9 持续 1h) ？
        └── YES → 禁止新开仓，REJECTED (News Storm)

[2] 总风险预算检查
    ├── 计算当前已占用风险 = Σ(各持仓 |Entry - Stop| / Entry × 名义仓位 / 账户余额)
    ├── 计算新信号预估风险 = |Entry - Stop| / Entry × (单笔风险上限 1% × 评分仓位系数)
    └── 当前风险 + 新信号风险 > total_risk_cap ？
        └── YES → REJECTED (Risk Budget Exceeded)

[3] 信号置信度与冲突检查
    ├── Signal.score 对应的仓位系数 == 0 ？
    │   └── YES → REJECTED (Low Score)
    ├── Hermes 推荐该策略权重 < 0.1 ？
    │   └── YES → REJECTED (RL Downgrade)
    └── 与现有持仓方向相反且相关系数 > 0.7 ？
        └── YES → REJECTED (Hedging Conflict, 不允许对锁)

[4] 微观结构滑点预估检查
    └── 当前买卖价差 > 0.1% 且 OBI 未显著偏向信号方向 ？
        └── YES → REJECTED (Liquidity Insufficient)

[5] 通过全部检查
    └── 进入资金分配计算模块 → 生成 APPROVED ExecutionOrder
```

## 3\.2 资金分配模型：风险约束 Kelly \+ 相关性调整 Risk Parity



### 3\.2\.1 总风险预算（Total Risk Cap）的三层计算



**第一层：理论 Kelly 参考值**

```Plain Text
f_star = (b × p - q) / b
```

- `p` = 策略层滚动 50 笔交易的胜率（来自回测引擎）。

- `b` = 平均盈亏比 = avg\_win\_r / \|avg\_loss\_r\|（R 倍数口径）。

- `q` = 1 \- p。

    

**第二层：环境系数调节**

```Plain Text
Kelly_adj = f_star × Env_Coefficient
```

|环境标签|Env\_Coefficient|逻辑说明|
|---|---|---|
|强趋势|0\.5|允许使用理论值的一半。|
|趋势回调|0\.4|回调介入风险略高。|
|震荡|0\.25|反转策略胜率虽高但盈亏比受限。|
|消息驱动|0\.15|高不确定性，极度保守。|
|极端避险|0\.0|强制不交易。|



**第三层：硬性风险上限（来自 Monte Carlo 安全约束）**

```Plain Text
Total_Risk_Cap = min(
    Account_Balance × 0.02,                     // 铁律：2% 总风险上限
    Account_Balance × (1 - MonteCarlo_DD_P95)   // 模拟得出的安全边际
)
```

- `MonteCarlo\_DD\_P95`：每月离线模拟输出的 95% 分位数最大回撤（例如 0\.34）。

    

**最终总风险预算**：

```Plain Text
Total_Risk = min(Total_Risk_Cap, Account_Balance × Kelly_adj)
```



### 3\.2\.2 单币种风险平价基础权重



对于候选池中通过评分的币种，先计算其 **波动率倒数权重**：

```Plain Text
Raw_Weight_i = 1 / ATR_Percent_i
```

- `ATR\_Percent\_i = ATR\(14\)\_i / Price\_i`（波动率百分比）。

    

示例：BTC 波动率 1\.2%，ETH 波动率 1\.8%，SOL 波动率 2\.5%。

|币种|ATR%|Raw Weight|归一化后|
|---|---|---|---|
|BTC|1\.2%|83\.33|55\.2%|
|ETH|1\.8%|55\.56|36\.8%|
|SOL|2\.5%|40\.00|8\.0%|



### 3\.2\.3 相关性惩罚矩阵（Risk Parity 增强）



当同时持有多个币种时，需对高相关性币种进行风险打折，防止风险预算被重复计算。



**简化分组算法（推荐实现）**：



1. 计算所有候选币种与当前持仓币种的 **1h 收益率滚动相关系数（窗口 72 根 K 线）**。

2. 若任意两币种相关系数 **\&gt; 0\.8**，则归入同一 **“高相关组”**。

3. 对于组内币种，**组总风险预算乘以惩罚系数 0\.6**。

    

**计算公式**：

```Plain Text
Group_Risk_Budget = Total_Risk × (组内原始权重之和) × 0.6
```



**组内二次分配**：

组内各币种按原始权重比例瓜分 `Group\_Risk\_Budget`。



**示例场景**：

- 持仓：BTC（已占用风险 1\.0%）。

- 新信号：ETH（与 BTC 相关系数 0\.85）。

- ETH 原始权重占比 40%。

    

**计算过程**：

1. 检测到 BTC 与 ETH 相关系数 0\.85 \&gt; 0\.8，归为同一组。

2. 组内原始权重：BTC 60% \+ ETH 40% = 100%。

3. 组总预算 = 2% × 100% × 0\.6 = **1\.2%**（而非原本的 2%）。

4. ETH 实际可用风险 = 1\.2% × 40% = **0\.48%**（而非原本的 2% × 40% = 0\.8%）。

    

**无高相关组的情况**：

- 若新信号币种（如 SOL）与现有持仓相关系数均 \&lt; 0\.8，则直接使用原始 Risk Parity 权重分配，不受惩罚。

    

### 3\.2\.4 风险预算到名义仓位的转换



```Plain Text
Position_Size_i = (Risk_Budget_i × Account_Balance) / Stop_Loss_Percentage_i
```

- `Stop\_Loss\_Percentage\_i = \|Entry\_i \- Stop\_i\| / Entry\_i`。

    

**安全限制（硬性）**：

- 单币种名义仓位 ≤ 账户余额 × 10（杠杆 ≤ 10x）。

- 所有持仓总名义仓位 ≤ 账户余额 × 30。

- 若计算出的仓位超过限制，等比缩减所有持仓。

# 第四章：决策与风控层——动态仓位与熔断体系

本章将详细展开连续动态仓位控制函数的数学原理、夏普比率调整机制、波动率熔断规则，以及完整的硬性熔断体系清单。

## 4\.1 动态仓位控制系统：连续函数设计



### 4\.1\.1 设计目标

传统量化系统通常采用“分段式降仓”（如回撤 10% 降仓至 50%，回撤 15% 降至 25%）。这种硬切换存在两个问题：

- **边界突变**：在阈值附近，仓位系数跳跃导致策略行为不稳定。

- **缺乏平滑反馈**：无法精细反映账户健康状况的连续变化。

    

V4\.6\.1 采用 **连续指数衰减函数** 替代分段规则，使仓位系数成为回撤幅度的平滑函数。



### 4\.1\.2 基础公式

```Plain Text
Equity_Curve_Scale = exp(-k × Current_Drawdown)
```

**变量定义**：

- `Current\_Drawdown = \(Peak\_Equity \- Current\_Equity\) / Peak\_Equity`，取值范围 \[0, 1\)。

- `k`：衰减系数，推荐值 **5\.0**（可通过回测优化，范围通常 3\~8）。

    

**函数特性**：

|回撤 \(Drawdown\)|仓位系数 \(k=5\)|含义|
|---|---|---|
|0%|1\.000|满额风险预算。|
|5%|0\.779|轻度回撤，小幅降仓。|
|10%|0\.607|中度回撤，仓位降至约六成。|
|15%|0\.472|显著回撤，仓位减半。|
|20%|0\.368|严重回撤，仓位降至三分之一强。|
|25%|0\.287|接近强制清仓线（20% 时已触发熔断，此值仅作理论参考）。|



**优势**：

- 每一次微小的回撤增加都会立即反映在仓位缩减上，形成 **负反馈闭环**。

- 避免了分段规则在阈值处的“悬崖效应”。

    

### 4\.1\.3 收益状态调整：滚动夏普比率修正

回撤仅反映亏损幅度，但未考虑近期盈利质量。系统引入 **最近 20 笔交易的滚动夏普比率** 作为正向调节因子。



**计算逻辑**：

1. 取最近 20 笔已平仓交易的 **R 倍数** 序列（若不足 20 笔则用全部历史）。

2. 计算 `Recent\_Sharpe = mean\(R\_list\) / std\(R\_list\)`（无风险利率视为 0）。

3. 根据夏普值调整系数：

```Python
if Recent_Sharpe > 1.5:
    Adjustment_Factor = 1.2
elif Recent_Sharpe < 0.5:
    Adjustment_Factor = 0.7
else:
    Adjustment_Factor = 1.0
```



**应用**：

```Plain Text
Adjusted_Scale = Equity_Curve_Scale × Adjustment_Factor
```

**限制范围**：最终 `Adjusted\_Scale` 必须裁剪至 **\[0\.1, 1\.5\]** 区间。

- 下限 0\.1：确保极端回撤时仍有微量仓位参与市场（避免完全空仓后无法捕捉反转）。

- 上限 1\.5：在极佳盈利状态下允许适度放大仓位（但仍受总风险 2% 的硬约束）。

    

### 4\.1\.4 波动率熔断（市场环境紧急制动）

当市场出现极端波动时，无论账户状态如何，必须强制降低仓位以规避流动性危机和插针风险。



**触发指标**：

```Plain Text
Market_Volatility = ATR(14) on 5m / Current_Price
```

- 数据源：5 分钟 K 线，滚动计算近 14 根（约 1 小时 10 分钟）的平均真实波幅。

    

**熔断规则**：

|波动率阈值|动作|系数|
|---|---|---|
|\&lt; 4%|正常|1\.0|
|≥ 4%|触发波动率熔断|**0\.5**|



**应用**：

```Plain Text
Final_Scale = Adjusted_Scale × Volatility_Breaker_Coefficient
```



### 4\.1\.5 最终仓位系数整合

将上述三层调节因子与策略层基础仓位系数融合：



```Plain Text
Final_Position_Scale = Base_Position_Coefficient  (来自三周期评分等级 A=1.0, B=0.5, C=0.2)
                       × Final_Scale              (来自动态仓位控制)
                       × Strategy_Weight          (来自 Hermes RL 权重)
```



**最终名义仓位**：

```Plain Text
Final_Quantity = (Total_Risk_Budget × Final_Position_Scale) / Stop_Loss_Percentage
```

（此公式继承自第三章资金分配模型，并叠加本章的动态调节因子。）

## 4\.2 硬性熔断与破产保护体系



动态仓位控制属于 **“软调节”**，而本节定义的规则属于 **“硬熔断”**。硬熔断一旦触发，将 **强制覆盖** 所有软调节结果，直接终止交易或清仓。



### 4\.2\.1 账户级熔断（不可逆触发）



|熔断类型|触发条件|执行动作|恢复条件|
|---|---|---|---|
|**单日亏损熔断**|当日累计已实现亏损 ≥ 账户余额 × 3%|立即停止所有开仓操作；已挂限价单全部撤销；仅允许平仓或止损。|UTC 0 点自动重置。|
|**连续亏损暂停**|连续亏损 3 笔（每笔均为净亏损）|暂停交易 **30 分钟**，期间不产生新信号。|冷却时间结束自动恢复。|
|**连续亏损停盘**|连续亏损 5 笔|当日停止交易，视为单日亏损熔断。|UTC 0 点重置。|
|**最大回撤清仓**|账户回撤（从历史最高权益起算） ≥ **20%**|**市价全平所有持仓**；撤销所有挂单；禁止一切交易活动。|**需人工登录后台确认并手动解除**。|



### 4\.2\.2 消息面熔断



|熔断类型|触发条件|执行动作|
|---|---|---|
|**消息风暴模式**|消息面强度评分 **≥ 9** 且持续时间超过 **1 小时**|强制将现有持仓减仓至 **50%**；新开仓指令全部 `REJECTED`；仅允许止损或手动干预。|
|**极端消息紧急状态**|检测到关键词：“交易所被盗”、“稳定币脱锚”、“监管全面禁止”|立即触发 **最大回撤清仓** 流程（等同于回撤 20%）。|



### 4\.2\.3 策略级熔断



|熔断类型|触发条件|执行动作|
|---|---|---|
|**单策略禁用**|某子策略（Trend/Pullback/Range/News）连续亏损 **10 笔**|自动将该策略的 `Strategy\_Weight` 强制设为 0；发送警报至监控面板。|
|**单币种暂停**|某币种在 1 小时内触发滑点保护 **3 次**|暂停该币种交易 **1 小时**，并将该币种移出候选池至下一轮更新。|



### 4\.2\.4 破产保护与 Monte Carlo 联动



**破产定义**：

- 账户权益 \&lt; 初始权益 × 30%（即亏损 70%）。在系统内部，若触发此条件，系统将进入 **只读模式**，禁止任何 API 写操作。

    

**Monte Carlo 预警联动**：

- 每月离线运行的 Monte Carlo 引擎输出 `risk\_of\_ruin`（破产概率）。

- 若 `risk\_of\_ruin \&gt; 0\.05`（5%），系统将在下月 **自动下调总风险上限** 至 **1\.5%**（覆盖 `system\_config\.json` 中的 `max\_total\_portfolio\_risk`）。

- 此调整由 Decision Gateway 在每次启动时读取配置生效。

## 4\.3 动态仓位与熔断的协同工作流



下图展示了一个交易信号从产生到执行过程中，如何依次经过动态仓位调节与硬熔断检查：



```Plain Text
[Signal] 基础仓位系数 (A=1.0)
    ↓
[动态仓位控制]
    ├── 计算当前回撤 → exp(-k × DD) = 0.78
    ├── 读取滚动夏普 > 1.5 → × 1.2 = 0.936
    └── 市场波动率 3.2% < 4% → 无熔断
    → Final_Scale = 0.936
    ↓
[RL 权重融合] × Strategy_Weight (Trend=0.6) = 0.5616
    ↓
[硬熔断检查]
    ├── 回撤 12% < 20% → 通过
    ├── 当日亏损 1.2% < 3% → 通过
    ├── 无连续亏损 → 通过
```

## 5\.1 执行层：OpenClaw 订单管理规范



执行层是系统中 **唯一** 有权调用交易所交易 API 的模块。其职责是将 `Approved ExecutionOrder` 安全、低成本地转化为交易所成交单。



### 5\.1\.1 分批挂单算法（TWAP\-lite）



为降低大额市价单的滑点冲击，同时最大化获取 Maker 手续费返佣，执行层采用 **固定数量分批挂单** 策略。



**参数配置（来自 ****`system\_config\.json`****）**：

- `split\_orders`：3（将总数量拆分为 3 份）。

- `order\_interval\_bps`：5（每份限价单间隔 0\.05%，基于当前买一/卖一价计算）。

- `maker\_only`：true（优先使用 Post\-Only 限价单）。

    

**算法逻辑（以做多为例）**：

```Python
total_qty = approved_order.quantity
base_price = orderbook.best_bid  # 做多时参考买一价

for i in range(split_orders):
    # 每份数量 = 总数量 / 份数（最后一份包含余数）
    slice_qty = total_qty / split_orders
    
    # 价格递减挂单：第一份挂买一，后续挂单价格逐步降低以捕捉回调
    slice_price = base_price * (1 - i * order_interval_bps / 10000)
    
    # 发送 Post-Only 限价单
    place_order(symbol, "BUY", slice_qty, slice_price, post_only=True)
    
    # 记录挂单时间戳
    order_timestamps.append(now())
```



**撤单与重挂逻辑**：

- 若某份订单在 `order\_expiry\_seconds`（30 秒）后仍未成交，**撤销该订单**。

- 重新评估当前订单簿：

    - 若价差 \&gt; 0\.1%，放弃重挂，剩余数量转为市价单（需额外滑点确认）。

    - 若价差正常，以最新买一价重新挂单。

        

### 5\.1\.2 微观结构配合：OBI 动态撤单



在挂单等待成交期间，执行层持续监听订单簿的 `smoothed\_OBI`（来自第二章\-B 的 HFT\-lite 模块）。



**撤单触发条件**：

```Python
if order.direction == "LONG":
    if smoothed_OBI < -0.15:  # 卖压突然增强，原挂单价可能成为“接盘价”
        cancel_order(order.id)
        wait_for_obi_recovery()
```



**恢复逻辑**：

- 撤销后进入 **冷却观察期**（2 秒）。

- 若 2 秒内 `smoothed\_OBI \&gt; 0\.1` 恢复，则重新挂单；否则标记为 `microstructure\_fail`，剩余数量 **放弃执行**，上报风控。

    

### 5\.1\.3 滑点保护与异常上报



**滑点计算**：

```Plain Text
Slippage_bps = |Actual_Fill_Price - Expected_Price| / Expected_Price × 10000
```

- `Expected\_Price`：信号生成时的 `Best\_Bid`（做多）或 `Best\_Ask`（做空）。

    

**处理规则**：

|滑点范围 \(bps\)|动作|说明|
|---|---|---|
|≤ 10 bps \(0\.1%\)|正常成交|记录日志。|
|\&gt; 10 bps|**交易作废**|立即市价平仓该笔成交（若已成交），并报告 `Slippage\_Exception`。|
|\&gt; 50 bps|**触发警报**|暂停该币种交易 1 小时，并通知运维。|



### 5\.1\.4 执行层与 Gateway 的签名校验



为防止 OpenClaw 内部逻辑错误或被篡改，所有来自 Gateway 的 `ExecutionOrder` 必须携带 HMAC\-SHA256 签名。



**校验流程**：

```Python
def verify_order(order):
    payload = f"{order.id}:{order.symbol}:{order.quantity}:{order.limit_price}"
    expected_sig = hmac.new(SECRET_KEY, payload, hashlib.sha256).hexdigest()
    if order.signature != expected_sig:
        raise InvalidOrderSignature("Order tampered or from untrusted source")
```

## 5\.2 学习层：Hermes RL 自进化引擎



### 5\.2\.1 算法选型：Contextual Bandit \(Thompson Sampling\)



**弃用 Q\-Learning 的原因**：

- 金融市场是非平稳环境，状态转移概率随时间变化，Q 表的收敛性无法保证。

- Q\-Learning 的 ε\-greedy 探索策略在连续亏损时可能过度探索导致更大回撤。

    

**Thompson Sampling 优势**：

- 通过 Beta 分布天然平衡 **探索与利用**（不确定性高的策略自动获得更多尝试机会）。

- 收敛速度快，适合在线学习。

- 输出的是 **策略权重分布**，而非离散动作，更符合组合资金分配需求。

    

### 5\.2\.2 状态空间定义



Hermes 从 State Center 拉取以下特征向量，用于区分不同市场上下文：



|特征名|类型|取值范围|说明|
|---|---|---|---|
|`market\_env`|One\-Hot|4 维|当前环境：趋势/回调/震荡/消息驱动。|
|`volatility`|Float|0\~0\.10|当前 5m ATR% 归一化值。|
|`btc\_trend`|Int|\-1, 0, 1|BTC 1h MA20 斜率方向。|
|`news\_score`|Float|0\~10|当前消息面强度评分。|
|`drawdown`|Float|0\~0\.20|当前账户回撤百分比。|



**状态向量示例**：

```JSON
[1, 0, 0, 0, 0.018, 1, 2.3, 0.06]
```

（表示：强趋势环境，波动率 1\.8%，BTC 上涨，消息面平静，回撤 6%）



### 5\.2\.3 模型结构：多臂老虎机维护



对于每个 **离散化的状态簇**（通过 K\-Means 或决策树将连续状态分桶），Hermes 维护一个独立的 Beta 分布参数集。



```Python
class ContextualBandit:
    def __init__(self, n_strategies=4):
        self.strategies = ["Trend", "Pullback", "Range", "News"]
        # 每个策略维护一对 (alpha, beta)
        self.alpha = {s: 1.0 for s in self.strategies}
        self.beta  = {s: 1.0 for s in self.strategies}
    
    def get_weights(self):
        """返回归一化权重向量"""
        samples = {s: np.random.beta(self.alpha[s], self.beta[s]) 
                   for s in self.strategies}
        total = sum(samples.values())
        return {s: samples[s]/total for s in self.strategies}
    
    def update(self, strategy_name, reward):
        """根据奖励更新 Beta 分布参数"""
        if reward > 0:
            self.alpha[strategy_name] += reward
        else:
            self.beta[strategy_name] += abs(reward)
```



### 5\.2\.4 奖励函数设计



**核心公式**：

```Python
reward = trade_R - 0.5 * max(drawdown_increase, 0)
```

- `trade\_R`：该笔交易的 R 倍数（盈利金额 / 初始止损金额）。

- `drawdown\_increase`：该笔交易持仓期间，账户回撤的 **新增幅度**（若未创新高回撤则为 0）。

    

**设计意图**：

- 鼓励在 **不扩大账户回撤** 的前提下获取正 R 收益。

- 若某策略虽然盈利但过程剧烈回撤（如扛单后反转），其奖励将被惩罚项显著降低。

    

### 5\.2\.5 状态聚类与在线学习流程



由于状态空间连续，Hermes 需要将相似状态归类以共享学习经验。



**聚类方法**：

- 使用 **MiniBatchKMeans**（预设 20 个簇），每日凌晨用历史状态数据重新训练。

- 实时推理时，将当前状态向量映射到最近的簇 ID，使用该簇对应的 `ContextualBandit` 实例。

    

**完整学习循环**：

```Plain Text
[1] 策略层产生 Signal（标记为 Trend 策略）
[2] Decision Gateway 批准并执行
[3] 交易平仓后，计算 R 与回撤增量 → reward
[4] Hermes 获取开仓时刻的状态向量 → 映射到 Cluster_ID
[5] 更新 Cluster_ID 下 Trend 策略的 Alpha/Beta 参数
[6] 下一笔交易前，Gateway 调用 Hermes API 获取最新权重
```



### 5\.2\.6 权重输出 API 定义



Gateway 在裁决每一笔信号前，调用 Hermes 的内部接口：

```Plain Text
GET /v1/weights?env=trend&vol=0.018&btc=1&news=2.3&dd=0.06
```



**响应示例**：

```JSON
{
  "Trend": 0.52,
  "Pullback": 0.23,
  "Range": 0.15,
  "News": 0.10,
  "timestamp": "2026-04-21T10:30:00Z"
}
```



Gateway 将此权重与信号所属策略匹配，作为 **Final Risk Gate** 第 3 步（RL 权重校验）的输入。

## 5\.3 离线 Monte Carlo 引擎实现细节



Monte Carlo 引擎作为 Hermes 的离线子模块，每月运行一次，输出关键风险参数供下月使用。

### 5\.3\.3 路径模拟与破产统计

对每条模拟路径：

1. 每日根据环境从对应 R 分布中随机抽取一笔交易结果。

- `risk\_of\_ruin`：破产路径数 / 1000。

- `expected\_return`：平均年化收益。

    

**联动机制**：若 `risk\_of\_ruin \&gt; 0\.05`，自动更新 `system\_config\.json` 中的 `max\_total\_portfolio\_risk` 为 1\.5%（原 2%）。

## 6\.2 冷启动四阶段流程与验收标准



**禁止** 在部署后立即启用全自动交易。必须严格按照以下阶段推进，每个阶段有明确的 **通过标准**。



### 阶段 1：基础设施验证（1\-3 天）

- **状态**：所有容器启动，行情服务运行，**执行引擎的 API Key 使用只读权限**。

- **验收标准**：

    * [ ] Redis 与 PostgreSQL 健康检查通过。

    * [ ] `market\-data` 成功推送 BTC/ETH 订单簿数据至 Redis，延迟 \&lt; 200ms。

    * [ ] `openclaw\-core` 能够生成选币池与市场环境标签，并在日志中可见。

    * [ ] Grafana 大盘显示实时价格与 OBI 曲线。

    

### 阶段 2：纯规则 Paper Trading（2 周）

- **状态**：执行引擎仍使用只读 API，**所有信号仅记录日志，不实际下单**。

- **系统配置**：`system\_config\.json` 中设置 `paper\_trading: true`。

- **验收标准**：

    * [ ] 连续运行 14 天无服务崩溃或内存泄漏。

    * [ ] 日均信号产生数量在合理范围（5\-30 笔）。

    * [ ] 模拟权益曲线回撤 \&lt; 15%（基于 Paper 日志计算）。

    * [ ] 滑点模拟统计（基于历史订单簿回放）\&lt; 0\.1% 的订单占比 \&gt; 95%。

    

### 阶段 3：Hermes 影子模式（1 周）

- **状态**：启动 Hermes 容器，**仅读取数据、输出权重日志，不向 Gateway 发送权重**。Gateway 仍使用默认等权。

- **验收标准**：

    * [ ] Hermes 每日成功聚类状态并更新 Beta 参数。

    * [ ] 权重输出无明显异常（如某策略权重突然升至 0\.99）。

    * [ ] 对比纯规则权益曲线与 Hermes 建议权益曲线（模拟），后者回撤无明显放大。

    

### 阶段 4：小资金上线（10% 仓位限制）

- **状态**：执行引擎切换为交易 API Key。`system\_config\.json` 中设置 `max\_total\_portfolio\_risk: 0\.002`（0\.2% 总风险，约为正常的 1/10）。

- **验收标准**：

    * [ ] 首日实际滑点 \&lt; 0\.1%。

    * [ ] 实际成交与 Paper 模拟的偏差 \&lt; 10%。

    * [ ] 连续 7 个交易日无硬熔断触发。

    * [ ] 若连续盈利且回撤 \&lt; 5%，可逐步将风险上限调至 1%（需人工确认）。

### 6\.3\.2 关键 PostgreSQL 查询（供 Grafana 使用）



**① 当前账户权益与回撤**：

```SQL
SELECT 
    equity, 
    (peak_equity - equity) / peak_equity AS drawdown
FROM account_snapshots 
ORDER BY timestamp DESC 
LIMIT 1;
```



**② 当日累计已实现盈亏**：

```SQL
SELECT SUM(realized_pnl) AS daily_pnl
FROM trades
WHERE close_time >= CURRENT_DATE;
```



**③ 各子策略滚动胜率（近 50 笔）**：

```SQL
SELECT 
    strategy_name,
    SUM(CASE WHEN net_pnl > 0 THEN 1 ELSE 0 END)::float / COUNT(*) AS win_rate
FROM trades
WHERE close_time >= NOW() - INTERVAL '7 days'
GROUP BY strategy_name;
```



**④ 近 20 笔交易 R 倍数序列（用于夏普计算）**：

```SQL
SELECT 
    (net_pnl / ABS(initial_stop_loss_usd)) AS r_multiple
FROM trades
ORDER BY close_time DESC
LIMIT 20;
```



**⑤ 滑点监控（近 1 小时）**：

```SQL
SELECT 
    AVG(ABS(fill_price - expected_price) / expected_price) * 10000 AS avg_slippage_bps,
    MAX(ABS(fill_price - expected_price) / expected_price) * 10000 AS max_slippage_bps
FROM trades
WHERE open_time >= NOW() - INTERVAL '1 hour';
```

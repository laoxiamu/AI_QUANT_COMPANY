# 第一部分：系统总览与策略层



## 第一章：系统总览与架构蓝图



### 1\.1 系统定位与设计哲学



#### 1\.1\.1 系统定位

V4\.6\.2 是一个面向生产环境的、具备自进化能力的智能交易操作系统，不是一个单纯的量化策略脚本。



|定位维度|描述|
|---|---|
|**功能定位**|加密货币永续合约的全自动、多策略、组合式交易系统。|
|**技术定位**|基于 Agent 架构的 Alpha 策略与 Infrastructure 风控解耦的分布式系统。|
|**风险定位**|以生存为首要目标的防御型系统，单日亏损与最大回撤具有不可绕过的硬件级熔断。|
|**进化定位**|具备环境自适应策略路由与技能沉淀能力，越运行越匹配当前市场状态。|



#### 1\.1\.2 核心设计哲学

系统遵循以下四条铁律：



1. **单向控制流**  

    - 决策权与执行权分离。Hermes（大脑）只能建议，不能下令；OpenClaw（手脚）只能执行，不能决策。  

    - Decision Gateway 是唯一拥有否决权的组件。

        

2. **风险绝对优先**  

    - 任何信号在触碰交易所 API 前，必须经过独立的 Decision Gateway 校验。  

    - 风控规则属于系统层硬编码，禁止任何 Agent 或 RL 模型修改。

        

3. **唯一真实数据源**  

    - 所有模块不持有私有状态，必须读写同一个 State Center（Redis \+ PostgreSQL）。

        

4. **环境自适应策略路由**（优化版新增）  

    - 系统先判断市场状态（趋势/回调/震荡/极端），再选择对应的子策略。  

    - 避免单一趋势跟踪逻辑在震荡市失效，解决信号评分“趋势偏置”问题。

        

        

### 1\.2 分层架构全景图

### 1\.3 数据流向与单向控制铁律



#### 1\.3\.1 标准交易请求流



```Plain Text
[1] 行情到达 → State Center（Redis 更新价格/OBI）
       ↓
[2] OpenClaw Signal Agent 读取状态 → 运行策略逻辑 → 生成 Signal
       ↓
[3] OpenClaw Main Agent 请求 Hermes 权重 → Hermes 返回建议
       ↓
[4] OpenClaw 组装 Final Signal → 发送至 Decision Gateway
       ↓
[5] Decision Gateway 读取 State Center → 风控校验 → 裁决
       ↓
[6] Gateway 生成 Approved Order → 发送至 Execution Engine
       ↓
[7] Execution Engine 签名发送至交易所 → 更新 State Center
       ↓
[8] 交易平仓后，Hermes 读取结果 → 更新 RL 模型 / 写入记忆
```



#### 1\.3\.2 单向控制强制约束



|约束项|规则|违规后果|
|---|---|---|
|**API 密钥隔离**|Execution Engine 持有交易密钥；Hermes 仅持有只读密钥|Hermes 无法构造有效签名|
|**网络防火墙**|Hermes 容器禁止访问 Execution Engine 容器端口|即使入侵也无法直接下单|
|**代码接口限制**|OpenClaw 的 `execute\_order\(\)` 仅接收带 Gateway 签名的 ApprovedOrder|绕过 Gateway 的请求会被拒绝|





### 1\.4 核心组件职责速查表



|组件 ID|层级|核心职责|关键限制|
|---|---|---|---|
|**Market Data Service**|L1|行情采集、订单簿 WebSocket 推送、消息面爬虫|只读，禁止修改原始数据|
|**Execution Engine**|L1|将合法订单转化为交易所指令，滑点监控|唯一可触碰交易所交易 API|
|**State Center**|L1|Redis（实时状态）\+ PostgreSQL（持久化数据）|禁止模块本地缓存状态|
|**OpenClaw Core**|L2|策略信号生成、多 Agent 路由、订单拆分|禁止自行计算仓位大小|
|**Decision Gateway**|L3|风控裁决、资金分配、熔断触发|唯一拥有否决权|
|**Hermes Agent**|L4|RL 权重学习、技能沉淀、记忆管理、Monte Carlo|禁止直接下单，禁止写入关键字段|
|**FastAPI Proxy**|L4|模型转发（替代 Hermes HTTP API 功能）|仅监听 127\.0\.0\.1|





## 第二章：策略层详解



策略层是系统的信号发生器。在 V4\.6\.2 优化版中，策略层实现了**环境自适应**——先判断市场状态，再选择对应的子策略，解决了原版单一趋势跟踪在震荡市失效的问题。



### 2\.1 选币雷达：三层动态候选池 \+ 五因子加权评分



#### 2\.1\.1 三层动态候选池



|池类型|来源规则|数量上限|设计意图|
|---|---|---|---|
|**Core Pool**|BTCUSDT, ETHUSDT, SOLUSDT（固定）|3|稳定流动性基准，系统永不离线|
|**Momentum Pool**|同时满足：① 成交量 \&gt; 24h均量×1\.5；② ATR环比上升 \&gt;10%；③ 价格突破近20根1h K线区间|3|捕捉刚启动的趋势币|
|**Extreme Pool**|24h 涨跌幅排名前3及后3（流动性合格）|6|捕捉情绪顶/恐慌底|



总候选池每日动态更新，控制在 6\-10 个。



#### 2\.1\.2 流动性硬性过滤

- **必要条件**：24h 成交额 ≥ 50,000,000 USD。

- 未通过过滤的币种直接放入禁做池，不参与后续任何评分。

    

#### 2\.1\.3 五因子加权评分模型（满分 17 分）



|因子|权重|计算方式|说明|
|---|---|---|---|
|**Trend**（趋势因子）|0\.25|\(收盘价/MA20 \- 1\) × 100|衡量价格相对于均线的偏离程度|
|**Volatility**（波动率因子）|0\.20|\(High \- Low\) / Price|评估波幅的适应性，甜点区 0\.3%\-1\.5%|
|**Liquidity**（流动性因子）|0\.20|log\(24h成交额\)|对数压缩后参与横截面排序|
|**RS**（相对强弱因子）|0\.20|币种涨跌幅 \- BTC涨跌幅|衡量独立 Alpha|
|**Micro**（微观因子）|0\.15|买卖价差、深度不平衡|评估盘口质量|



**评级与动作映射**：

- A 级 \(14\-17分\)：主做池，满额风险预算

- B 级 \(11\-13分\)：备选池，轻仓介入

- C 级 \(≤10分\)：禁做池

    

### 2\.2 市场环境分类与策略路由



#### 2\.2\.1 技术环境判定矩阵



|环境名称|量化条件（满足 ≥2 条触发）|对应子策略倾向|
|---|---|---|
|**强趋势日**|① MA20 斜率 \&gt; 0\.1%/K；② ADX\(14\) \&gt; 25；③ BOLL 带宽 \&gt; 过去20期均值×1\.2|**TrendStrategy**|
|**趋势回调日**|① 1h 方向明确（ADX 15\-25）；② 15m 回撤未破大周期结构；③ 5m 回踩 MA20 后反弹|**PullbackStrategy**|
|**震荡日**|① MA20 斜率绝对值 \&lt; 0\.05%/K；② ADX \&lt; 20；③ BOLL 带宽收窄至 20 期均值的 0\.8 倍|**RangeStrategy**（区间反转）|
|**情绪极端日**|① 最近1h 振幅 \&gt; 5%；② 1m/5m K线连续3根反向吞没；③ 成交量暴增200%但价格原地踏步|**禁止开仓，等待**|



#### 2\.2\.2 消息面强度评分算法



`Score = BaseWeight × 0\.5^\(hours/halflife\) × \(1 ± sentiment\_coefficient\)`



- BaseWeight：1\-10（美联储决议 10、日常新闻 1）

- 半衰期：2小时

- 极端消息（强度≥9）→ 触发“消息风暴模式”，强制减仓至50%或空仓

    

**关键修正规则**：

- 利多不涨 → 强度降级 50%

- 利空不跌 → 强度降级 50%

    

#### 2\.2\.3 环境综合判定与策略路由



|最终环境|触发逻辑|策略响应|
|---|---|---|
|**强趋势**|技术面=强趋势 且 消息强度\&lt;4|TrendStrategy：趋势跟踪|
|**趋势回调**|技术面=回调 且 消息强度\&lt;5|PullbackStrategy：回调介入|
|**震荡**|技术面=震荡 且 消息强度\&lt;4|RangeStrategy：区间反转|
|**消息驱动**|消息强度≥6|NewsStrategy：超短线动能|
|**极端避险**|技术极端 或 消息强度≥9|空仓，禁止开仓|



**策略路由实现**：`environment\.py` 输出环境标签后，`entry\_signal\.py` 的 `\_score\_22\(\)` 根据环境选择评分逻辑——趋势市保留原22分制，震荡市切换为区间反转评分（接近区间下沿→做多，接近区间上沿→做空）。



### 2\.3 多空强弱评分（10分制）



#### 2\.3\.1 多头强势评分



|维度|分值|标准|
|---|---|---|
|回踩浅|0\-2|回调 \&lt; 0\.5×ATR → \+2|
|修复快|0\-2|突破前高 ≤ 5根K → \+2|
|创高能力|0\-2|连续新高 ≥3次 → \+2|
|上涨放量回踩缩量|0\-2|量比\&gt;1\.5 且 回踩量\&lt;0\.8 → \+2|
|相对BTC强|0\-2|涨幅差 \&gt;1% → \+2|



≥8分 → 强多候选。≤4分 → 不适合做多。



#### 2\.3\.2 空头弱势评分



|维度|分值|标准|
|---|---|---|
|反弹弱|0\-2|反弹 \&lt; 0\.3×ATR → \+2|
|破低快|0\-2|连续新低 ≥3次 → \+2|
|下跌放量反弹缩量|0\-2|量比\&gt;1\.5 且 反弹量\&lt;0\.8 → \+2|
|相对BTC弱|0\-2|跌幅 \&gt; BTC跌幅\+1% → \+2|
|中轨压制|0\-2|连续5根K收盘价在中轨下方 → \+2|



≥8分 → 强空候选。≤4分 → 不适合做空。



### 2\.4 三周期入场信号（硬过滤 \+ 22分制评分 \+ 环境自适应）



#### 2\.4\.1 硬过滤条件（必须全部满足）



|编号|条件|检查逻辑|
|---|---|---|
|F1|大周期方向一致|4h MA20同侧 且 斜率\&gt;0\.05%/K|
|F2|中周期结构清晰|1h 近10根K形成≥2个推进波，未反向突破|
|F3|小周期到达关键区|15m 距离支撑/压力 \&lt; 0\.5×ATR|
|F4|盈亏比预估值 ≥ 1\.5|\(TP\-Entry\)/\(Entry\-SL\) ≥ 1\.5|



#### 2\.4\.2 22分制评分细则



**① 大周期方向（4分）**

- 价格在MA20同侧：\+1

- MA20斜率\&gt;0\.1%/K：\+1

- 高低点递增/递减：\+1

- 与中周期方向一致：\+1

    

**② 中周期趋势（6分）**

- 价格在1h MA20同侧：\+1

- 推进波次≥2：\+2（1次\+1）

- BOLL中轨同侧支撑：\+1

- ADX\(14\)\&gt;25：\+1

- 趋势方向量\&gt;反向量×1\.2：\+1

    

**③ 小周期位置（6分）**

- 触及BOLL轨或关键位（差值\&lt;0\.2×ATR）：\+2

- 多重共振：\+1

- 回踩幅度\&lt;1\.5×ATR且未破结构：\+1

- 盈亏比≥1\.5：\+1

- 距离关键位\&lt;0\.2×ATR：\+1

    

**④ 执行周期确认（6分）**

- 反转K线（实体\&gt;70%，影线\&lt;30%）：\+2

- 假突破（突破后2根K内收回）：\+1

- 反转K线量\&gt;前5根均量×1\.5：\+1

- 价格重新站上/跌破15m MA20：\+1

- RSI\&gt;50（多）/RSI\&lt;50（空）或MACD金叉死叉：\+1

    

#### 2\.4\.3 环境自适应调整



**震荡市（Range）调整**：

- 评分逻辑切换：不再依赖趋势条件，改用区间反转逻辑。

- 接近区间下沿 → 倾向 LONG，给予评分加成。

- 接近区间上沿 → 倾向 SHORT，给予评分加成。

- 区间内无明显趋势 → 不强制大周期方向一致。

- 止损止盈更紧（幅度约为趋势市的50%）。

    

**趋势回调日（Pullback）调整**：

- 保留趋势评分体系，但降低 ADX 权重。

- 增加“回踩 MA20 后反弹确认”作为加分项。

- 止损设置更紧（趋势回调风险高于趋势延续）。

    

#### 2\.4\.4 稳定性字段



每个信号增加 `stability` 字段：基于最近5周期K线方向一致性计算，衡量信号可靠性。高稳定性 \+ 高评分 = 强信号。

|总分|等级|仓位系数|动作|
|---|---|---|---|
|18\-22|A|1\.0|正常开仓|
|14\-17|B|0\.5|轻仓介入（震荡市放宽至13分）|
|10\-13|C|0\.0|仅记录，不发送|
|\&lt;10|D|0\.0|放弃|

### 3\.1 Final Risk Gate（最终裁决器）完整逻辑



#### 3\.1\.1 模块定位

`Decision Gateway` 是系统中 **唯一拥有交易否决权** 的组件。策略层产生的任何信号，无论评分多高、微观结构多配合，**必须经过此网关校验**，否则无法到达执行层。



**输入数据源**：

|输入项|来源|格式|
|---|---|---|
|`Signal`|OpenClaw 策略层|包含 symbol、direction、score、confidence、entry\_zone、stop\_loss、tp1|
|`State`|State Center \(Redis\)|实时持仓列表、当前总风险占用、当日已实现亏损、权益曲线高点与当前回撤|
|`Weights`|Hermes Agent|各子策略的推荐权重（若无则为默认等权）|
|`Config`|system\_config\.json|风险上限参数、熔断阈值、相关性惩罚系数|



**输出**：

- `APPROVED`：生成 `ExecutionOrder` 发送至执行引擎。

- `REJECTED`：记录拒绝原因，丢弃信号。

- `ADJUSTED`：修改仓位大小后批准（如因相关性惩罚降低名义仓位）。

    

#### 3\.1\.2 裁决逻辑判断树（强制执行顺序）



Gateway 按照以下 **优先级顺序** 逐条校验，**任一条件触发 REJECTED 则立即终止**，不再进行后续检查。



```Plain Text
[1] 账户级熔断检查
    ├── 当前回撤 (Peak - Current) / Peak ≥ 0.20 ？
    │   └── YES → REJECTED (Max Drawdown Liquidation)
    ├── 当日累计已实现亏损 ≥ 账户余额 × 0.03 ？
    │   └── YES → REJECTED (Daily Loss Stop)
    └── 消息风暴模式激活 (消息强度 ≥ 9 持续 1h) ？
        └── YES → 禁止新开仓，REJECTED (News Storm)

[2] 总风险预算检查
    ├── 计算当前已占用风险 = Σ(各持仓 |Entry - Stop| / Entry × 名义仓位 / 账户余额)
    ├── 计算新信号预估风险 = |Entry - Stop| / Entry × (单笔风险上限 1% × 评分仓位系数)
    └── 当前风险 + 新信号风险 > total_risk_cap ？
        └── YES → REJECTED (Risk Budget Exceeded)

[3] 信号置信度与冲突检查
    ├── Signal.score 对应的仓位系数 == 0 ？
    │   └── YES → REJECTED (Low Score)
    ├── Hermes 推荐该策略权重 < 0.1 ？
    │   └── YES → REJECTED (RL Downgrade)
    └── 与现有持仓方向相反且相关系数 > 0.7 ？
        └── YES → REJECTED (Hedging Conflict, 不允许对锁)

[4] 微观结构滑点预估检查
    └── 当前买卖价差 > 0.1% 且 OBI 未显著偏向信号方向 ？
        └── YES → REJECTED (Liquidity Insufficient)

[5] 通过全部检查
    └── 进入资金分配计算模块 → 生成 APPROVED ExecutionOrder
```

### 3\.2 资金分配模型：风险约束 Kelly \+ 相关性调整 Risk Parity



#### 3\.2\.1 总风险预算（Total Risk Cap）的三层计算



**第一层：理论 Kelly 参考值**

```Plain Text
f_star = (b × p - q) / b
```

- `p` = 策略层滚动 50 笔交易的胜率（来自回测引擎）。

- `b` = 平均盈亏比 = avg\_win\_r / \|avg\_loss\_r\|（R 倍数口径）。

- `q` = 1 \- p。

    

**第二层：环境系数调节**

```Plain Text
Kelly_adj = f_star × Env_Coefficient
```

|环境标签|Env\_Coefficient|逻辑说明|
|---|---|---|
|强趋势|0\.5|允许使用理论值的一半。|
|趋势回调|0\.4|回调介入风险略高。|
|震荡|0\.25|反转策略胜率虽高但盈亏比受限。|
|消息驱动|0\.15|高不确定性，极度保守。|
|极端避险|0\.0|强制不交易。|



**第三层：硬性风险上限（来自 Monte Carlo 安全约束）**

```Plain Text
Total_Risk_Cap = min(
    Account_Balance × 0.02,                     // 铁律：2% 总风险上限
    Account_Balance × (1 - MonteCarlo_DD_P95)   // 模拟得出的安全边际
)
```

- `MonteCarlo\_DD\_P95`：每月离线模拟输出的 95% 分位数最大回撤（例如 0\.34）。

    

**最终总风险预算**：

```Plain Text
Total_Risk = min(Total_Risk_Cap, Account_Balance × Kelly_adj)
```



#### 3\.2\.2 单币种风险平价基础权重



对于候选池中通过评分的币种，先计算其 **波动率倒数权重**：

```Plain Text
Raw_Weight_i = 1 / ATR_Percent_i
```

- `ATR\_Percent\_i = ATR\(14\)\_i / Price\_i`（波动率百分比）。

    

示例：BTC 波动率 1\.2%，ETH 波动率 1\.8%，SOL 波动率 2\.5%。

|币种|ATR%|Raw Weight|归一化后|
|---|---|---|---|
|BTC|1\.2%|83\.33|55\.2%|
|ETH|1\.8%|55\.56|36\.8%|
|SOL|2\.5%|40\.00|8\.0%|



#### 3\.2\.3 相关性惩罚矩阵（Risk Parity 增强）



当同时持有多个币种时，需对高相关性币种进行风险打折，防止风险预算被重复计算。



**简化分组算法（推荐实现）**：



1. 计算所有候选币种与当前持仓币种的 **1h 收益率滚动相关系数（窗口 72 根 K 线）**。

2. 若任意两币种相关系数 **\&gt; 0\.8**，则归入同一 **“高相关组”**。

3. 对于组内币种，**组总风险预算乘以惩罚系数 0\.6**。

    

**计算公式**：

```Plain Text
Group_Risk_Budget = Total_Risk × (组内原始权重之和) × 0.6
```



**组内二次分配**：

组内各币种按原始权重比例瓜分 `Group\_Risk\_Budget`。



**示例场景**：

- 持仓：BTC（已占用风险 1\.0%）。

- 新信号：ETH（与 BTC 相关系数 0\.85）。

- ETH 原始权重占比 40%。

    

**计算过程**：

1. 检测到 BTC 与 ETH 相关系数 0\.85 \&gt; 0\.8，归为同一组。

2. 组内原始权重：BTC 60% \+ ETH 40% = 100%。

3. 组总预算 = 2% × 100% × 0\.6 = **1\.2%**（而非原本的 2%）。

4. ETH 实际可用风险 = 1\.2% × 40% = **0\.48%**（而非原本的 2% × 40% = 0\.8%）。

    

**无高相关组的情况**：

- 若新信号币种（如 SOL）与现有持仓相关系数均 \&lt; 0\.8，则直接使用原始 Risk Parity 权重分配，不受惩罚。

    

#### 3\.2\.4 风险预算到名义仓位的转换



```Plain Text
Position_Size_i = (Risk_Budget_i × Account_Balance) / Stop_Loss_Percentage_i
```

- `Stop\_Loss\_Percentage\_i = \|Entry\_i \- Stop\_i\| / Entry\_i`。

    

**安全限制（硬性）**：

- 单币种名义仓位 ≤ 账户余额 × 10（杠杆 ≤ 10x）。

- 所有持仓总名义仓位 ≤ 账户余额 × 30。

- 若计算出的仓位超过限制，等比缩减所有持仓。

## 第四章：决策与风控层——动态仓位与熔断体系



### 4\.1 动态仓位控制系统：连续函数设计



#### 4\.1\.1 设计目标

传统量化系统通常采用“分段式降仓”（如回撤 10% 降仓至 50%，回撤 15% 降至 25%）。这种硬切换存在两个问题：

- **边界突变**：在阈值附近，仓位系数跳跃导致策略行为不稳定。

- **缺乏平滑反馈**：无法精细反映账户健康状况的连续变化。

    

V4\.6\.2 采用 **连续指数衰减函数** 替代分段规则，使仓位系数成为回撤幅度的平滑函数。



#### 4\.1\.2 基础公式

```Plain Text
Equity_Curve_Scale = exp(-k × Current_Drawdown)
```

**变量定义**：

- `Current\_Drawdown = \(Peak\_Equity \- Current\_Equity\) / Peak\_Equity`，取值范围 \[0, 1\)。

- `k`：衰减系数，推荐值 **5\.0**（可通过回测优化，范围通常 3\~8）。

    

**函数特性**：

|回撤 \(Drawdown\)|仓位系数 \(k=5\)|含义|
|---|---|---|
|0%|1\.000|满额风险预算。|
|5%|0\.779|轻度回撤，小幅降仓。|
|10%|0\.607|中度回撤，仓位降至约六成。|
|15%|0\.472|显著回撤，仓位减半。|
|20%|0\.368|严重回撤，仓位降至三分之一强。|
|25%|0\.287|接近强制清仓线（20% 时已触发熔断，此值仅作理论参考）。|



**优势**：

- 每一次微小的回撤增加都会立即反映在仓位缩减上，形成 **负反馈闭环**。

- 避免了分段规则在阈值处的“悬崖效应”。

    

#### 4\.1\.3 收益状态调整：滚动夏普比率修正

回撤仅反映亏损幅度，但未考虑近期盈利质量。系统引入 **最近 20 笔交易的滚动夏普比率** 作为正向调节因子。



**计算逻辑**：

1. 取最近 20 笔已平仓交易的 **R 倍数** 序列（若不足 20 笔则用全部历史）。

2. 计算 `Recent\_Sharpe = mean\(R\_list\) / std\(R\_list\)`（无风险利率视为 0）。

3. 根据夏普值调整系数：

```Python
if Recent_Sharpe > 1.5:
    Adjustment_Factor = 1.2
elif Recent_Sharpe < 0.5:
    Adjustment_Factor = 0.7
else:
    Adjustment_Factor = 1.0
```



**应用**：

```Plain Text
Adjusted_Scale = Equity_Curve_Scale × Adjustment_Factor
```

**限制范围**：最终 `Adjusted\_Scale` 必须裁剪至 **\[0\.1, 1\.5\]** 区间。

- 下限 0\.1：确保极端回撤时仍有微量仓位参与市场（避免完全空仓后无法捕捉反转）。

- 上限 1\.5：在极佳盈利状态下允许适度放大仓位（但仍受总风险 2% 的硬约束）。

    

#### 4\.1\.4 波动率熔断（市场环境紧急制动）

当市场出现极端波动时，无论账户状态如何，必须强制降低仓位以规避流动性危机和插针风险。



**触发指标**：

```Plain Text
Market_Volatility = ATR(14) on 5m / Current_Price
```

- 数据源：5 分钟 K 线，滚动计算近 14 根（约 1 小时 10 分钟）的平均真实波幅。

    

**熔断规则**：

|波动率阈值|动作|系数|
|---|---|---|
|\&lt; 4%|正常|1\.0|
|≥ 4%|触发波动率熔断|**0\.5**|



**应用**：

```Plain Text
Final_Scale = Adjusted_Scale × Volatility_Breaker_Coefficient
```



#### 4\.1\.5 最终仓位系数整合

将上述三层调节因子与策略层基础仓位系数融合：



```Plain Text
Final_Position_Scale = Base_Position_Coefficient  (来自三周期评分等级 A=1.0, B=0.5, C=0.2)
                       × Final_Scale              (来自动态仓位控制)
                       × Strategy_Weight          (来自 Hermes RL 权重)
```



**最终名义仓位**：

```Plain Text
Final_Quantity = (Total_Risk_Budget × Final_Position_Scale) / Stop_Loss_Percentage
```

（此公式继承自第三章资金分配模型，并叠加本章的动态调节因子。）

### 4\.2 硬性熔断与破产保护体系



动态仓位控制属于 **“软调节”**，而本节定义的规则属于 **“硬熔断”**。硬熔断一旦触发，将 **强制覆盖** 所有软调节结果，直接终止交易或清仓。



#### 4\.2\.1 账户级熔断（不可逆触发）



|熔断类型|触发条件|执行动作|恢复条件|
|---|---|---|---|
|**单日亏损熔断**|当日累计已实现亏损 ≥ 账户余额 × 3%|立即停止所有开仓操作；已挂限价单全部撤销；仅允许平仓或止损。|UTC 0 点自动重置。|
|**连续亏损暂停**|连续亏损 3 笔（每笔均为净亏损）|暂停交易 **30 分钟**，期间不产生新信号。|冷却时间结束自动恢复。|
|**连续亏损停盘**|连续亏损 5 笔|当日停止交易，视为单日亏损熔断。|UTC 0 点重置。|
|**最大回撤清仓**|账户回撤（从历史最高权益起算） ≥ **20%**|**市价全平所有持仓**；撤销所有挂单；禁止一切交易活动。|**需人工登录后台确认并手动解除**。|



#### 4\.2\.2 消息面熔断



|熔断类型|触发条件|执行动作|
|---|---|---|
|**消息风暴模式**|消息面强度评分 **≥ 9** 且持续时间超过 **1 小时**|强制将现有持仓减仓至 **50%**；新开仓指令全部 `REJECTED`；仅允许止损或手动干预。|
|**极端消息紧急状态**|检测到关键词：“交易所被盗”、“稳定币脱锚”、“监管全面禁止”|立即触发 **最大回撤清仓** 流程（等同于回撤 20%）。|



#### 4\.2\.3 策略级熔断



|熔断类型|触发条件|执行动作|
|---|---|---|
|**单策略禁用**|某子策略（Trend/Pullback/Range/News）连续亏损 **10 笔**|自动将该策略的 `Strategy\_Weight` 强制设为 0；发送警报至监控面板。|
|**单币种暂停**|某币种在 1 小时内触发滑点保护 **3 次**|暂停该币种交易 **1 小时**，并将该币种移出候选池至下一轮更新。|



#### 4\.2\.4 破产保护与 Monte Carlo 联动



**破产定义**：

- 账户权益 \&lt; 初始权益 × 30%（即亏损 70%）。在系统内部，若触发此条件，系统将进入 **只读模式**，禁止任何 API 写操作。

    

**Monte Carlo 预警联动**：

- 每月离线运行的 Monte Carlo 引擎输出 `risk\_of\_ruin`（破产概率）。

- 若 `risk\_of\_ruin \&gt; 0\.05`（5%），系统将在下月 **自动下调总风险上限** 至 **1\.5%**（覆盖 `system\_config\.json` 中的 `max\_total\_portfolio\_risk`）。

- 此调整由 Decision Gateway 在每次启动时读取配置生效。

下图展示了一个交易信号从产生到执行过程中，如何依次经过动态仓位调节与硬熔断检查：

## 第五章：执行层——OpenClaw 执行规范与多 Agent 路由



执行层是系统中 **唯一** 有权调用交易所交易 API 的模块。在 V4\.6\.2 中，执行层基于 OpenClaw 构建，并扩展了多 Agent 协作、定时任务规范与会话管理能力。



### 5\.1 OpenClaw 订单管理规范



#### 5\.1\.1 分批挂单算法（TWAP\-lite）



为降低大额市价单的滑点冲击，同时最大化获取 Maker 手续费返佣，执行层采用 **固定数量分批挂单** 策略。



**参数配置（来自 ****`system\_config\.json`****）**：

- `split\_orders`：3（将总数量拆分为 3 份）。

- `order\_interval\_bps`：5（每份限价单间隔 0\.05%，基于当前买一/卖一价计算）。

- `maker\_only`：true（优先使用 Post\-Only 限价单）。

    

**算法逻辑（以做多为例）**：

```Python
total_qty = approved_order.quantity
base_price = orderbook.best_bid  # 做多时参考买一价

for i in range(split_orders):
    # 每份数量 = 总数量 / 份数（最后一份包含余数）
    slice_qty = total_qty / split_orders
    
    # 价格递减挂单：第一份挂买一，后续挂单价格逐步降低以捕捉回调
    slice_price = base_price * (1 - i * order_interval_bps / 10000)
    
    # 发送 Post-Only 限价单
    place_order(symbol, "BUY", slice_qty, slice_price, post_only=True)
    
    # 记录挂单时间戳
    order_timestamps.append(now())
```



**撤单与重挂逻辑**：

- 若某份订单在 `order\_expiry\_seconds`（30 秒）后仍未成交，**撤销该订单**。

- 重新评估当前订单簿：

    - 若价差 \&gt; 0\.1%，放弃重挂，剩余数量转为市价单（需额外滑点确认）。

    - 若价差正常，以最新买一价重新挂单。

        

#### 5\.1\.2 微观结构配合：OBI 动态撤单



在挂单等待成交期间，执行层持续监听订单簿的 `smoothed\_OBI`（来自第二章的 HFT\-lite 模块）。



**撤单触发条件**：

```Python
if order.direction == "LONG":
    if smoothed_OBI < -0.15:  # 卖压突然增强，原挂单价可能成为“接盘价”
        cancel_order(order.id)
        wait_for_obi_recovery()
```



**恢复逻辑**：

- 撤销后进入 **冷却观察期**（2 秒）。

- 若 2 秒内 `smoothed\_OBI \&gt; 0\.1` 恢复，则重新挂单；否则标记为 `microstructure\_fail`，剩余数量 **放弃执行**，上报风控。

    

#### 5\.1\.3 滑点保护与异常上报



**滑点计算**：

```Plain Text
Slippage_bps = |Actual_Fill_Price - Expected_Price| / Expected_Price × 10000
```

- `Expected\_Price`：信号生成时的 `Best\_Bid`（做多）或 `Best\_Ask`（做空）。

    

**处理规则**：

|滑点范围 \(bps\)|动作|说明|
|---|---|---|
|≤ 10 bps \(0\.1%\)|正常成交|记录日志。|
|\&gt; 10 bps|**交易作废**|立即市价平仓该笔成交（若已成交），并报告 `Slippage\_Exception`。|
|\&gt; 50 bps|**触发警报**|暂停该币种交易 1 小时，并通知运维。|



#### 5\.1\.4 执行层与 Gateway 的签名校验



为防止 OpenClaw 内部逻辑错误或被篡改，所有来自 Gateway 的 `ExecutionOrder` 必须携带 HMAC\-SHA256 签名。



**校验流程**：

```Python
def verify_order(order):
    payload = f"{order.id}:{order.symbol}:{order.quantity}:{order.limit_price}"
    expected_sig = hmac.new(SECRET_KEY, payload, hashlib.sha256).hexdigest()
    if order.signature != expected_sig:
        raise InvalidOrderSignature("Order tampered or from untrusted source")
```

## 第六章：学习层——Hermes RL 引擎与记忆进化



学习层是系统的“大脑”，负责从历史交易中提炼经验、优化策略、沉淀技能。V4\.6\.2 在 V4\.6\.1 的 RL 权重学习基础上，新增了 **技能自沉淀**、**四层记忆架构**、**上下文压缩配置** 等能力。



### 6\.1 Hermes RL 权重自适应



#### 6\.1\.1 算法选型：Contextual Bandit \(Thompson Sampling\)



**弃用 Q\-Learning 的原因**：

- 金融市场是非平稳环境，状态转移概率随时间变化，Q 表的收敛性无法保证。

- Q\-Learning 的 ε\-greedy 探索策略在连续亏损时可能过度探索导致更大回撤。

    

**Thompson Sampling 优势**：

- 通过 Beta 分布天然平衡 **探索与利用**（不确定性高的策略自动获得更多尝试机会）。

- 收敛速度快，适合在线学习。

- 输出的是 **策略权重分布**，而非离散动作，更符合组合资金分配需求。

    

#### 6\.1\.2 状态空间定义



Hermes 从 State Center 拉取以下特征向量，用于区分不同市场上下文：



|特征名|类型|取值范围|说明|
|---|---|---|---|
|`market\_env`|One\-Hot|4 维|当前环境：趋势/回调/震荡/消息驱动。|
|`volatility`|Float|0\~0\.10|当前 5m ATR% 归一化值。|
|`btc\_trend`|Int|\-1, 0, 1|BTC 1h MA20 斜率方向。|
|`news\_score`|Float|0\~10|当前消息面强度评分。|
|`drawdown`|Float|0\~0\.20|当前账户回撤百分比。|



**状态向量示例**：

```JSON
[1, 0, 0, 0, 0.018, 1, 2.3, 0.06]
```

（表示：强趋势环境，波动率 1\.8%，BTC 上涨，消息面平静，回撤 6%）



#### 6\.1\.3 模型结构：多臂老虎机维护



对于每个 **离散化的状态簇**（通过 K\-Means 或决策树将连续状态分桶），Hermes 维护一个独立的 Beta 分布参数集。



```Python
class ContextualBandit:
    def __init__(self, n_strategies=4):
        self.strategies = ["Trend", "Pullback", "Range", "News"]
        # 每个策略维护一对 (alpha, beta)
        self.alpha = {s: 1.0 for s in self.strategies}
        self.beta  = {s: 1.0 for s in self.strategies}
    
    def get_weights(self):
        """返回归一化权重向量"""
        samples = {s: np.random.beta(self.alpha[s], self.beta[s]) 
                   for s in self.strategies}
        total = sum(samples.values())
        return {s: samples[s]/total for s in self.strategies}
    
    def update(self, strategy_name, reward):
        """根据奖励更新 Beta 分布参数"""
        if reward > 0:
            self.alpha[strategy_name] += reward
        else:
            self.beta[strategy_name] += abs(reward)
```



#### 6\.1\.4 奖励函数设计



**核心公式**：

```Python
reward = trade_R - 0.5 * max(drawdown_increase, 0)
```

- `trade\_R`：该笔交易的 R 倍数（盈利金额 / 初始止损金额）。

- `drawdown\_increase`：该笔交易持仓期间，账户回撤的 **新增幅度**（若未创新高回撤则为 0）。

    

**设计意图**：

- 鼓励在 **不扩大账户回撤** 的前提下获取正 R 收益。

- 若某策略虽然盈利但过程剧烈回撤（如扛单后反转），其奖励将被惩罚项显著降低。

    

#### 6\.1\.5 状态聚类与在线学习流程



**聚类方法**：

- 使用 **MiniBatchKMeans**（预设 20 个簇），每日凌晨用历史状态数据重新训练。

- 实时推理时，将当前状态向量映射到最近的簇 ID，使用该簇对应的 `ContextualBandit` 实例。

    

**完整学习循环**：

```Plain Text
[1] 策略层产生 Signal（标记为 Trend 策略）
[2] Decision Gateway 批准并执行
[3] 交易平仓后，计算 R 与回撤增量 → reward
[4] Hermes 获取开仓时刻的状态向量 → 映射到 Cluster_ID
[5] 更新 Cluster_ID 下 Trend 策略的 Alpha/Beta 参数
[6] 下一笔交易前，Gateway 调用 Hermes API 获取最新权重
```



#### 6\.1\.6 权重输出 API 定义



Gateway 在裁决每一笔信号前，调用 Hermes 的内部接口：

```Plain Text
GET /v1/weights?env=trend&vol=0.018&btc=1&news=2.3&dd=0.06
Host: hermes-agent:8081
```



**响应示例**：

```JSON
{
  "Trend": 0.52,
  "Pullback": 0.23,
  "Range": 0.15,
  "News": 0.10,
  "timestamp": "2026-04-23T10:30:00Z"
}
```

### 6\.3 记忆系统详细设计（V4\.6\.2 新增）



#### 6\.3\.1 四层记忆架构



Hermes 的记忆系统 = **内置记忆 \+ 外部记忆提供商 \+ 运行时上下文**。



|层级|目录/文件|用途|维护者|容量限制|
|---|---|---|---|---|
|**L1: working**|会话上下文|当前会话的临时工作记忆|系统自动|受上下文窗口限制|
|**L2: episodic**|`\~/\.hermes/memory/episodic/`|每次执行的行为日志（YYYY\-MM\-DD\.md）|Agent 自动追加|无硬限制，建议每日 \&lt; 500 行|
|**L3: semantic**|`\~/\.hermes/memory/semantic/LESSONS\.md`|沉淀下来的通用经验|Agent 策展|**2,200 字符**|
|**L4: personal**|`\~/\.hermes/memory/personal/PREFERENCES\.md`|用户个人偏好|Agent 策展|**1,375 字符**|



#### 6\.3\.2 冻结快照机制

#### 6\.3\.3 Agent\-curated memory 策展机制



Hermes 的内置记忆是 **Agent 策展** 的，不是全量记录。



**记忆更容易被保留的场景**：

- 用户明确表达了偏好（“我喜欢/不喜欢 xxx”）

- 发现了环境事实（“这台机器装了 xxx”）

- 纠正了 Agent 的错误做法

- 完成了一个重要任务里程碑

- 用户明确要求记住某件事

    

**主动引导记忆**：

> “记住我的偏好：所有止损设置必须外扩 0\.2×ATR，不允许使用固定金额止损。”
> 
>

### 6\.5 离线 Monte Carlo 引擎实现细节



Monte Carlo 引擎作为 Hermes 的离线子模块，每月运行一次，输出关键风险参数供下月使用。



#### 6\.5\.1 交易分布构建

从 PostgreSQL 中提取过去 90 天的所有平仓交易，按 **环境标签** 分组存储 R 倍数列表：

```JSON
{
  "trend": [1.2, -1.0, 2.5, -0.8, ...],
  "range": [0.6, -0.9, 0.3, -1.1, ...],
  "news": [2.8, -2.0, 1.5, -3.0, ...]
}
```



#### 6\.5\.2 马尔可夫环境序列生成

使用历史环境转移频率构建转移矩阵：

```Python
P = {
    "trend":  {"trend": 0.55, "range": 0.30, "news": 0.15},
    "range":  {"trend": 0.25, "range": 0.55, "news": 0.20},
    "news":   {"trend": 0.35, "range": 0.40, "news": 0.25}
}
```

模拟时，从当前环境开始，按概率抽样下一日环境，生成 252 个交易日（约一年）的路径。



#### 6\.5\.3 路径模拟与破产统计

对每条模拟路径：

1. 每日根据环境从对应 R 分布中随机抽取一笔交易结果。

2. 应用动态仓位函数 `exp\(\-k × DD\)` 计算当日仓位系数。

3. 更新权益曲线。

4. 若权益跌破初始资金的 30%，记录 `ruin=True` 并终止该路径。

    

运行 1000 条路径后，输出：

- `risk\_of\_ruin`：破产路径数 / 1000。

- `max\_drawdown\_p95`：95% 分位数最大回撤。

- `expected\_return`：平均年化收益。

    

**联动机制**：若 `risk\_of\_ruin \&gt; 0\.05`，自动更新 `system\_config\.json` 中的 `max\_total\_portfolio\_risk` 为 1\.5%（原 2%）。

### 7\.2 冷启动四阶段流程与验收标准



**禁止** 在部署后立即启用全自动交易。必须严格按照以下阶段推进。



#### 阶段 1：基础设施验证（1\-3 天）

- **状态**：所有容器启动，行情服务运行，执行引擎使用只读 API Key。

- **验收标准**：

    * [ ] Redis 与 PostgreSQL 健康检查通过。

    * [ ] `market\-data` 成功推送 BTC/ETH 订单簿数据至 Redis，延迟 \&lt; 200ms。

    * [ ] `openclaw\-core` 能够生成选币池与市场环境标签，并在日志中可见。

    * [ ] Grafana 大盘显示实时价格与 OBI 曲线。

    * [ ] Hermes Agent 成功连接数据库，日志显示 `Connected to PostgreSQL`。

    

#### 阶段 2：纯规则 Paper Trading（2 周）

- **状态**：执行引擎仍使用只读 API，所有信号仅记录日志，不实际下单。

- **系统配置**：`system\_config\.json` 中设置 `paper\_trading: true`。

- **验收标准**：

    * [ ] 连续运行 14 天无服务崩溃或内存泄漏。

    * [ ] 日均信号产生数量在合理范围（5\-30 笔）。

    * [ ] 模拟权益曲线回撤 \&lt; 15%（基于 Paper 日志计算）。

    * [ ] 滑点模拟统计（基于历史订单簿回放）\&lt; 0\.1% 的订单占比 \&gt; 95%。

    * [ ] Cron 定时任务（每日复盘）能正常触发并生成报告。

    

#### 阶段 3：Hermes 影子模式（1 周）

- **状态**：启动 Hermes 容器，仅读取数据、输出权重日志，不向 Gateway 发送权重。

- **验收标准**：

    * [ ] Hermes 每日成功聚类状态并更新 Beta 参数。

    * [ ] 权重输出无明显异常（如某策略权重突然升至 0\.99）。

    * [ ] 对比纯规则权益曲线与 Hermes 建议权益曲线（模拟），后者回撤无明显放大。

    * [ ] Skill 候选生成功能正常，无异常提案涌入。

    

#### 阶段 4：小资金上线（10% 仓位限制）

- **状态**：执行引擎切换为交易 API Key，总风险设为 0\.2%。

- **验收标准**：

    * [ ] 首日实际滑点 \&lt; 0\.1%。

    * [ ] 实际成交与 Paper 模拟的偏差 \&lt; 10%。

    * [ ] 连续 7 个交易日无硬熔断触发。

    * [ ] 若连续盈利且回撤 \&lt; 5%，可逐步将风险上限调至 1%（需人工确认）。

### 7\.4 Token 成本优化策略



#### 7\.4\.1 模型切换与精简

- 所有 Agent 统一使用 `deepseek\-chat`，性价比极高。

- `tools\.profile` 精简为 `none`，避免不必要的工具定义占 Token。

- 禁用未使用的插件（deepseek 插件、bonjour 插件），减少加载开销。

    

#### 7\.4\.2 缓存命中原则

- 核心原则：一个 session 里尽量只做一件事。

- 初始文件尽量简练：`SOUL\.md ≤ 1KB`，`MEMORY\.md ≤ 2,200 字符`。

    

#### 7\.4\.3 Gateway 目录规范

Gateway 必须从 `$HOME` 启动，避免加载项目级 `AGENTS\.md` 浪费 Token。

### 8\.5 策略铁律（11条）



1. 总风险 ≤ 账户 2%，且 ≤ Monte Carlo 安全上限。

2. 单笔风险 ≤ 账户 1%。

3. 微观结构信号仅用于过滤和执行优化，不独立产生信号。

4. 同一币种同方向 10 秒内最多一次微观触发。

5. 回撤 \&gt; 20% 立即清仓停盘，需人工恢复。

6. 每月运行 Monte Carlo，破产概率 \&gt; 5% 则下月强制降险至 1\.5%。

7. Hermes RL 权重每笔交易后更新，禁止人工干预。

8. 滑点超过 0\.1% 的交易自动作废并记录异常。

9. 任何子策略连续亏损 10 笔 → 自动禁用，需人工复核。

10. 消息面强度 ≥ 9 持续 1 小时 → 强制减仓 50%，禁止开新仓。

11. 任何需要持久化的规则必须写入配置文件或注册为 Cron，禁止依赖 Agent 口头承诺。

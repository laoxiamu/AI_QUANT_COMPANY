## 第一章：系统总览与架构蓝图





### 1\.1 系统定位与设计哲学





#### 1\.1\.1 系统定位



V4\.6\.2是一个面向生产环境的、具备部分自进化能力的智能交易操作系统，不是一个单纯的量化策略脚本。



|定位维度|描述|
|---|---|
|**功能定位**|加密货币永续合约的全自动、多策略、组合式交易系统|
|**技术定位**|基于Agent架构的Alpha策略与Infrastructure风控解耦的分布式系统|
|**风险定位**|以生存为首要目标的防御型系统，单日亏损与最大回撤具有不可绕过的硬件级熔断|
|**进化定位**|具备环境自适应策略路由、数据驱动迭代与因子验证能力|



**阶段性定位（2026\-05\-08起）**：V4\.6\.2冻结为Baseline Benchmark。引擎停止运行，策略代码不再修改。系统保留Docker基础服务和PostgreSQL数据归档。V5作为独立Research Branch新建，V4的唯一后续作用是作为V5所有setup的对照基准。





#### 1\.1\.2 核心设计哲学



系统遵循以下六条铁律：



**单向控制流（One\-Way Control Flow）**：决策权与执行权分离。Hermes（大脑）只能建议，不能下令；OpenClaw（手脚）只能执行，不能决策。Decision Gateway是唯一拥有否决权的组件。



**风险绝对优先（Risk\-First Architecture）**：任何信号在触碰交易所API前，必须经过独立的Decision Gateway校验。风控规则属于系统层硬编码，禁止任何Agent或RL模型修改。



**唯一真实数据源（Single Source of Truth）**：所有模块不持有私有状态，必须读写同一个State Center（Redis \+ PostgreSQL）。2026年5月7日已正式确定PostgreSQL为唯一权威状态源。`\_positions`字典和CSV文件降级为缓存或历史快照。任何风控决策不能仅依赖非权威源中的数据。



**环境自适应策略路由（Environment\-Adaptive Strategy Routing）**：系统先判断市场状态（趋势/回调/震荡/极端），再选择对应的子策略。避免单一趋势跟踪逻辑在震荡市失效。



**数据驱动迭代（Data\-Driven Iteration）**：所有参数修改必须有回测数据支撑。评分系统的有效性必须通过统计验证（IC分析、固定持仓回测）。当数据证明某个模块失效时（如22分制无预测力），系统具备替换该模块的架构灵活性。



**先可靠再盈利（Reliability Before Profitability）**：系统的Risk Integrity必须被证明可靠后，才允许讨论策略Alpha。此铁律源自Ghost Position Incident的确诊——系统无法保证风控完整性时，任何策略Alpha都无法被有效评估。





#### 1\.1\.3 架构设计原则



以下原则基于V4\.6\.2观察期内暴露的全部架构缺陷总结，已写入AGENTS\.md为永久设计约束：



|原则|内容|反例|
|---|---|---|
|**唯一权威源**|任何数据在任何时刻只有一个权威存储位置。持仓→PostgreSQL，参数→config\.py，计划状态→MEMORY\.md，Agent行为规则→AGENTS\.md|`\_positions`和DB同时存持仓状态，且没有对账机制|
|**职责隔离**|策略层只输出信号，风控层只做裁决，执行层只做执行。任何一层不能跨层直接操作其他层的数据|Phase 1从`\_positions`（策略层内存结构）读取持仓列表，跳过PostgreSQL（数据层权威源）|
|**Trade ≠ Position**|每一笔交易有终身唯一的trade\_id。同一Position可能经历多次Trade变更（加仓/减仓/替换），但每次变更必须有完整的审计记录|以`symbol\_direction`作为唯一标识，无法区分同向的多笔交易|
|**风控独立于策略**|Gateway的风控规则是硬编码的。策略输出的信号在被执行之前必须经过Gateway的独立校验|Ghost Position中，风控系统失去了对策略层产生的持仓的追踪能力|
|**修改前评估，修改后验证**|任何涉及持仓生命周期的代码变更必须单独做风险评估。修复后必须验证Phase 1是否覆盖全部活跃持仓|FIX\-5引入同向覆盖时未做风险评估，Ghost Position三天后才被偶然发现|

### 1\.3 数据流向与单向控制铁律





#### 1\.3\.1 标准交易请求流（Happy Path）



```Plain Text
[1] 行情到达 → State Center（Redis更新价格/OBI）
    ↓
[2] OpenClaw Signal Agent 读取状态 → 运行策略逻辑 → 生成Signal
    ↓
[3] OpenClaw Main Agent 请求Hermes权重 → Hermes返回建议
    ↓
[4] OpenClaw 组装Final Signal → 发送至Decision Gateway
    ↓
[5] Decision Gateway 读取State Center → 风控校验 → 裁决
    ↓
[6] Gateway 生成Approved Order → 发送至Execution Engine
    ↓
[7] Execution Engine 签名发送至交易所 → 更新State Center
    ↓
[8] 交易平仓后，Hermes读取结果 → 更新RL模型 / 写入记忆
```





#### 1\.3\.2 Ghost Position Incident暴露的数据流断裂



2026年5月8日深夜发现的Ghost Position Incident揭示了上述标准流程中的一个致命断裂点：



```Plain Text
正常路径（设计）：
  PostgreSQL（6笔活跃持仓）→ Phase 1读取 → 检查止损 → 触发/跳过

实际路径（5月8日22:01-23:01）：
  PostgreSQL（6笔活跃持仓）→ Phase 1读取_positions（4笔）→ 漏检2笔LONG
                              ↑
                    同向覆盖已删除旧持仓
```



**断裂点**：Phase 1没有从PostgreSQL权威源读取活跃持仓列表，而是依赖已被覆盖的`\_positions`字典。这是2026年5月7日确认PostgreSQL为唯一权威源之前的历史遗留设计。





#### 1\.3\.3 单向控制强制约束（代码级防呆设计）



|约束项|规则内容|违规后果|
|---|---|---|
|**API密钥隔离**|Execution Engine持有交易密钥；Hermes仅持有只读密钥|Hermes无法构造有效签名|
|**网络防火墙**|Hermes容器禁止访问Execution Engine容器端口|即使入侵也无法直接下单|
|**代码接口限制**|OpenClaw的execute\_order\(\)仅接收带Gateway签名的ApprovedOrder|绕过Gateway的请求会被拒绝|
|**风控输入源隔离**|Phase 1必须从PostgreSQL权威源读取持仓列表。`\_positions`仅为缓存|依赖不可靠输入源导致Ghost Position|
|**同向覆盖禁用**|已有同币种同方向活跃持仓时，直接拒绝新信号，禁止静默覆盖|旧仓盈亏丢失\+风控失明|

## 第二章：策略层详解



策略层是系统的信号发生器。在V4\.6\.2优化版中，策略层实现了环境自适应——先判断市场状态，再选择对应的子策略，解决了原版单一趋势跟踪在震荡市失效的问题。在Update1中，通过两份系统诊断报告和DX\-1固定持仓回测，发现22分制评分系统不具备预测区分力。



**当前状态（2026\-05\-08起）**：22分制评分系统已冻结为Baseline Benchmark。策略引擎已停止运行。后续V5将转向Structure Event System。

|池类型|来源规则|数量上限|设计意图|
|---|---|---|---|
|**Core Pool**|BTCUSDT, ETHUSDT, SOLUSDT（固定）|3|稳定流动性基准，系统永不离线|
|**Momentum Pool**|同时满足：①成交量\&gt;24h均量×1\.5；②ATR环比上升\&gt;10%；③价格突破近20根1h K线区间|3|捕捉刚启动的趋势币|

#### 2\.1\.2 流动性硬性过滤



**必要条件**：24h成交额 ≥ 50,000,000 USD。未通过过滤的币种直接放入禁做池，不参与后续任何评分。





#### 2\.1\.3 五因子加权评分模型（满分17分）



|因子|权重|计算方式|说明|
|---|---|---|---|
|**Trend**（趋势因子）|0\.25|\(收盘价/MA20 \- 1\) × 100|衡量价格相对于均线的偏离程度|
|**Volatility**（波动率因子）|0\.20|\(High \- Low\) / Price|评估波幅的适应性，甜点区0\.3%\-1\.5%|
|**Liquidity**（流动性因子）|0\.20|log\(24h成交额\)|对数压缩后参与横截面排序|
|**RS**（相对强弱因子）|0\.20|币种涨跌幅 \- BTC涨跌幅|衡量独立Alpha|
|**Micro**（微观因子）|0\.15|买卖价差、深度不平衡|评估盘口质量|



**已知局限**：micro因子当前实现永远返回1分，五因子实际退化为四因子。此问题已在诊断报告中记录，修复优先级排在因子验证之后。



**评级与动作映射**：A级\(14\-17分\)→主做池满额风险预算；B级\(11\-13分\)→备选池轻仓介入；C级\(≤10分\)→禁做池。





### 2\.2 市场环境分类与策略路由





#### 2\.2\.1 技术环境判定矩阵



|环境名称|量化条件（满足≥2条触发）|对应子策略倾向|
|---|---|---|
|**强趋势日**|①MA20斜率\&gt;0\.1%/K；②ADX\(14\)\&gt;25；③BOLL带宽\&gt;过去20期均值×1\.2|**TrendStrategy**|

#### 2\.2\.2 消息面强度评分算法



Score = BaseWeight × 0\.5^\(hours/halflife\) × \(1 ± sentiment\_coefficient\)。BaseWeight：1\-10（美联储决议10、日常新闻1）。半衰期：2小时。极端消息（强度≥9）→触发“消息风暴模式”，强制减仓至50%或空仓。



**关键修正规则**：利多不涨→强度降级50%；利空不跌→强度降级50%。





#### 2\.2\.3 环境综合判定与策略路由



|最终环境|触发逻辑|策略响应|
|---|---|---|
|**强趋势**|技术面=强趋势且消息强度\&lt;4|TrendStrategy：趋势跟踪|
|**趋势回调**|技术面=回调且消息强度\&lt;5|PullbackStrategy：回调介入|
|**震荡**|技术面=震荡且消息强度\&lt;4|RangeStrategy：区间反转|
|**消息驱动**|消息强度≥6|NewsStrategy：超短线动能|
|**极端避险**|技术极端或消息强度≥9|空仓，禁止开仓|



**策略路由实现**：environment\.py输出环境标签后，entry\_signal\.py的\_score\_22\(\)根据环境选择评分逻辑——趋势市保留原22分制，震荡市切换为区间反转评分（接近区间下沿→倾向LONG，接近区间上沿→倾向SHORT）。止损止盈幅度约为趋势市的50%。

# 第三章：决策与风控层——风控裁决与资金分配



### 3\.1 Final Risk Gate（最终裁决器）完整逻辑



#### 3\.1\.1 模块定位



Decision Gateway 是系统中**唯一拥有交易否决权**的组件。策略层产生的任何信号，无论评分多高、微观结构多配合，**必须经过此网关校验**，否则无法到达执行层。



**输入数据源**：Signal（OpenClaw策略层，包含symbol/direction/score/confidence/entry\_zone/stop\_loss/tp1）、State（State Center Redis，实时持仓列表、当前总风险占用、当日已实现亏损、权益曲线高点与当前回撤）、Weights（Hermes Agent推荐权重，若无则为默认等权）、Config（system\_config\.json，风险上限参数、熔断阈值、相关性惩罚系数）。



**输出**：APPROVED（生成ExecutionOrder发送至执行引擎）、REJECTED（记录拒绝原因，丢弃信号）、ADJUSTED（修改仓位大小后批准，如因相关性惩罚降低名义仓位）。



**Update1修复**：信号入库时strategy\_runner\.py中status匹配已从`if status == \&\#34;SUCCESS\&\#34;`修复为`if status in \(\&\#34;SUCCESS\&\#34;, \&\#34;APPROVED\&\#34;\)`（FIX\-1）。在此之前，Gateway返回APPROVED但入库逻辑不匹配，导致所有paper\-trade信号丢失。



#### 3\.1\.2 裁决逻辑判断树（强制执行顺序）



Gateway按照以下**优先级顺序**逐条校验，**任一条件触发REJECTED则立即终止**，不再进行后续检查。



**\[1\] 账户级熔断检查**：当前回撤\(Peak \- Current\)/Peak ≥ 0\.20→REJECTED\(Max Drawdown Liquidation\)。当日累计已实现亏损 ≥ 账户余额 × 0\.03→REJECTED\(Daily Loss Stop\)。消息风暴模式激活\(消息强度 ≥ 9 持续 1h\)→禁止新开仓，REJECTED\(News Storm\)。



**\[2\] 总风险预算检查**：计算当前已占用风险 = Σ\(各持仓\|Entry \- Stop\|/Entry × 名义仓位 / 账户余额\)。计算新信号预估风险 = \|Entry \- Stop\|/Entry × \(单笔风险上限1% × 评分仓位系数\)。当前风险 \+ 新信号风险 \&gt; total\_risk\_cap→REJECTED\(Risk Budget Exceeded\)。



**\[3\] 信号置信度与冲突检查**：Signal\.score对应的仓位系数==0→REJECTED\(Low Score\)。Hermes推荐该策略权重\&lt;0\.1→REJECTED\(RL Downgrade\)。与现有持仓方向相反且相关系数\&gt;0\.7→REJECTED\(Hedging Conflict，不允许对锁\)。



**\[4\] 微观结构滑点预估检查**：当前买卖价差\&gt;0\.1%且OBI未显著偏向信号方向→REJECTED\(Liquidity Insufficient\)。



**\[5\] 通过全部检查**→进入资金分配计算模块→生成APPROVED ExecutionOrder。

### 3\.2 资金分配模型：风险约束Kelly \+ 相关性调整Risk Parity



#### 3\.2\.1 总风险预算（Total Risk Cap）的三层计算



**第一层：理论Kelly参考值**：f\_star = \(b×p \- q\)/b。其中p=策略层滚动50笔交易的胜率（来自回测引擎），b=平均盈亏比=avg\_win\_r/\|avg\_loss\_r\|（R倍数口径），q=1\-p。



**第二层：环境系数调节**：Kelly\_adj = f\_star × Env\_Coefficient。各环境系数：强趋势0\.5、趋势回调0\.4、震荡0\.25、消息驱动0\.15、极端避险0\.0。



**第三层：硬性风险上限**：Total\_Risk\_Cap = min\(Account\_Balance × 0\.02, Account\_Balance × \(1 \- MonteCarlo\_DD\_P95\)\)。MonteCarlo\_DD\_P95为每月离线模拟输出的95%分位数最大回撤。



**最终总风险预算**：Total\_Risk = min\(Total\_Risk\_Cap, Account\_Balance × Kelly\_adj\)。



#### 3\.2\.2 单币种风险平价基础权重



Raw\_Weight\_i = 1/ATR\_Percent\_i。ATR\_Percent\_i = ATR\(14\)\_i / Price\_i（波动率百分比）。波动率越大的币种权重越低，实现风险平价。



#### 3\.2\.3 相关性惩罚矩阵（Risk Parity增强）



当同时持有多个币种时，需对高相关性币种进行风险打折，防止风险预算被重复计算。



**简化分组算法**：计算所有候选币种与当前持仓币种的1h收益率滚动相关系数（窗口72根K线）。若任意两币种相关系数\&gt;0\.8，则归入同一“高相关组”。对于组内币种，组总风险预算乘以惩罚系数0\.6。组内各币种按原始权重比例瓜分Group\_Risk\_Budget。



**无高相关组的情况**：若新信号币种与现有持仓相关系数均\&lt;0\.8，则直接使用原始Risk Parity权重分配，不受惩罚。



#### 3\.2\.4 风险预算到名义仓位的转换



Position\_Size\_i = \(Risk\_Budget\_i × Account\_Balance\) / Stop\_Loss\_Percentage\_i。Stop\_Loss\_Percentage\_i = \|Entry\_i \- Stop\_i\|/Entry\_i。当前止损比率为1\.5%。



**安全限制（硬性）**：单币种名义仓位≤账户余额×10（杠杆≤10x）。所有持仓总名义仓位≤账户余额×30。若计算出的仓位超过限制，等比缩减所有持仓。



### 3\.3 决策层输出与执行层交接规范



Gateway完成裁决与资金分配后，生成**签名后的执行指令**发送至Execution Engine。执行层**不得修改**以下字段：symbol/direction/quantity/stop\_loss\_price（硬止损价，执行层必须挂上）。limit\_price可微调——执行层可根据订单簿深度微调（范围±0\.05%）。



## 第四章：决策与风控层——动态仓位与熔断体系



### 4\.1 动态仓位控制系统：连续函数设计



#### 4\.1\.1 设计目标



传统量化系统通常采用“分段式降仓”（如回撤10%降仓至50%，回撤15%降至25%）。这种硬切换存在边界突变问题（在阈值附近仓位系数跳跃导致策略行为不稳定）和缺乏平滑反馈问题（无法精细反映账户健康状况的连续变化）。



V4\.6\.2采用**连续指数衰减函数**替代分段规则，使仓位系数成为回撤幅度的平滑函数。



#### 4\.1\.2 基础公式



Equity\_Curve\_Scale = exp\(\-k × Current\_Drawdown\)。k为衰减系数，推荐值5\.0（可通过回测优化，范围通常3\~8）。



函数特性：0%回撤→仓位系数1\.000（满额风险预算）、5%回撤→0\.779（轻度回撤小幅降仓）、10%回撤→0\.607（中度回撤仓位降至约六成）、15%回撤→0\.472（显著回撤仓位减半）、20%回撤→0\.368（严重回撤仓位降至三分之一强）、25%回撤→0\.287（接近强制清仓线，此值仅作理论参考）。优势在于每一次微小的回撤增加都会立即反映在仓位缩减上，形成负反馈闭环，避免分段规则在阈值处的悬崖效应。



#### 4\.1\.3 收益状态调整：滚动夏普比率修正



回撤仅反映亏损幅度，但未考虑近期盈利质量。系统引入最近20笔交易的滚动夏普比率作为正向调节因子。计算逻辑：取最近20笔已平仓交易的R倍数序列，计算Recent\_Sharpe = mean\(R\_list\)/std\(R\_list\)。若Recent\_Sharpe\&gt;1\.5→Adjustment\_Factor=1\.2；\&lt;0\.5→0\.7；否则1\.0。



**应用**：Adjusted\_Scale = Equity\_Curve\_Scale × Adjustment\_Factor。最终Adjusted\_Scale必须限制在\[0\.1, 1\.5\]区间。下限0\.1确保极端回撤时仍有微量仓位参与市场（避免完全空仓后无法捕捉反转），上限1\.5在极佳盈利状态下允许适度放大仓位（但仍受总风险2%的硬约束）。



#### 4\.1\.4 波动率熔断（市场环境紧急制动）



当市场出现极端波动时，无论账户状态如何，必须强制降低仓位以规避流动性危机和插针风险。触发指标Market\_Volatility = ATR\(14\) on 5m/Current\_Price。数据源为5分钟K线，滚动计算近14根（约1小时10分钟）的平均真实波幅。



波动率\&lt;4%→正常，系数1\.0。波动率≥4%→触发波动率熔断，系数0\.5。应用：Final\_Scale = Adjusted\_Scale × Volatility\_Breaker\_Coefficient。



#### 4\.1\.5 ATR动态仓位（P2\-2，Update1新增）



在基础动态仓位函数之上，增加基于ATR的仓位调节因子，实现position ∝ 1/ATR的风险平价思想。核心逻辑：高波动时自动缩小仓位，低波动时放大仓位。



计算方式：atr\_factor = base\_atr/max\(atr\_val, 0\.003\)，限制在\[0\.5, 1\.5\]区间。base\_atr已校准为0\.01匹配BTC实际1h ATR分布。



**回测依据**：ATR动态仓位总R提升59%（\+15\.00→\+23\.85），两个环境（Trend和Range）都受益。



#### 4\.1\.6 最终仓位系数整合



Final\_Position\_Scale = Base\_Position\_Coefficient\(来自三周期评分等级\)×Final\_Scale\(来自动态仓位控制\)×Strategy\_Weight\(来自Hermes RL权重\)×atr\_factor\(来自ATR动态仓位\)×env\_weight\(Range环境0\.5×，Trend环境1\.0×\)。



**最终名义仓位**：Final\_Quantity = \(Total\_Risk\_Budget × Final\_Position\_Scale\)/Stop\_Loss\_Percentage。



### 4\.2 硬性熔断与破产保护体系



动态仓位控制属于“软调节”，而本节定义的规则属于“硬熔断”。硬熔断一旦触发，将**强制覆盖**所有软调节结果，直接终止交易或清仓。



#### 4\.2\.1 账户级熔断（不可逆触发）



**单日亏损熔断**：条件为当日累计已实现亏损≥账户余额×3%。执行动作立即停止所有开仓操作、已挂限价单全部撤销、仅允许平仓或止损。恢复条件为UTC 0点自动重置。



**连续亏损暂停**：条件为连续亏损3笔（每笔均为净亏损）。执行动作暂停交易30分钟，期间不产生新信号。恢复条件为冷却时间结束自动恢复。



**连续亏损停盘**：条件为连续亏损5笔。执行动作当日停止交易，视为单日亏损熔断。恢复条件为UTC 0点重置。



**最大回撤清仓**：条件为账户回撤（从历史最高权益起算）≥20%。执行动作**市价全平所有持仓**、撤销所有挂单、禁止一切交易活动。恢复条件为**需人工登录后台确认并手动解除**。



#### 4\.2\.2 消息面熔断



**消息风暴模式**：条件为消息面强度评分≥9且持续时间超过1小时。执行动作强制将现有持仓减仓至50%，新开仓指令全部REJECTED，仅允许止损或手动干预。



**极端消息紧急状态**：检测到关键词“交易所被盗”“稳定币脱锚”“监管全面禁止”。执行动作立即触发最大回撤清仓流程（等同于回撤20%）。



#### 4\.2\.3 策略级熔断



**单策略禁用**：条件为某子策略（Trend/Pullback/Range/News）连续亏损10笔。执行动作自动将该策略的Strategy\_Weight强制设为0，发送警报至监控面板。



**单币种暂停**：条件为某币种在1小时内触发滑点保护3次。执行动作暂停该币种交易1小时，并将该币种移出候选池至下一轮更新。



#### 4\.2\.4 破产保护与Monte Carlo联动



**破产定义**：账户权益\&lt;初始权益×30%（即亏损70%）。系统内部若触发此条件，系统将进入只读模式，禁止任何API写操作。



**Monte Carlo联动**：每月离线运行输出risk\_of\_ruin（破产概率）。若risk\_of\_ruin\&gt;0\.05（5%），系统将在下月自动下调总风险上限至1\.5%（覆盖system\_config\.json中的max\_total\_portfolio\_risk）。此调整由Decision Gateway在每次启动时读取配置生效。



**已知局限**：Monte Carlo离线模拟尚未部署，当前依赖固定配置参数。相关功能排在P3阶段。



### 4\.3 动态仓位与熔断的协同工作流



信号附带基础仓位系数→动态仓位控制（当前回撤计算→滚动夏普调整→ATR动态仓位→环境权重→波动率熔断）→RL权重融合→硬熔断检查（回撤/日亏/连续亏损/消息强度）→最终输出仓位。每一步都在前一步的基础上叠加约束，确保风险暴露在任何单一维度超标时都能被自动约束。

## 第七章：学习层——Hermes RL 引擎与记忆进化



### 7\.1 Hermes RL 权重自适应



#### 7\.1\.1 算法选型：Contextual Bandit \(Thompson Sampling\)



弃用 Q\-Learning 的原因：金融市场是非平稳环境，Q 表的收敛性无法保证；ε\-greedy 策略在连续亏损时可能过度探索导致更大回撤。Thompson Sampling 优势：通过 Beta 分布天然平衡探索与利用，收敛速度快，输出策略权重分布而非离散动作。



#### 7\.1\.2 状态空间与奖励函数



状态特征包括 market\_env（当前环境：趋势/回调/震荡/消息驱动）、volatility（当前5m ATR%归一化值）、btc\_trend（BTC 1h MA20斜率方向）、news\_score（当前消息面强度评分）、drawdown（当前账户回撤百分比）。



奖励函数：reward = trade\_R \- 0\.5 × max\(drawdown\_increase，0\)。trade\_R 为该笔交易的 R 倍数，drawdown\_increase 为持仓期间账户回撤的新增幅度。



#### 7\.1\.3 状态聚类与在线学习



使用 MiniBatchKMeans（预设20个簇），每日凌晨用历史状态数据重新训练。实时推理时将当前状态向量映射到最近的簇 ID，使用该簇对应的 ContextualBandit 实例。



**当前状态**：Hermes 原生 RL 学习功能尚未正式启用，当前使用 FastAPI 代理临时替代其模型调用功能。此为 P3 阶段待办事项，排在策略层稳定运行之后。



### 7\.2 技能自沉淀机制



将 Hermes 在 RL 学习过程中发现的高价值策略模式和风控经验固化为可复用的 Skill。每个 Skill 包含触发条件（市场环境、技术指标阈值等）、操作步骤（确认方向、计算止损止盈、验证盈亏比）、验证方法和注意事项。



自动沉淀流程：同一策略模式连续成功≥3 次→Hermes 生成候选 Skill 草稿→人工 Review→采纳则写入 skills 目录，拒绝则记录原因防止重复提案。每月自动运行技能审计检测功能重复、触发条件模糊、步骤过时的 Skill。



### 7\.3 记忆系统详细设计



四层记忆架构：L1 working（会话上下文，系统自动，受上下文窗口限制）、L2 episodic（每次执行的行为日志，Agent 自动追加，无硬限制）、L3 semantic（沉淀下来的通用经验，Agent 策展，2200字符限制）、L4 personal（用户个人偏好，Agent 策展，1375字符限制）。



冻结快照机制确保 LESSONS\.md 和 PREFERENCES\.md 在每次会话开始时作为冻结快照注入上下文，保证前缀稳定和缓存命中。Nudge 配置控制 Agent 进行记忆反思的频率，交易主流程推荐 3\~5。



### 7\.4 上下文压缩配置



推荐配置：threshold 0\.75（上下文使用率达到此值触发压缩），target\_ratio 0\.25（压缩后保留原长度的比例），protect\_last\_n 30（保护最近 N 条消息不被压缩）。



压缩降级策略：压缩≥2次且提示 accuracy may degrade→准备转移，保存核心决策→开新会话。连续2次压缩无效→压缩器自动停止建议 /new。彻底压不动→Agent 终止，任务标记失败。



### 7\.5 离线 Monte Carlo 引擎



Monte Carlo 引擎作为 Hermes 的离线子模块，每月运行一次。交易分布构建从 PostgreSQL 中提取过去90天的所有平仓交易，按环境标签分组存储 R 倍数列表。马尔可夫环境序列生成使用历史环境转移频率构建转移矩阵，生成252个交易日的路径。路径模拟运行1000条路径后输出 risk\_of\_ruin、max\_drawdown\_p95、expected\_return。



联动机制：若 risk\_of\_ruin \&gt; 0\.05（5%），系统自动更新 system\_config\.json 中的 max\_total\_portfolio\_risk 为 1\.5%。此调整由 Decision Gateway 在每次启动时读取配置生效。



**当前状态**：Monte Carlo 离线模拟模块尚未部署，属于 P3 阶段的待办事项。

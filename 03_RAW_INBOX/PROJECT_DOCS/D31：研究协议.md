# 研究协议

**文档编号**：D31
**版本**：v1.0
**创建日期**：2026-05-09
**关联文档**：D10 INCIDENT_REPORT, D11 OBSERVATION_AUDIT_REPORT, D37 CHANGE_CONTROL


## 一、文档目的

本协议定义V4.6.2项目后续所有策略研究、因子验证、参数调整和观察期执行的标准流程。观察期失效的根本原因之一就是研究协议从未存在——没有实验隔离规则、没有版本命名规则、没有统计有效性标准、没有无效观察期声明规则。本协议是未来所有验证工作的基础方法论。


## 二、观察期执行协议

### 2.1 进入观察期的前提条件

| 条件 | 检查方式 |
|------|---------|
| 所有已知Bug已修复并经代码审查确认 | EXP经验库中无未修复的P0/P1缺陷 |
| 所有核心文件的修改影响评估已完成 | 变更控制记录（D37）中标注为“已完成风险评估” |
| 观察期的核心变量已明确 | 单变量原则：一个观察期只验证一个变量的效果 |
| 预定义终止条件已写入观察期计划 | 样本量阈值、时间上限、异常退出条件全部预定义 |
| 数据基础设施已确认就绪 | PostgreSQL权威源已确认、巡检已切换DB直读、复盘Cron正常运行 |

### 2.2 观察期内的强制约束

| 约束 | 说明 |
|------|------|
| **禁止修改核心策略逻辑** | `entry_signal.py`、`sell_manager.py`、`config.py`中的评分、止损、过滤、覆盖等核心逻辑全部冻结 |
| **禁止新增策略功能** | 不在观察期内引入新的因子、新的过滤机制、新的退出类型 |
| **允许修复P0/P1缺陷** | 但修复后必须记录到变更日志，并在评估报告中标注 |
| **允许修改巡检/监控/告警** | 这些不影响策略信号，属于基础设施维护 |

### 2.3 观察期的终止条件

| 条件 | 行动 |
|------|------|
| 平仓样本 ≥ 预定义阈值 | 进入效果评估阶段 |
| 观察期时间超过上限 | 强制终止，基于现有数据做评估 |
| 发现P0致命缺陷 | 立即终止，转入事故审计。不继续积累数据 |

### 2.4 无效观察期声明

如果一个观察期结束时评估结论的统计有效性无法被保证（如因Bug修复导致变量混杂、因数据记录不完整导致样本不足、因违反冻结规则导致观察期内变量不唯一），必须在评估报告中明确声明“本次观察期数据不具备统计结论效力”，并标注无效原因。声明不是失败，是对研究过程的诚实记录。


## 三、实验隔离规则

### 3.1 单变量原则

**一次只验证一个变量。** V4.6.2观察期违反了这条规则——13天内修改了7个核心文件，涉及评分门槛、止损参数、波动压缩加成、微观结构过滤、SellManager架构、同向覆盖逻辑等多个相互影响的变量。结束时已无法统计每个改动对系统表现的独立贡献。

### 3.2 实验版本命名规则

每个实验拥有独立版本号。格式：`v{主版本}_{setup}_{方向}_{版本号}`。例如 `v5_sweep_bull_v1`。

禁止的命名：`final_v2_latest_new`、`v5_test_modified_final`、任何包含日期但无setup描述的版本名。

### 3.3 实验结果归档

每个实验结束后，其数据、配置、评估报告和代码状态全部归档为独立目录。下一个实验从新的基线开始，不继承上一个实验的任何运行时状态。


## 四、统计有效性协议

### 4.1 样本量要求

| 研究类型 | 最小样本量 | 说明 |
|---------|-----------|------|
| 单Setup回测 | 1000次穿仓 | 确保统计显著性 |
| 单Setup实盘验证 | 50笔平仓 | 确保PnL分布可评估 |
| 因子IC分析 | 500个观测点 | 确保IC的标准误会收敛 |
| 退出机制对比 | 每个退出类型至少20笔平仓 | 确保各类型独立评估 |

### 4.2 核心评估指标

| 指标 | 计算方式 | 用途 |
|------|---------|------|
| **Expectancy（期望值）** | 胜率×平均盈利 + 败率×平均亏损 | 评估setup是否整体有利 |
| **盈亏比（RR）** | 平均盈利/平均亏损 | 评估不对称性 |
| **最大回撤** | 权益峰值到谷值的最大跌幅 | 评估风险 |
| **穿透幅度分布** | 止损触发时的偏离止损线的幅度 | 评估止损设置的合理性 |
| **夏普比率** | 收益率/波动率 | 评估风险调整后收益 |

### 4.3 统计污染的处理

任何存在以下情况的平仓记录从统计中剔除：生命周期断裂（`trade_status = INVALIDATED`）、已被覆盖（`trade_status = REPLACED`但旧仓盈亏未记录）、孤儿持仓（`trade_status = ORPHANED`）。

剔除的样本在评估报告中单独标注数量和原因。


## 五、数据完整性协议

### 5.1 权威数据源

所有研究数据必须从PostgreSQL `signals`表读取。禁止使用CSV文件、`_positions`内存字典、或任何非持久化的中间数据源。

### 5.2 数据质量检查

| 检查项 | 频率 | 说明 |
|--------|------|------|
| 权威源对账 | 每轮Phase 1启动前 | DB vs `_positions`缓存 |
| 孤儿持仓检测 | 每轮Phase 1启动前 | `last_risk_check_at`超时 |
| 数据完整性校验 | 每次评估前 | 确认全部APPROVED信号有完整的生命周期记录 |
| 复盘口径一致性 | 每次评估前 | 确认复盘报告的数量口径与DB直接查询一致 |

### 5.3 数据标注规范

| 标注 | 含义 |
|------|------|
| `INVALIDATED` | 生命周期断裂，从统计中剔除 |
| `REPLACED` | 已被新持仓替换，旧仓盈亏已记录 |
| `ORPHANED` | 未被风控检查，需人工审核 |


## 六、回测验证协议

### 6.1 回测作为验证工具

回测用于验证某个setup在历史数据上是否表现出正期望。回测本身不代表实盘盈利能力——过拟合的回测结果≠可盈利的实盘策略。

### 6.2 回测方法

**Walk-forward**：将历史数据分为训练集和验证集。在训练集上确定参数，在验证集上测试。验证集数据不参与任何参数优化。

**Replay Only**：V5必须内置Replay Engine。基于历史K线和事件数据重放市场状态，确保回测的一致性。

### 6.3 回测输出的评估标准

| 标准 | 阈值 | 说明 |
|------|------|------|
| 正期望 | Expectancy > 1.0 | setup可进入下一阶段验证 |
| 正夏普 | Sharpe > 0.5 | 风险调整后收益可接受 |
| 可接受回撤 | MaxDD < 25% | 风险可控 |
| 稳定IC | IC > 0.02 | 因子有统计意义的预测力 |

以上阈值不是固定标准，是对当前V4.6.2基线（0%胜率，IC≈50%）的边际改善参考值。每个setup可根据其特征设定更合适的目标。


## 七、变更控制协议

### 7.1 需要重新观察的变更

| 变更类型 | 需要重新观察的原因 |
|---------|------------------|
| 修改止损参数（百分比/结构/门槛） | 直接影响退出行为 |
| 修改入场逻辑（因子/信号/过滤） | 直接影响信号质量 |
| 修改持仓生命周期（覆盖/替换/状态机） | 影响风控完整性 |
| 修改仓位管理（动态/环境/连亏） | 影响风险暴露 |

### 7.2 变更记录要求

每次变更必须记录：变更内容、变更原因、影响范围评估、回退计划、验证方式、观察周期和评估标准。


## 八、版本管理协议

### 8.1 代码版本

每次实验前创建独立Git分支。分支名匹配实验版本号（如 `v5_sweep_bull_v1`）。实验结束后分支冻结归档，不在原分支上继续修改。

### 8.2 数据版本

每次实验的数据独立归档。数据文件标注实验版本号和日期范围。禁止将不同实验的数据混合评估。

### 8.3 文档版本

每个实验有独立的评估报告。报告文件命名：`{YYYYMMDD}_{setup}_{version}_EVAL.md`。所有评估报告引用同一份研究协议（本文档）。

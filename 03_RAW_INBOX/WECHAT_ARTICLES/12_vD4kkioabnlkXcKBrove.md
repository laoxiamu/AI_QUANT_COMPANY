# AI 把代码写了，工程团队的管理流程怎么改

Fiona Fung 是 Anthropic 的工程总监，管着 Claude Code 和 Claude Cowork 两条产品线。6 月 3 号她在 Code w/ Claude SF 2026 上做了一场分享，主题很直接：

当 agentic coding 成为团队的默认工作方式之后，工程管理的流程该怎么改。

这并非又一篇「AI 会取代工程师」的讨论。她讲的是写代码变便宜之后，瓶颈转移到哪里，以及她接手团队 18 个月以来实际改掉的流程、踩过的坑、和现在招人时看重的特质。

Fung 的职业生涯有一个很好的锚点：2000 年代初她在微软做 Visual Studio，那时候软件刻在 CD-ROM 上，有硬性的出厂死线。后来在线分发让持续交付成为可能。现在她认为团队正在经历第三次工作方式的重构。这一次，围绕着「写代码」这件事的时间和人力成本急剧下降。

她的原话是：

「写代码、写测试、重构这些事已经很少拖慢我们了。」

但瓶颈没有消失。它转移到了验证（verification）、代码审查（code review）和安全（security）。这也是她被其他工程领导者问得最多的问题：「你们团队的人类是怎么跟上代码审查节奏的？」

这场分享最硬核的地方在于，Fung 讲的每一个结论背后都有她团队的实际数据支撑。下面是几个最反直觉的判断，后面会逐个拆开：

六个月的路线图，三个月就过时了。

并非因为计划能力差，是因为 Claude Code 自己把开发速度拉得太快，原来规划的事情还没做完就变了。现在他们做 JIT 规划——just-in-time，够用就行，不多做。

「谁写的这段代码」不再是一个有用的问题。

既然所有 PR 都经过 Claude，问「谁写的」得不到你真正需要的答案。现在的做法是直接问 Claude：这段变更是谁引入的、为什么引入、上下文是什么。

代码审查里人类只管「专业判断」。

风格、lint、PR 反馈请求、抓 bug、补测试——Claude 全包。人类只在涉及安全边界、法律风险、产品品味的地方介入。而且这个边界不是固定的，模型每升一次级，Fung 就重新评估一次。

招人最不看重的就是代码吞吐量。

因为模型已经把这块包了。她现在重点招两种人：有产品嗅觉的创意型 builder，和有深厚系统功底的工程师。

下面把她的分享按原文章节顺序完整拆一遍。每一个管理实验我都尽量还原她当时的决策逻辑和背后的数据。

这并非翻译，是我读完之后的消化版，顺手带上了我自己的判断。

1

那些默默失效的流程

Fung 有一个观察我很认同：流程是为了弥补某个缺口或改进某个问题而设立的。当缺口消失了，流程很少会自动消失，它会在原地变成惯性。Claude Code 团队发现，当 agentic coding 成为默认工作方式之后，很多过去合理的流程突然不成立了。下面这四个是她们改得最彻底的。

规划：从六个月路线图到 JIT。

Fung 刚接手 Claude Code 团队时，做了一个扎实的六个月路线图。三个月后，因为 Claude Code 自己的迭代速度太快，很多事情已经变了（路线图作废）。她借用了编译器里的概念来描述现在的做法：JIT 规划，只在需要的时候做刚好够用的规划。具体来说，设计文档被 PR 里的讨论和原型替代了；产品评审几乎取消；当前的标准流程是：快速出原型 → 拉大量内部用户上来用 → 根据反馈迭代。这条跟传统的「先想清楚再动手」反着来。说白了，当实现的成本降到足够低，动手比动脑更能暴露真正的问题。

上下文获取：问 Claude，别问作者。

以前的 SOP 是：遇到问题 → 找到写这段代码的人 → 问。现在所有 PR 都是 Claude 辅助完成的，「谁写的」这个问题的信息量急剧缩水。Fung 的做法是反过来：先搞清楚你到底想了解什么。是想知道谁引入了这次回归？找某个领域的专家回答客户问题？了解某个技术决策的上下文？这些事情直接问 Claude，它给出来的信息往往比找人问更完整。而且她团队的习惯是：不管什么问题，顺嘴多问一句「这事能自动化吗」。Fung 自己举了个例子：每天早上边喝咖啡边手动翻客户反馈渠道做摘要，后来让 Claude 自动跑，现在这件事在她不知情的情况下就完成了。

代码审查：信任但要验证。

这一块是全文信息密度最高的部分，也是我在实际项目里验证过的方向。Claude 现在包办了所有风格检查、lint、PR 反馈请求、抓 bug 并在完整提交前修掉、补充测试。人类仍然需要介入的地方只有一个关键词——

专业判断

（expertise）。具体来说：涉及法律风险决策的，Fung 永远让法务合伙人参与；涉及信任边界和安全敏感代码的，必须有领域专家把关；涉及产品品味和体验的，PM 和设计师必须介入。

但 Fung 特意强调了一点：这个「信任 vs. 验证」的平衡点并非固定的。原话是「the right balance of trust vs. verify will keep changing as the models improve」。模型每升一次级，原来需要人类判断的事情可能就变成可以自动化的。也就是说，代码审查策略不是一个「设好就忘」的东西：它需要持续的重新校准。这一点我深有体会：在用 Claude Code 做前端项目的过程中，最开始 PR 我会逐行看逻辑，后来发现 Claude 在样式和交互逻辑上的判断越来越准，我现在基本只看架构决策和安全相关的部分。

团队构成：角色边界在消融。

这是四个变化里最根本的一个。PM 开始日常写代码了。Fung 说这件事「看着很有意思」。非传统码农背景的人承担了更多工程工作，而工程师反过来开始做内容和设计（那些过去被认为是「非技术」的工作）。角色边界消融并非管理上的放任。agentic coding 拉低了实现门槛，这件事是自然会发生的。

Fung 现在招人的重心完全放在两类人身上。第一类是

有产品嗅觉的创意型 builder

（那些有强烈好奇心、痴迷于做出能解决问题的产品的人）。第二类是

有深厚系统功底的工程师

（她接手团队后发现系统背景存在缺口，而这个缺口在构建 Claude Code on the Web（让 Claude 能在任何环境下运行）时变得极为关键）。反过来，她最不看重的就是

原始代码吞吐量

（模型已经把这部分包了）。招聘的重心从「谁能写得多」转向了「在模型写完之后，哪些地方还需要人类的专业判断」。

把这四个变化放在一起看，有一条主线是清晰的：

模型的角色是从「工具」变成了「默认的协作方式」。

当所有 PR 都是 Claude 辅助的、所有上下文都可以先问 Claude、所有重复性审查都自动化之后，管理者的核心问题从「怎么让团队写得更快」变成了「人类的判断应该放在哪里」。前者是效率问题，后者是结构问题。

2

新规范怎么推行下去

改流程容易，让团队真正接受新流程才是难的部分。Fung 的策略是把规则分成两层：少数几条硬的团队原则（非协商），其余的交给各个 pod 自己定。

三条铁的团队原则：

第一条：

死磕自己的产品。

Claude Code 团队每一个人，包括跨职能的同事，日常都在用 Claude Code 和 Claude Cowork。他们持续找各种方式让 Claude 帮自己工作得更快。这不是「我们做了一个工具所以用一下」。Fung 把这个当硬性要求。

第二条：

团队尽量扁平。

Fung 刚加入时立了一个规矩：每个经理必须先做 IC（个人贡献者），先亲自交付、亲身体验在 Anthropic 做工程师是什么感觉，然后才能带人。整个 Claude Code 和 Claude Cowork 共享一个团队使命。经理们支持不同的 pod，但保持组织的灵活性（人可以随工作流动，而不是被钉在某个组里）。

第三条：

别犹豫，砍掉不再有效的流程。

团队被赋予了明确的权限：质疑和废除已经不再合理的旧流程。这件事说起来容易做起来难：大多数组织里，砍流程比加流程的政治成本高得多。Fung 把它写成团队原则，等于给每个人发了「砍流程许可证」。

Pod 自治：

在这三条硬规则之内，每个 pod 有相当大的自主权。具体怎么用 Claude 做 triage、怎么开站会、哪些工作流先被「Claude 化」，pod 自己决定。这个设计很聪明：硬规则保证了底线（所有人都吃自己的狗粮、组织不膨胀、流程不僵化），自治权给了一线团队根据自己场景做最优化的空间。既非无政府状态，也非自上而下的流程强推。

3

怎么知道新流程在起作用

改流程最怕的是「感觉变好了但实际没变化」。Fung 给了三个她认为每个工程领导者都应该开始追踪的数字，这三个指标放在一起，比任何定性判断都更能说明问题。

第一，新人上手时间在缩短。

一个工程师、设计师或 PM 从入职到能有效交付要多久？这个数字在他们的团队比一年前快得多。Fung 说现在新工程师第一周内就能交付真实代码。这不只是文档写得好的问题，是 Claude 作为「随身导师」让新人的学习曲线被压得更短了。新人对代码库不熟悉？直接问 Claude。不知道内部工具怎么用？Claude 教你。

第二，PR 周期时间在缩短。

这个指标有意思的地方在于，往下挖会发现系统的真实瓶颈在哪。Fung 提到，随着更多代码被生成出来，构建系统和 CI 可能跟不上。也就是说，PR 周期变短了，但 CI 队列可能变长了。如果只看吞吐量不看周期时间分布，你根本不知道瓶颈已经移到了基础设施上。

第三，Claude 辅助提交的占比在上升。

他们团队默认每一条提交都是 Claude 辅助的。Fung 说她过去四个月里没见过一条非 Claude 辅助的提交。这个数字的意义不在于「AI 替代了人」，而在于它说明了团队的默认工作方式已经彻底从手动编码切换到了 agentic coding。它并非目标，是结果。

Fung 在这里给了一个很重要的警告：

不要把吞吐量和成功划等号。

吞吐量只是一个指标，真正的指标是你在解决的问题本身。方向对了，吞吐量帮你更快解决问题；方向错了，吞吐量帮你更快跑偏。这个提醒放在当下特别重要。太多团队在追「AI 辅助下的代码产出量」这个数字，但产出量从来并非终点，它只是到达终点的速度。

4

从哪开始

如果前面这些听起来都对、但不知道从哪下手，Fung 给了一条我见过的最务实的建议：

找你团队里最吵的那条工作流。

最贵的那个、你最怕的那个、或者团队最不想碰的那个。问两个问题：它还在服务于最初的目的吗？如果是，它能被自动化吗？

她讲了一个很有画面感的例子：曾经在一个团队里，每周有一次昂贵的集体评审：一大屋子人坐在会议室里，除了轮到自己汇报的时候抬起头来讲两句，其他时间所有人都在敲笔记本。她问了一句：「我们到底为什么开这个会？看起来是对大家时间的一种昂贵使用。」这句话让所有人意识到这个会早就不需要了，当场取消。

这个例子的妙处在于：没人需要她去发现什么隐藏问题。所有人早就知道这个会没用，只是没人问那个显而易见的问题。很多流程之所以活着，不是因为它们在起作用。从来没有人认真问过「我们为什么还在做这个」。

Fung 收尾的问题是：「你团队的工作流里面，有没有哪一块是你觉得可以自动化、或者干脆可以砍掉的？」

全文读完，几点判断

这篇博客放在更大的行业图景里，有几个信号值得留意。

第一，Anthropic 正在用自己证明自己的产品逻辑。

Fung 的分享并非 PR 话术。她给出的数据（四个月零人工提交、新人一周上线、路线图作废重新来）是实打实的运营结果。当一个做 AI 工具的公司自己因为用了自己的 AI 工具而被迫改变管理方式，这件事本身比任何 benchmark 都有说服力。

第二，工程管理的核心问题正在从「效率」转向「判断力分配」。

过去十年我们一直在优化效率（敏捷、DevOps、CI/CD，本质上都是让代码从脑子到生产环境更快）。但当写代码本身不再构成瓶颈时，核心问题变成了「人类的专业判断应该投放在哪些环节」。Fung 的答案很明确：安全边界、法律风险、产品品味。这三个领域目前是模型的盲区：不是模型不够聪明，是这些判断需要的上下文和责任感超出了当前模型的运作方式。

第三，流程的「自动过期」机制比流程本身更重要。

Fung 给的第三条团队原则（别犹豫，砍掉不再有效的流程）是全文最容易被忽略但最核心的一条。大多数组织的流程是单向累积的：不断往上加，很少往下减。当 AI 把开发速度拉快之后，流程的过期速度也跟着加快。六个月路线图三个月就过时，代码审查策略每次模型升级都要重新评估。一个没有「流程废止」文化的团队，在这种速度下会被自己的规矩拖死。

第四，这篇博客最值得反复读的，并非她具体改了哪些流程，是她思考流程的那套方式。

Fung 对每一个旧流程的处理步骤是一样的：这个流程当初是为了解决什么问题设立的 → 那个问题现在还存在吗 → 如果还在，有没有更轻的办法解决它 → 如果不在，为什么它还没被砍掉。这套思考框架不需要任何工具支撑，任何一个工程 lead 从明天开始就能用。但多数人不会，因为批判自己亲手建立或维护的流程，是管理中最不舒服的事之一。

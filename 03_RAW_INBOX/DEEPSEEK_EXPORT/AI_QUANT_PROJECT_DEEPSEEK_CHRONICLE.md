## 第一阶段：项目早期——从想法到方案

### 1.1 初始接触

项目创始人找到我时，已经有一个明确的目标：实现自动化加密货币交易。当时他对技术细节不太了解，对交易策略也处于学习阶段。但他有一个清晰的认知——需要一个系统化的方案，而不是零散的策略片段。

最初的工作方式是：我提出方案设计，创始人把方案发给第三方（ChatGPT或其他来源）做评价验证，然后把第三方的结论再给我做二次验证。这种“交叉验证”模式在项目早期起到了重要的质量把关作用，但也埋下了后期“信息搬运成本过高”的隐患。

### 1.2 V4.6.2 方案的形成

经过多轮讨论和第三方验证，形成了V4.6.2-Optimized-Update1完整方案。方案的核心设计哲学包括：
- 单向控制流：决策权与执行权分离
- 风险绝对优先：风控规则属于系统层硬编码
- 唯一真实数据源：所有模块读写同一个State Center
- 环境自适应策略路由：先判断市场状态，再选择对应子策略
- 数据驱动迭代：所有参数修改必须有回测数据支撑

这些设计原则在后续的项目推进中被反复验证，至今仍然是项目的核心设计理念。

### 1.3 执行落地计划的制定

方案定稿后，制定了详细的执行落地计划（方案第九章）。项目分为十个阶段，从环境准备到小资金上线。当时的目标是在几周内完成从基础设施搭建到实盘验证的全流程。

**需要重点关注的问题**：当时的时间预估过于乐观。实际执行中，观察期就消耗了13天（远超原计划的3天），且最终因为Ghost Position Incident提前终止。这个偏差反映了项目早期的核心问题——对系统复杂度的低估。


## 第二阶段：策略层开发与部署

### 2.1 策略模块开发

在方案定稿后，陆续完成了全部策略模块的开发：
- 选币雷达（coin_selector.py）
- 市场环境分类（environment.py）
- 三周期入场信号（entry_signal.py）
- 微观结构过滤（microstructure.py）
- 卖出管理（sell_manager.py）
- 策略状态机（state_manager.py）
- 主引擎（strategy_runner.py）

这些模块构成了V4.6.2策略层的完整功能。

### 2.2 P0修复：信号链路修复

5月1日左右，完成了四个关键修复：
- FIX-1：信号入库status匹配——`if status == "SUCCESS"` → `if status in ("SUCCESS", "APPROVED")`
- FIX-2：health_check增加引擎进程存活检测
- FIX-3：Gateway OBI阈值0.20→0.12同步
- FIX-4：flow_ratio 1.5→1.2

这四个修复看似简单，但直接影响了信号链路的完整性。FIX-1的缺失意味着所有paper-trade信号都在入库环节丢失了——如果不是及时发现，观察期的数据将完全空白。

### 2.3 IC因子分析

5月3日，对动量、波动压缩、均值回归三个基础因子做了完整的IC分析。结果全部未通过预测力验证（IC<0.02，RankIC<0.03）。

这是项目第一次用统计方法验证策略的核心假设。结论是残酷的——22分制评分系统不具备方向预测力。IC≈50%，等同于抛硬币。

**需要重点关注的问题**：IC分析的执行过程中出现了数据泄露问题（12h IC=1.0），经排查是时间对齐错误导致。这个发现本身也很重要——验证了“在缺乏系统验证框架的情况下，回测结果可能严重失真”。后来ChatGPT也反复强调这一点，认为“没有系统验证框架时，回测结果是不可信的”。

### 2.4 HOLD分支修补

基于IC分析的结论，按方案10.5.4节决策路径回退到HOLD分支——修补22分制而非替换它。

- HOLD-2：A级门槛 SCORE_A 18→16（降低门槛，增加信号通过率）
- HOLD-3：成交量死路径修复（if False短路移除）
- HOLD-1：方向对称性回测——胜率从~30%提升至~40%，但方向从SHORT偏好转为100% LONG极端。决定不部署。根因是_score_22()对LONG方向存在结构性加分。

**需要重点关注的问题**：HOLD-1的验证过程揭示了22分制的一个深层设计缺陷——评分函数对LONG方向有结构性偏向。这不是参数的偏差，而是评分逻辑本身对多头条件更“友好”。这个发现在后来的多方论证中被再次确认。

### 2.5 状态分层IC验证

利用已有的environment.py对三个因子按Trend/Range/Volatile环境重新计算IC。关键发现：波动压缩因子在volatile环境下有显著正向预测力（BTC +0.0579, SOL +0.0816）。动量和均值回归仍然无效。

环境分类器对加密货币过于敏感——99.7%样本判为volatile，Trend和Range样本数为0。但不调整阈值——加密货币本就是高度volatile的市场，这是市场特征，不是分类器缺陷。

### 2.6 波动压缩因子补充加成

基于分层验证结论，将波动压缩因子作为22分制的补充加成项部署：
- BTC +2分、SOL +2分、ETH不适用、其他默认+1分
- 触发条件：当前20周期振幅标准差 < 历史均值×0.7

**这是整个V4.6.2系统中唯一被统计验证通过的Alpha来源。** 整个系统的期望值几乎依赖这个单独的微结构特征。


## 第三阶段：Kimi 配置与故障排查

### 3.1 Kimi K2.6 双版本配置

5月2日-3日，为OpenClaw配置了Kimi K2.6模型：
- 国内版：moonshot/kimi-k2.6（api.moonshot.cn，主用）
- 海外版：moonshot-intl/kimi-k2.6（api.moonshot.ai，备用）

**需要重点关注的问题**：两个Provider的模型ID相同（都是kimi-k2.6）。当时有过一个关键讨论——是否需要把海外版的模型ID改为kimi-k2.6-intl以防止冲突。最终确认OpenClaw通过完整路径provider/model-id区分路由，不需要改ID。**但后续发现，两个同名模型在配置文件中的共存确实增加了排查复杂性。**

### 3.2 Kimi 海外版 API 瞬断

5月3日凌晨，海外版Kimi在执行Hermes配置检查任务后不回复。这是Kimi问题链的起点。排查过程暴露了项目早期的故障诊断方法缺陷——第一反应是改配置文件，而不是先确认API本身的连通性。

**需要重点关注的问题**：这次排查消耗了大量时间（从凌晨到清晨），期间多次修改openclaw.json、反复重启Gateway。后来证明问题只是海外版API的短暂中断。**“先排查API连通性，再查配置”——这条教训后来被写入AGENTS.md作为永久规则。**

### 3.3 Kimi 上下文超限死循环

5月3日夜间-5月4日凌晨，Kimi在长对话中反复出现上下文满后不回复的问题。根因是Kimi的特殊错误处理机制——当上下文超限时不返回overflow错误，而是直接进入超时状态。OpenClaw的auto-compaction依赖模型返回溢出错误才会触发，因此陷入死循环。

这次排查促成了compaction配置的修正——将mode从default改为safeguard，添加reserveTokens、keepRecentTokens等参数。

### 3.4 Kimi 60K+ 上下文超时——stream:true修复

5月4日凌晨，OpenClaw在Kimi官方FAQ中发现了关键信息：**非流式模式下，Kimi等全部内容生成完才发HTTP响应头，链路上某个网关触发idle timeout断开连接。** 上下文越大→首token延迟越长→越容易触发网关超时。

修复：在两个Kimi Provider中启用stream:true。之后80K tokens的上下文从120s超时改善到31s完成。

### 3.5 Kimi 上下文超时的第三次修复

5月6日，发现即使启用了stream:true，Kimi在96K+上下文时首token延迟仍超过30秒。OpenClaw进一步调整了compaction阈值（reserveTokens从40000提升至60000，keepRecentTokens从30000提升至40000，maxHistoryShare从0.5降至0.4）并增加了超时容忍度（timeoutSeconds从30调整至60）。

**需要重点关注的问题**：Kimi的问题经历了三次迭代才达到稳定状态。每一次修复都揭示了一个新的层面——第一次是compaction机制，第二次是stream模式，第三次是首token延迟的根本限制。这反映了复杂系统中故障诊断的典型特征：不是单一根因，而是多层问题叠加。


## 第四阶段：观察期的波折

### 4.1 观察期的三次延长

观察期原定3天（至5月5日），实际经历了三次延长：
- 第1次（至5月7日）：引擎停机约30h，数据空白
- 第2次（至5月9日）：0笔平仓，SellManager五个Bug修复
- 第3次（至5月10日）：FIX-5排查+DB清理耗时约3h

**每一次延长都不是因为“数据还需要积累”，而是因为“又发现了需要修复的问题”。** 这是Bug驱动的延期，不是计划驱动的延期。

### 4.2 SellManager 五个Bug修复（FIX-5）

5月4日-5日，发现全部持仓0平仓。经深度排查，SellManager存在五个Bug：
- Bug 1：check_all()传入current_price==entry_price，盈亏永远为0%
- Bug 2：每轮创建新SellManager实例，best_prices/max_profits归零
- Bug 3：holding_days硬编码为0
- Bug 4：_positions key不含方向，同币种LONG/SHORT互相覆盖
- Bug 5：log_trade()的UPDATE...ORDER BY LIMIT语法不兼容PostgreSQL

**Bug 4的修复是双重性的**——它在解决“LONG/SHORT互相覆盖”问题的同时，引入了“同向新信号覆盖旧持仓”的新逻辑。这个新逻辑在部署时作为顺带优化引入，**未做独立风险评估**。三天后才被Ghost Position Incident证明为致命缺陷。

### 4.3 Ghost Position Incident

5月8日深夜，发现593号ETHUSDT LONG从风控系统视野中消失，亏损穿透至-5.47%。根因链：同向覆盖→_positions字典中旧持仓被删除→Phase 1不再检查该持仓→止损从未被调用。

**这是整个项目最重要的技术发现之一。** 它证明了：
- 风控引擎依赖不可靠的内存状态作为权威源是架构级错误
- 以symbol_direction为唯一标识无法区分同向的多笔交易
- 缺乏对账循环和孤儿检测机制意味着Ghost Position只能被偶然发现

### 4.4 观察期提前终止

5月8日深夜，基于Ghost Position的诊断结论，三方一致同意观察期提前终止。继续运行只会产生更多被覆盖的亏损数据。引擎停止运行，数据定格。

**观察期的真实价值**：不是证明了策略能赚钱，而是用最小的代价暴露了系统最深层的架构缺陷。13天内发现的缺陷包括5个致命Bug、1个架构级事故、49个僵尸持仓、6个底层设计缺陷。


## 第五阶段：多方论证与认知升级

### 5.1 六大底层缺陷的诊断

5月6-8日，OpenClaw、DeepSeek、ChatGPT三方对策略底层进行了深度诊断，确认了六大设计缺陷：
1. 完全后视镜——“马后炮”入场
2. 周期错配：花20h确认趋势，只给1.5%容错空间
3. 无突破回调入场逻辑
4. 无K线形态判断（三方一致否定在当前阶段投入）
5. 无“顺大逆小”机制
6. 无“不做”的条件

这些缺陷的诊断过程揭示了22分制评分的根本问题：**所有评分项基于已收盘K线的历史指标，没有一个在前瞻性地判断价格会去哪里。**

### 5.2 ChatGPT多轮讨论详细记录

ChatGPT在项目推进过程中参与了多次关键讨论。以下是各次讨论的核心内容、判断和影响。

#### 5.2.1 V5架构蓝图讨论（5月初）

**背景**：V4.6.2方案刚定稿，IC因子分析尚未完成。

**ChatGPT的核心判断**：
- 首次提出了从“规则驱动”转向“因子驱动”的V5方向，建议建立五层架构（数据层→因子引擎→因子验证→组合引擎→决策层）
- 明确指出22分制评分系统不具备预测区分力，应该被替代
- 强调了IC分析必须作为验证标准，统计验证优先于主观判断

**讨论结果**：方向正确但过度设计。我们决定先走HOLD分支修补路线，在IC分析完成后再评估因子替代的可行性。这个决策的逻辑后来被写入方案10.5.4节。**ChatGPT当时提出的“因子驱动”方向后来被三方一致确认为正确，只是实施路径需要务实调整。**

#### 5.2.2 Claude Code引入评估（5月中旬）

**背景**：创始人开始了解Claude Code，考虑是否引入作为项目工具。

**ChatGPT的核心判断**：
- **当前阶段不应引入Claude Code用于策略研发。** 原因：三条前置条件均未满足（baseline策略未稳定、回测体系未可靠、系统结构仍在变动）
- 给出了明确的三条前置条件，以及终局定位（离线策略研发引擎，不接入实时交易链路）
- 建议当前可用场景仅限于服务器运维和脚本编写等纯工程任务

**讨论结果**：我们采纳了ChatGPT的判断，将Claude Code的引入决策记为CC-1，标注为“待条件触发”。**这个决策的逻辑至今仍然有效，并且在后续被写入了AGENTS.md。** 这也是项目第一次明确“先证明Alpha，再谈自动化”。

#### 5.2.3 Exit状态机完整方案（5月中旬）

**背景**：观察期发现4笔全止损，Exit体系被确认为核心短板。

**ChatGPT的核心贡献**：
- 给出了包含TradeState状态机、四种退出类型和完整代码补丁包的P0级Exit体系设计方案
- 明确指出了当前固定止损在加密市场极度脆弱的问题，建议改为结构止损（swing low + ATR buffer）
- 给出了浮盈回撤止盈、时间止损、结构破坏退出的完整触发条件和参数建议
- 特别强调了状态机设计的重要性，指出当前系统“没有交易生命周期模型”

**讨论结果**：OpenClaw对ChatGPT的方案进行了工程化评估和修正（多持仓并发支持、参数回测验证、DB接口适配）。最终方案融合了ChatGPT的架构设计和OpenClaw的工程实现。Exit体系当前未落地是因为观察期约束，5月10日后作为P0优先执行。

#### 5.2.4 SMC宗教化警告（5月中旬）

**背景**：讨论V5的Entry设计方向，涉及FVG、CHoCH、Liquidity Sweep等SMC概念。

**ChatGPT的核心警告**：
- **不要相信FVG一定回补、CHoCH一定反转、Liquidity Sweep一定有效。很多SMC内容是后验解释，不是稳定Alpha。**
- 正确做法是把SMC概念拆解为可统计验证的行为因子（如突破幅度、回撤深度、displacement candle大小），然后做IC分析
- 建议V5第一阶段只做一个Setup（BTC only, 4H bullish regime, sweep+CHoCH+FVG tri-confirm entry），先证明正期望再扩展

**讨论结果**：这个警告被三方一致采纳，直接影响了V5的方向定位——不是“采用SMC框架”，而是“把SMC概念转化为可验证的行为因子”。ChatGPT的“单Setup实验室”建议被确认为V5第一阶段的执行原则。

#### 5.2.5 工具选型评估（5月中旬）

**背景**：OpenClaw给出了两轮社区Skill推荐，创始人需要决策哪些工具值得引入。

**ChatGPT的核心建议**：

- **强调“先可靠再盈利”的工程哲学**：Infra优先于Alpha，风控完整性优先于策略优化

**讨论结果**：采纳了ChatGPT的全部评估结论。quant-research-platform和trading-devbox被确认为当前阶段优先安装的工具，binance-pro被明确否决（实盘阶段再评估），self-improving-agent被明确否决（等策略有盈利样本后再评估）。

#### 5.2.6 Ghost Position事故的独立分析（5月8-9日）

**背景**：Ghost Position Incident刚被发现，正在深度排查根因。

**ChatGPT的核心贡献**：
- 将事故定性为**“Production-Critical Risk（生产级关键风险）”**和**“Ghost Position Incident（幽灵持仓事故）”**
- 明确指出**“风控引擎失去了对真实持仓的控制权”**是自动交易系统中最高等级问题之一
- 提出**“Phase 1依赖volatile in-memory state是架构级错误”**的诊断
- 补充了三项必须立即加入的防御机制：Position Reconciliation Loop（持仓对账循环）、Orphan Position Detection（孤儿持仓检测）、Risk Check Timestamp（风控检查时间戳）
- 特别强调“你们这次其实很幸运——问题发生在观察期，不是高杠杆阶段。否则一次dict corruption可能直接变成账户级事故”

**讨论结果**：ChatGPT的诊断与OpenClaw的系统状态分析完全一致。他的“Ghost Position Incident”定性被直接采纳，写入了INCIDENT_REPORT_2026_05_09.md。他的三项补充防御机制被确认为V5 Phase B的核心任务。

#### 5.2.7 “受控架构重置”建议（5月8日深夜）

**背景**：Ghost Position确认后，讨论V4是否应该全推翻重写。

**ChatGPT的核心判断**：
- **不全推翻，但也绝不继续在V4.6.2上缝缝补补。** 正确路线是“Controlled Architectural Reset（受控架构重置）——保留经验，重构核心”
- V4冻结为Baseline，V5新建独立目录，从零重写核心模块，继承所有基础设施
- **“先Reliable再Profitable”——Infra修复优先级高于一切策略优化**
- 给出了分阶段重建路线：Phase A（事故审计）→ Phase B（核心重建）→ Phase C（研究引擎）→ Phase D（单Setup验证）

**讨论结果**：这个判断被三方一致采纳。V4.6.2于5月8日深夜正式冻结。ChatGPT的“受控架构重置”成为V5启动的核心原则。

#### 5.2.8 MASTER_CONTEXT补充建议（6月2日）

- 强调“第24章（创始人认知演化）是整个文档最重要的章节，因为整个项目最大的变量不是V4，不是Ghost Position，而是创始人的认知变化”

### 5.3 参数一致性问题

5月7日，发现holding_days的代码值（5天）与文档计划值（24h）不一致。经全面排查，共发现11个参数完全对齐、6个存在临时值、3个标注为待执行、0个需要立即修复的不一致。排查覆盖了sell_manager.py、strategy_runner.py、config.py、entry_signal.py全部核心文件。

### 5.4 数据一致性问题

同一天，发现巡检报告基于48小时未更新的CSV死文件，而非PostgreSQL实时数据。复盘报告的verdict口径也与数据库不一致。此后确立PostgreSQL为唯一权威数据源，所有CSV降级为历史快照。

### 5.5 Claude Code 引入决策

经过两轮深度讨论，确认：当前阶段不引入Claude Code用于策略研发。终局定位是离线策略研发引擎（Dev Agent），不接入实时交易链路。引入前置条件：①有稳定运行的baseline策略②回测体系与IC验证框架可靠③系统结构不再大改。

**需要重点关注的问题**：这是项目第一次明确“先证明Alpha，再谈自动化”。这个决策的逻辑至今仍然有效。

## 需要重点关注的问题

### 1. 多AI协作的治理

项目经历了单AI→多AI平权→主脑模式的演变。当前的Claude主脑模式是经过反复验证后确认的最佳架构。任何试图回到“多AI平权”的建议都已被否决。Claude是项目主控AI，DeepSeek和ChatGPT是辅助角色，Codex是工程执行AI。

### 2. 故障诊断的方法论

项目早期最严重的问题之一是故障诊断方法——遇到问题时第一反应是改配置文件，而不是先确认API本身的连通性。EXP-1（Kimi海外版瞬断）消耗了数小时，期间反复修改openclaw.json、反复重启Gateway，结果只是API服务端短暂中断。**“先排查API连通性，再查配置”——这条规则是用数小时的无效排查换来的。**

### 3. 修改前评估、修改后验证

FIX-5引入同向覆盖逻辑时没有做独立风险评估，三天后才被Ghost Position Incident证明为致命缺陷。**任何涉及持仓生命周期的代码变更必须单独做风险评估。修复后必须验证Phase 1是否覆盖全部活跃持仓。** 这是用-5.47%的亏损换来的铁律。

### 4. 观察期的真正价值

观察期没有达到“证明策略能赚钱”的原始目标，但完成了另一个更重要的任务——用最小的代价全面暴露了系统最深层的架构缺陷。如果在实盘高杠杆环境下发生Ghost Position，后果将是账户级的。

### 5. 文档体系是持久化记忆层

聊天记录不可追溯、不可治理、不可版本化。文档体系是唯一可靠的持久化记忆层。所有关键决策、参数变更、方案讨论必须在讨论结束后写入对应文档，不依赖任何单一AI的“记忆”。

### 6. 当前阶段不是V5开发

项目当前处于Phase 0A——AI Governance Infrastructure建设阶段。V5的启动需要满足多个前置条件（事故报告完成、架构修复完成、研究协议建立、文档体系L0治理骨架建立）。在当前阶段推动V5开发是本末倒置。

### 7. ChatGPT在项目中的角色

ChatGPT在项目推进过程中扮演了多个关键角色：战略方向校准（V5架构蓝图、先可靠再盈利原则）、方案保真校核（Exit状态机方案的SMC宗教化警告）、风险识别（Ghost Position事故定性为Production-Critical Risk）、方法论指导（Triple Barrier Method、Event-based sampling、单Setup实验室）。ChatGPT的优势在于宏观工程视角和方法论框架，但无法访问系统实际状态——这是它与OpenClaw的核心差异。ChatGPT的任何判断在采纳之前，必须经过OpenClaw的系统状态验证。

## §1 AI原生公司运营核心理念

### 1.1 Anthropic 创始人手册的四阶段模型

**来源：** 文章01/16/14/15（Anthropic《The Founder's Playbook》）

AI时代创业生命周期不再是「验证→融资→招人→建造→再融资」的线性路径。AI把三件事拆解了：研究、编码、流程自动化。创始人角色从「写代码的IC」变成「指挥agent的orchestrator」。

**四个阶段及退出条件：**

| 阶段 | 核心问题 | 退出条件 | 最大陷阱 |
|------|---------|---------|---------|
| Idea | 这值不值得做？ | 3个yes：问题真实、方案匹配、有足够信号 | 把「造原型」当「验证完成」（42%创业公司死于无人需要的产品）|
| MVP | 到底先造什么？ | 留存+收入+推荐三信号 | 没写CLAUDE.md，每次session重新推导，代码无共同心智模型 |
| Launch | 把traction变引擎 | 增长可重复+产品扛得住+运营不卡你 | 「1小时决策开始拖一周」=失控信号 |
| Scale | 挖护城河 | 盈利/IPO/并购三选一 | 没有差异化，对手复制后用户流失 |

**与本项目的映射：**
- 本项目当前处于 Idea→MVP 边界，验收条件5（第一个研究闭环）等同于找到problem-solution fit
- 核心威胁与文章描述完全匹配：「前进速度超过理解所能支撑的范围」（风险C）

### 1.2 YC合伙人的 AI原生公司三要素

**来源：** 文章11（YC合伙人Diana）

**三个核心判断：**

**① 把每个流程做成「闭环」**
- 开环：做决定、执行、不系统度量结果、不回头调整
- 闭环：持续监测输出、自我调节、随时间改进
- 量化研究项目天然适合闭环——回测结果→调整假设→再验证

**② 让整家公司「可被查询」（queryable）**
- 每个重要动作产出一个 artifact（制品），让智能体能从中学习
- 用AI记录关键决策、在所有渠道嵌入agent、为所有维度建自定义看板
- 本项目的DECISION_LOG、CURRENT_STATE、PROJECT_CONTEXT 已经是这个思路的实践

**③ 三种员工原型**
- **一线贡献者（IC/Builder）**：亲手做东西、运营，开会带能跑的原型
- **DRI（直接责任人）**：对结果负有明确责任，一人一个结果
- **AI创始人型**：在最前线示范「能力跃升」长什么样
- 本项目对应：Codex=Builder，Claude=DRI（架构），Founder=AI创始人

**关键公式：** 最大化 token 用量，而不是最大化人头（token maxing，非 headcount）

### 1.3 AI确认偏误风险

**来源：** 文章16（Anthropic手册）

> 「让AI验证你的创业想法，它能找出佐证；让它估算潜在市场，它能找到让TAM看起来值得融资的数字。AI会沿着你的方向走。」

**防范措施（已内置于本项目CLAUDE.md）：**
- 先验证问题本身，再给出方案
- 建议与决策分离，未经用户确认不得作为既定前提
- 反顺延与拒绝权：当方向偏差时必须先指出

## §2 Claude Code 最佳实践

**来源：** 文章03/05/12/13（Claude Code官方最佳实践）

### 2.1 核心约束：上下文窗口管理

**最重要的单一约束：** 上下文窗口填充越满，性能越差。所有最佳实践都围绕这一约束展开。

**核心规则：**
- 任务之间用 `/clear` 重置上下文
- 连续纠正Claude超过两次后，必须 `/clear` 并重新开始
- 用 `/compact <instructions>` 自定义压缩行为
- 调查类工作委托给 subagents（它们在独立上下文中探索）

### 2.2 四阶段工作流（先探索，再规划，最后编码）

```
探索 → 规划 → 实现 → 提交
```

1. **探索**：进入计划模式，Claude只读文件回答问题，不做修改
2. **规划**：让Claude创建详细实现计划（先在SPEC.md里写清楚）
3. **实现**：切出计划模式，编码+验证
4. **提交**：Claude提交代码并创建PR

**何时跳过规划：** 范围明确的小任务（修拼写、加日志行）直接执行

### 2.3 有效的 CLAUDE.md 写法

**应该包含：**
- Claude猜不到的Bash命令
- 与默认值不同的代码风格规则
- 测试说明和首选测试运行器
- 仓库规范（分支命名、PR约定）
- 项目特定的架构决策
- 开发环境的特殊要求

**不应该包含：**
- Claude通过阅读代码就能知道的内容
- Claude已经知道的标准约定
- 频繁变化的信息
- 冗长的解释或教程

### 2.4 验证驱动开发

**给Claude验证工作的方法是最重要的单一实践。**

| 场景 | 不好的做法 | 好的做法 |
|------|---------|---------|
| 函数实现 | "实现一个验证邮箱的函数" | "写validateEmail函数，测试用例：user@example.com→true，invalid→false" |
| UI变更 | "让仪表盘更好看" | "粘贴截图，实现这个设计，截图对比差异并修复" |
| Bug修复 | "构建失败了" | "构建失败报这个错误：[粘贴错误]，修复并验证构建成功" |

### 2.5 动态工作流（Dynamic Workflow）

**三种Agent工作模式对比：**

| 模式 | 适用场景 | 限制 |
|------|---------|------|
| SubAgent | 少量并行任务 | MainAgent成为瓶颈，无法直接通信 |
| AgentTeams | 协作任务（≤5个成员） | 会话中断后无法继续 |
| 动态工作流 | 大规模并行（最多16个子Agent，理论1000个） | Token消耗高；需用户确认首次计划 |

**动态工作流关键优势：**
- 对抗性检查：Agent从独立角度尝试反驳其他Agent结果
- 中断可恢复：保存任务进度，可从中断处继续
- 自动编排：只需描述目标，Claude决定如何拆分

**最佳实践：** 先从小规模任务入手评估Token消耗，再推广

### 2.6 常见失败模式

| 问题 | 解决方案 |
|------|---------|
| 大杂烩会话（从一个任务跳到不相关任务） | 任务间用 `/clear` |
| 反复纠正（Claude做错了，纠正后还是错） | 两次纠正失败后 `/clear` 重写提示 |
| 过度配置CLAUDE.md（文件太长） | 无情删减，Claude已正确执行的指令删除 |
| 信任但不验证（看似合理但未处理边界情况） | 始终提供验证（测试、脚本、截图） |
| 无限探索（让Claude"调查"但无限定范围） | 窄化范围或使用subagents |

## §3 Claude Prompt 工程指南（Opus 4.8）

**来源：** 文章20（Anthropic官方Prompt工程文档精读）

### 3.1 Effort 参数（最重要的单一参数）

Claude Opus 4.8 的 `effort` 参数五个级别：

| 级别 | 适用场景 | 注意 |
|------|---------|------|
| `max` | 某些场景有收益，但可能过度思考 | 慎用 |
| `xhigh` | coding和agentic场景最佳起点 | **推荐默认** |
| `high` | 大多数intelligence-sensitive场景的最低保底 | — |
| `medium` | 成本敏感场景 | 推理可能太浅 |
| `low` | 短任务、延迟敏感、非智力密集型 | — |

**重要：** Claude Opus 4.8 默认不开thinking，必须显式设置：
```python
thinking={"type": "adaptive"}
```

在 `max`/`xhigh` effort下，`max_tokens` 至少设64k（模型和子代理需要思考空间）

### 3.2 Agent 系统全栈调校（12个维度）

**与本项目最相关的6个：**

**① 状态管理**
- 多个Context Window工作流：把长线任务分解成多个独立session
- 每个session开始时加载上一个session的结论（即本项目的Memory Core机制）
- 在文件系统而非上下文中保存状态

**② 安全边界分层**
- 三层：系统级约束（不可覆盖）→ 操作者级规则（Founder确认）→ 用户级灵活度
- 关键操作必须有人类审批闸门（review gate）

**③ Subagent 编排**
- 能直接在单次回复中完成的工作不要spawn子代理
- 跨条目展开或读取多个文件时，同一轮中spawn多个子代理

**④ Prompt Chaining**
- 复杂任务拆成有序的步骤链
- 每步的输出作为下一步的输入，而非一次性大任务

**⑤ 过度工程化防范**
- 当方案复杂度超过当前阶段实际需要时主动指出
- 当前阶段未验证稳定前，不展开下一阶段大规模设计（已内置于CLAUDE.md MODULE 6）

**⑥ 幻觉最小化**
- 对不确定的内容要求模型标注「我不确定」而非编造
- 关键数据点要求引用来源

### 3.3 回复长度与风格调校

Claude Opus 4.8 对 prompt 做字面理解和显式遵循。

**直接指令示例（已可用于本项目）：**
```
Provide concise, focused responses. Skip non-essential context, and keep examples minimal.
Apply this formatting to every section, not just the first one.
```

**子代理控制：**
```
Do not spawn a subagent for work you can complete directly in a single response.
Spawn multiple subagents in the same turn when fanning out across items or reading multiple files.
```

## §4 AI原生工程管理流程变革

**来源：** 文章12（Anthropic工程总监Fiona Fung分享）

### 4.1 瓶颈已转移

> 「写代码、写测试、重构这些事已经很少拖慢我们了。瓶颈转移到了验证（verification）、代码审查（code review）和安全（security）。」

**旧瓶颈 → 新瓶颈：**
- 旧：谁来写代码
- 新：人类的判断应该放在哪里

### 4.2 四个已废弃的旧流程

| 旧流程 | 废弃原因 | 新做法 |
|--------|---------|--------|
| 六个月路线图 | AI开发速度太快，三个月就过时 | JIT规划（just-in-time，够用就行） |
| 「谁写的这段代码？」 | 所有PR都经过Claude，问谁没意义 | 直接问Claude：这段变更是谁引入的、为什么、上下文是什么 |
| 人工代码审查（全量） | AI能包办风格/lint/PR反馈/抓bug | 人类只介入：安全边界、法律风险、产品品味 |
| 固定角色边界 | PM开始写代码，工程师做设计 | 角色边界消融，按工作流动而非固定在组里 |

### 4.3 代码审查的「信任vs验证」原则

**人类必须介入的情况：**
- 涉及法律风险决策 → 永远让法务参与
- 涉及信任边界和安全敏感代码 → 领域专家把关
- 涉及产品品味和体验 → PM和设计师介入

**关键提醒：** 这个平衡点不是固定的。模型每升一次级，原来需要人类判断的事情可能就变成可以自动化的。需要持续重新校准。

### 4.4 值得追踪的三个指标

1. **新人上手时间**（从入职到能有效交付要多久）——应该越来越短
2. **PR周期时间**（往下挖会发现真实瓶颈，比如CI队列变长）
3. **Claude辅助提交占比**（说明团队默认工作方式是否已切换）

**警告：** 不要把吞吐量和成功划等号

## §5 Agent Routine 设计：从「人肉闹钟」到「AI Worker」

**来源：** 文章17/18（Agent Routine系列）

### 5.1 核心判断

> 「人应该卡在少数真正需要判断的位置：方向选择、公开发布、资金操作、生产环境、客户承诺。其他能按规则推进的步骤，就应该自己推进。」

### 5.2 Routine 的8个结构要素

| 要素 | 定义 | 量化研究场景示例 |
|------|------|-----------------|
| **trigger** | 什么事件触发 | 定时触发（每天盘后）/ 数据到达触发 |
| **route** | 归到哪个工作区/负责人 | 路由到 06_RESEARCH/ 对应策略目录 |
| **context** | 必须加载哪些知识库 | 策略规格、历史回测参数、DECISION_LOG |
| **contract** | 预期产出、验收标准、边界 | 「输出夏普比率、最大回撤、IC值；样本<300次=不成立」|
| **tools** | 能用哪些工具，哪些不能自动做 | VectorBT回测OK；实盘下单永远需要人工确认 |
| **verification** | 能机器检查的不靠AI说「我确认了」 | 跑统计检验（t-test/IC）、对比benchmark |
| **state** | 结果写回哪里 | 写入 06_RESEARCH/[策略名]/RESULT_LOG.md |
| **recovery** | 失败时怎么建修复任务 | 检查失败→创建待处理假设条目→通知Founder |

### 5.3 Routine 适用判断标准

**适合做成Routine的工作：**
- 同类工作反复发生（如每周回测）
- 有明确验收标准可以机器检查
- 失败情况有清晰处理路径
- 人类的判断只需要在少数节点介入

**不适合做成Routine的工作：**
- 假设定义（需要人类判断，不可自动化）
- 架构决策（需要Founder确认）
- 新策略方向探索（探索性，不确定性高）

### 5.4 CI-式的研究验证流程（类比）

借用文章中CI失败处理的思路，映射到量化研究验证：

```
触发：新假设定义完成
  ↓
分类：
  - 样本量<300 → 直接标记"不成立"，不继续
  - 数据质量问题 → block，要求补充数据，不进入回测
  - 假设清晰可测试 → 进入回测流程
  ↓
执行：VectorBT回测（含资金费率、强平价距建模）
  ↓
验证：IC检验、夏普比率、样本量是否≥500
  ↓
写回：结果写入RESULT_LOG + 更新CURRENT_STATE
  ↓
Review Gate：结论好坏都需Founder看一遍后决定是否进实盘候选
```

## §6 Paperclip：多Agent公司级编排参考

**来源：** 文章10（Paperclip开源项目深度拆解，6.6万star）

### 6.1 Paperclip 解决的核心问题

> 「如果说Claude Code、Codex这些Agent是你的'员工'，那Paperclip就是你的'公司'——有组织架构、有预算管理、有目标对齐、有治理审批。」

**对应本项目的痛点：**
- 多个Agent上下文割裂 → 任务向上流动到目标链，Agent知道「做什么」和「为什么做」
- 成本失控 → 每个Agent有月度预算上限，达到限额自动停止
- 重复性工作需手动触发 → 心跳机制（Heartbeat）按计划调度

### 6.2 可借鉴的设计原则（当前阶段不需要Paperclip本身）

**原子化检入（Atomic Check-in）：**
- 带执行锁，杜绝两个Agent同时处理同一任务
- 零重复工作，零上下文丢失
- **本项目实践**：Claude工作时Codex不能同时修改同一文件

**目标对齐（Goal Alignment）：**
- 每个工单携带完整目标链：公司目标→项目目标→里程碑→本工单
- Agent始终知道「我在为谁做什么、为什么做」
- **本项目实践**：CLAUDE.md + DECISION_LOG 已经实现了这个目标链

**治理审批（Governance）：**
- 没有Founder的签字，什么都上不了线
- 完整审计日志
- **本项目实践**：DECISION_LOG 的「决策与建议分离」原则

### 6.3 何时考虑引入 Paperclip

当以下情况出现时：
- Phase 1 后，同时运行的Codex任务超过3个
- Agent之间有复杂的依赖关系需要协调
- 需要跨会话持久化任务状态
- 月度API成本超过500元需要精细管控

**当前阶段（Phase 0A）不引入，复杂度超过实际需要。**

## §7 真正的 AI Agent 判断标准

**来源：** 文章09

> 「'下一步该干什么，是谁决定的？'
> 答'我事先连好的线决定的'——那不是Agent；
> 答'模型看了现状自己决定的'——那才是Agent。」

**区分标准：**
- **编排对大模型的调用** → 工作流工具（如固定pipeline）
- **让大模型来编排** → 真正的Agent

**对本项目的含义：**
- Phase 0A-1：固定研究流程（Pipeline式）是合理起点，成本低、可预测
- Phase 2+：转向真正的Agent（让Claude自主决定研究路径）需要先验证固定流程的有效性

## §8 关键洞见汇总（可直接行动）

### 立即可用（Phase 0A当前）

1. **CLAUDE.md已内置本文大量最佳实践** — 当前治理框架与Anthropic手册高度对齐
2. **验证驱动开发** — 给Codex的每个任务都要附带明确验收标准（类比文章§2.2）
3. **上下文管理纪律** — Claude会话之间主动用 `/clear`，防止上下文污染
4. **JIT规划** — 不做六个月路线图，只做当前阶段所需最小规划（已在PROJECT_MASTER_PLAN_v2中体现）
5. **Effort参数** — 在API调用中显式设置 `effort: "xhigh"` 和 `thinking: {"type": "adaptive"}`

### Phase 0B/1 准备时考虑

6. **Routine化研究循环** — 用§5的8要素设计结构化研究Routine
7. **CI-式验证闸门** — 回测流程加入自动验证（样本量<300直接拒绝，无需人工判断）
8. **「可被查询」的项目** — 强化artifact输出，每次研究都产出结构化制品

### Phase 2+ 参考

9. **Paperclip架构** — 当多Agent协调复杂度超出人工管理能力时引入
10. **动态工作流** — 当需要大规模并行假设验证时（16个子Agent同时跑不同参数组合）
